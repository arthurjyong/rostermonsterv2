### D-0020: `clasp run` requires both GCP consent-screen scope allowlist and project-scope login
- **Date:** 2026-04-21
- **Status:** Accepted
- **Context:** During M1 C4 execution, a live `clasp run` attempt against the production Apps Script project surfaced `You do not have permission to call SpreadsheetApp.openById. Required permissions: https://www.googleapis.com/auth/spreadsheets`, even though steps 1–4 of the M1 README's "API executable / clasp run" prerequisite list had been satisfied (user-managed GCP project, Apps Script API enabled, `executionApi.access: MYSELF` in the manifest, API Executable deployment created, Desktop OAuth client created). The failure mode was not a configuration bug in the generator — it was two independent OAuth gates neither of which is visible at login time.
- **Decision:** Treat `clasp run` as requiring both of the following, in addition to the existing prerequisites, as a stable operator-facing operational requirement:
  1. Every scope declared in the Apps Script manifest's `oauthScopes` (including `https://www.googleapis.com/auth/spreadsheets`) must be added to the GCP OAuth consent screen's Data Access allowlist for the user-managed GCP project.
  2. `clasp login` must be run with `--use-project-scopes --include-clasp-scopes` (in addition to `--creds <path>`), so the resulting clasp token actually requests the manifest's declared scopes rather than clasp's baseline scopes alone.
- **Rationale:** The two gates are independent. Allowlisting a scope on the consent screen does nothing if the clasp token does not request that scope; requesting a scope at login does nothing if the consent screen disallows it. Omitting either gate fails at runtime with the same opaque "Required permissions" error, which is easy to misdiagnose as a manifest or deployment issue. Making this explicit in the README and decision log prevents the same failure recurring on future operator onboarding.
- **Consequences:** `apps_script/launcher/README.md` documents both gates in the "API executable / `clasp run`" section so operators do not have to rediscover this. Any future Apps Script project in this repo that declares additional scopes must extend both the consent-screen allowlist and the clasp login invocation, not only the manifest.
- **Follow-up actions:** Keep the README prerequisite list authoritative for per-operator setup. Do not fork this guidance into multiple places.
- **Related docs:** `apps_script/launcher/README.md`

### D-0021: Addendum-milestone convention and activation of Milestone 1.1
- **Date:** 2026-04-21
- **Status:** Accepted
- **Context:** After M1 closed on operator delivery (D-0019), it became clear that M1's operator-facing story was not fully complete: the generator remained maintainer-only, and the named monthly-rotation pilot operators could not invoke it without direct Apps Script execution. Extending the generator with a thin operator-facing launcher is derivative of M1's operator-facing surface, not a compute-line milestone like M2/M3/M4/M5. D-0019 had noted that any future M1-scoped change should be treated as a new milestone or a deliberate M1 reopen; neither of those quite fits an operator-facing addendum that does not alter generated-sheet semantics.
- **Decision:** Introduce an explicit **addendum-milestone** concept in the planning vocabulary.
  1. An addendum milestone extends a parent milestone's operator-facing story without reopening it. The parent milestone stays `Completed`; the addendum is a distinct milestone with its own checkpoints, tasks, and exit criteria.
  2. Addendum milestones are numbered `M<parent>.<n>`, where `n` is an integer counter starting at 1 (`M1.1`, `M1.2`, `M1.3` …). Nested decimal sub-numbering (for example `M1.1.1` or `M1.15`) is not used.
  3. Only one milestone is active at a time per the existing one-active-milestone rule; activating an addendum returns its siblings or later milestones to Planned until the addendum closes.
  4. Activate `M1.1` — `Operator-facing launcher` — as the first addendum milestone under this convention. M2 (`Minimal local compute pipeline`) returns to Planned while M1.1 is active.
- **Rationale:** Addendum framing is more honest than either option allowed by D-0019 for this specific slice. It preserves M1 closure integrity (M1 is genuinely Completed; the launcher is additive), keeps M2's compute-core direction untouched, and gives a reusable planning pattern for future operator-facing extensions that do not merit a full top-line milestone. Integer-only addendum numbering avoids the decimal-depth cliff (`M1.5` → `M1.75` → `M1.875`) that arises from ad-hoc decimal insertion.
- **Consequences:** `docs/delivery_plan.md`, `docs/roadmap.md`, `README.md`, and `AGENTS.md` reflect M1.1 as the active milestone and M2 as Planned. Any future addendum to any parent milestone uses the same `M<parent>.<n>` scheme. Addenda are normal milestones for execution-discipline purposes (checkpoints, tasks, sign-off) even though they are derivative in framing. A parent milestone must already be Closed before an addendum can be activated; addenda cannot be used to sneak a reopen past D-0019's guidance.
- **Follow-up actions:** Reflect the convention in `docs/delivery_plan.md` §5/§6/§12/§15, `docs/roadmap.md`, `README.md` current focus, and `AGENTS.md` current-focus-at-time-of-writing line. Keep D-0019 valid as historical record rather than marking it Superseded.
- **Related docs:** `README.md`, `AGENTS.md`, `docs/roadmap.md`, `docs/delivery_plan.md`

### D-0022: Milestone 1.1 launcher architecture — stack, access model, and boundary placement
- **Date:** 2026-04-21
- **Status:** Accepted
- **Context:** M1.1 (activated under D-0021) requires a narrow operator-facing launcher that wraps the existing M1 generation entrypoints (`generateIntoNewSpreadsheet` / `generateIntoExistingSpreadsheet`). Architecture-first discussion for this thread settled a coherent set of decisions — implementation stack, access model, architectural boundary placement, and operator-rotation mechanics — that belong recorded together rather than scattered as individual entries.
- **Decision:** For the M1.1 pilot, accept the following as a single coherent architecture position.
  1. **Implementation stack:** Google Apps Script Web App, deployed with "Execute as: User accessing the web app" and "Who has access: Anyone with a Google Account." Launcher code lives in `apps_script/launcher/` alongside the existing generator.
  2. **Architectural boundary:** the launcher is a thin front-end inside the sheet-adapter layer (Blueprint §7 boundary #2). It is not a new architectural boundary. Launcher code must not absorb parser/normalizer/rule/solver/scorer logic, must not alter generated-sheet structure or semantics, and must not persist per-operator state beyond Google's OAuth session.
  3. **Access model:** identity is established by native Google OAuth under "Execute as user"; authorization is granted by adding the operator's Google account to the GCP OAuth consent screen's **Test Users** list for the launcher's GCP project. The launcher itself does not maintain or check a separate operator allowlist for pilot scope.
  4. **Operator rotation:** monthly operator rotation is handled by the maintainer editing the Test Users list between cycles. Rotation is not encoded in app logic.
  5. **Input normalization:** the operator-supplied spreadsheet reference accepts both a bare spreadsheet ID and a full Google Sheets URL containing one. Normalization happens centrally in `normalizeAndValidateConfig_`, not in the launcher. Extraction rule is recorded in `docs/sheet_generation_contract.md` §12.5.
- **Rationale:** Apps Script Web App is the narrowest launcher path that (a) reuses the existing Google-native deployment model for the generator, (b) gets OAuth identity for free under "Execute as user," (c) introduces no new runtime infrastructure, and (d) does not force a cross-service hosting choice during the pilot. Keeping access gating external to the app (Test Users list) avoids building a per-operator allowlist/role model before the pilot has justified one. Centralizing spreadsheet-reference normalization in the existing config helper keeps the launcher thin and lets future callers — including smoke tests — inherit the more forgiving input form at no extra cost.
- **Consequences:** Alternative launcher platforms (static page over Apps Script API Executable, Sheets workspace add-on, Cloud Run front-end) are deliberately deferred and recorded as deferred in `docs/delivery_plan.md` §10. If pilot usage outgrows the Apps Script HTML UI, the natural next step is to front-end the existing generator via API Executable without changing server-side generation code. The `spreadsheetId` field in `docs/sheet_generation_contract.md` §3A widens backward-compatibly to accept URL or bare ID; §12 adds the launcher surface as a narrow contract section rather than spawning a new contract doc. D-0018's ownership split (Apps Script for sheet-facing surface, compute-heavy core outside Apps Script) is preserved — the launcher is sheet-facing surface, not compute.
- **Follow-up actions:** Implementation-phase work lands in a follow-up code patch on `claude/oauth-roster-webapp-DCasP` (command-line CC session), tracked under M1.1 C1 tasks T1–T4 in `docs/delivery_plan.md` §9.
- **Related docs:** `docs/delivery_plan.md`, `docs/sheet_generation_contract.md`, `docs/blueprint.md`, `AGENTS.md`, `apps_script/launcher/README.md`

### D-0023: Auto-share new spreadsheets via Drive Advanced Service (v3), not `DriveApp.setSharing`
- **Date:** 2026-04-22
- **Status:** Accepted
- **Context:** The M1.1 launcher's new-spreadsheet mode creates the roster sheet inside the operator's own Drive (under `executeAs: USER_ACCESSING`). Without an explicit sharing step the file is private to the operator, forcing a per-cycle manual "click Share → Anyone with link → Editor" gesture before the URL is useful to doctors. Automating that one step was an obvious pilot-friction win, but the right way to do it turned out to require decision-level discipline after three implementation attempts converged.
- **Decision:** Auto-share is implemented via the Drive Advanced Service (REST v3): `Drive.Permissions.create({ type: 'anyone', role: 'writer' }, fileId)`, called right after `SpreadsheetApp.create` in `generateIntoNewSpreadsheet`. The call is wrapped in try/catch — on failure the sheet still generates, the response carries `autoShared: false` plus the underlying error text in `autoShareError`, and the success view renders a manual-share hint instead of faking success. On success the response carries `autoShared: true` and the success view confirms the sharing state. `DriveApp.setSharing` / `DriveApp.getFileById` are not used. The declared OAuth scopes are `spreadsheets`, `drive.file`, and `userinfo.email`; `dependencies.enabledAdvancedServices` declares Drive v3; the linked user-managed GCP project must have the Drive API turned on. Existing-spreadsheet mode is unchanged and does not auto-share (parent-file sharing is inherited).
- **Rationale:** Three paths were attempted in sequence during M1.1 pilot testing:
  1. **`DriveApp.setSharing` with `drive.file`** — failed at runtime. Apps Script's static scope analysis routes any `DriveApp.getFileById` call to the restricted full `drive` scope regardless of manifest declarations, so the narrow scope was effectively ignored and the call threw "Required permissions: drive.readonly || drive."
  2. **`DriveApp.setSharing` with full `drive`** — redeployed at version `@8` with the widened scope. Still failed with the same runtime permission error on both maintainer accounts, even after scope change + redeploy. Root cause was not pinned down, but OAuth grant caching interacting with Test-users mode is the most plausible explanation. Committing the project to the restricted `drive` scope would also have imposed a considerable verification cost (CASA assessment) if the app ever moves out of Testing.
  3. **Drive Advanced Service v3 with `drive.file`** — works cleanly. The Drive REST API's `permissions.create` accepts `type: 'anyone'` under `drive.file` for files the app itself created, which covers every new-mode generation by construction. Re-consent triggered correctly when the new scope was added to the manifest, and subsequent runs succeeded without OAuth friction.
- **Consequences:**
  - OAuth footprint stays on non-restricted scopes, preserving an easy path to future Google app verification.
  - The response object for `generateIntoNewSpreadsheet` gains two additive fields (`autoShared`, `autoShareError`); `generateIntoExistingSpreadsheet` is unchanged. `docs/sheet_generation_contract.md` §12 already describes the launcher contract surface and does not need structural changes to accommodate these additive fields — the auto-share behavior was in-scope there.
  - Operator onboarding gains one prerequisite: the linked GCP project must have the Drive API enabled. Recorded in `apps_script/launcher/README.md`.
  - Future features that need to read or modify files not created by the app (e.g. cross-cycle spreadsheet discovery, reading operator-attached files) cannot piggyback on `drive.file` alone and would need to revisit this decision.
- **Related docs:** `apps_script/launcher/README.md`, `docs/sheet_generation_contract.md` §12, `docs/delivery_plan.md` §9 M1.1 C1, `docs/blueprint.md` §6–§7

### D-0024: Rule engine architecture contract — stateless surface, full-violation canonical ordering, scoped fixed-assignment handling
- **Date:** 2026-04-23
- **Status:** Accepted
- **Context:** M2 (`Minimal local compute pipeline`) needed a normative rule-engine contract before parser/normalizer implementation closure, so downstream solver/scorer planning would rest on a fixed validity-evaluation boundary instead of an informal understanding drawn from blueprint §7.4 and `docs/domain_model.md` §9.2 alone. Several design questions were live: whether the rule engine's public surface should expose a prepare-then-query lifecycle (cheaper for incremental search but harder to reason about and test) or a stateless pure-predicate surface; whether the engine should return only the first-hit violation or the full list of applicable violations; how to handle `FixedAssignment` entries that the parser admitted under its scoped exception (`docs/parser_normalizer_contract.md` §14); and how a future incremental-state implementation should be allowed in without replacing the reference behavior. The scoring-awareness question was also live — specifically, whether soft-effect reading should sit in the rule engine alongside hard-validity checks or in the scorer — and the answer interacts with D-0025 and D-0027.
- **Decision:** Adopt `docs/rule_engine_contract.md` as the normative rule-engine boundary with the following coherent set of decisions:
  1. **Stateless public surface.** The rule engine is a pure function `(normalizedModel, ruleState, proposedUnit) → Decision`. Implementations MAY internally cache/index as an optimization but MUST NOT expose session handles or prepare-then-query lifecycles at the contract surface.
  2. **Full violation list with canonical ordering.** `Decision.reasons` is always a list ordered canonically cheapest-first (`BASELINE_ELIGIBILITY_FAIL`, `SAME_DAY_HARD_BLOCK`, `SAME_DAY_ALREADY_HELD`, `UNIT_ALREADY_FILLED`, `BACK_TO_BACK_CALL`). When `valid = false`, `reasons` MUST include every applicable violation; partial violation lists are non-compliant. List length carries meaningful information (exactly the applicable-violation count) and is not implementation-defined.
  3. **First-release hard-rule enumeration with stable codes** aligned to `docs/domain_model.md` §9.2.
  4. **Scoped fixed-assignment handling.** Fixed assignments admitted at parser boundary are facts in `ruleState`; the rule engine never re-adjudicates them against themselves, but downstream-validity checks (including `BACK_TO_BACK_CALL` against fixed neighbors) fire on any non-fixed `proposedUnit` exactly as they would against solver-placed assignments. The parser-stage exception does not widen to other placements.
  5. **Separation from soft-effect evaluation.** The rule engine handles hard validity only. Soft effects (`prevDayCallSoftPenaltyTrigger`, `callPreferencePositive`) are read by the scorer directly from `DailyEffectState`; the rule engine does not expose a "what triggers are active" query. See D-0025 and D-0027.
  6. **Equivalence-test discipline for future non-stateless implementations.** Any future incremental-state implementation MUST reproduce the stateless reference's `Decision` outputs byte-identically (including canonical ordering) against a contract-owned fixture corpus that covers at minimum each hard invariant and the `FixedAssignment` scoped admission cases. The corpus is contract-owned, not implementation-owned, and divergences are contract-breaking defects.
- **Rationale:** A stateless surface is the cheapest thing to review, test, and re-implement. Every prepare-then-query lifecycle the rule engine could expose would leak into solver strategy code and constrain future strategies. Full-list violations preserve the information operators and diagnostic tooling will eventually want (rejection-reason distributions for benchmark campaigns; cf. FW-0012); the per-rule checks are each O(1) or O(small) lookups against indexed state, so running the full rule set on every invalid probe is negligible at first-release scale — the design tradeoff that would have motivated a short-circuit optimization does not materialize. Pulling soft effects fully out of rule-engine scope prevents the subtle drift where "this trigger applies on this date" ends up with two authoritative sources — one in the rule engine's internal state and one in the scorer's `DailyEffectState` read — which is how v1-style hidden coupling happens. The equivalence-test discipline is the narrow gate that lets a faster implementation ship without the reference becoming a "legacy path nobody maintains."
- **Consequences:** `docs/rule_engine_contract.md` is accepted as the normative boundary. `docs/solver_contract.md` (D-0026) consumes the rule engine through this interface only; `docs/scorer_contract.md` (D-0025) reads `DailyEffectState` directly rather than routing soft-effect queries through the rule engine. Any first-release implementation ships the stateless reference; incremental-state work is parked in FW-0003 under the equivalence-test gate. The rule-engine contract does not yet prescribe concrete API signatures or module decomposition — those are implementation-slice concerns.
- **Follow-up actions:** During rule-engine implementation, author the minimum equivalence-test corpus (at least one fixture per hard invariant plus `FixedAssignment` scoped admission cases) so FW-0003 can be triggered later without scrambling to build the corpus. Keep `docs/rule_engine_contract.md` aligned with any future changes to `docs/domain_model.md` §9.2 invariant enumeration.
- **Related docs:** `docs/rule_engine_contract.md`, `docs/domain_model.md` §9.2, §10.1, `docs/parser_normalizer_contract.md` §14, `docs/blueprint.md` §5 and §7.4.

### D-0025: Scorer architecture contract — pure function, required component breakdown, direction-guard invariant, `crReward` diminishing marginal utility
- **Date:** 2026-04-23
- **Status:** Accepted
- **Context:** M2 needed a normative scorer contract before compute-pipeline implementation started, for two reasons. First, several scorer-shape questions were live in parallel with the rule-engine work: whether scoring should be a pure function of `(allocation, normalizedModel, scoringConfig)` or a stateful service; whether `ScoreResult` should carry only `totalScore` or mandate a component breakdown; how the `HIGHER_IS_BETTER` direction invariant should be enforceable as a property test rather than enforced by convention; how `crReward` should behave to avoid a single doctor hoarding all honored `CR` requests without bolting on a separate fairness component; and whether streaming/delta scoring should be permitted alongside the pure-function reference. Second, the scorer's relationship to soft effects interlocks with D-0024 and D-0027 — `prevDayCallSoftPenaltyTrigger` and `callPreferencePositive` could live either in the rule engine (validated alongside hard rules) or in the scorer (read directly from `DailyEffectState`), and the placement matters for whether "apply this soft effect" is a lookup or a delegation.
- **Decision:** Adopt `docs/scorer_contract.md` as the normative scorer boundary with the following coherent set of decisions:
  1. **Pure-function public surface.** Scorer is `score(allocation, normalizedModel, scoringConfig) → ScoreResult`. No lifecycle, no side effects, no reads outside declared inputs.
  2. **Required component breakdown on every `ScoreResult`.** Every first-release component identifier from `docs/domain_model.md` §11.2 (`unfilledPenalty`, `pointBalanceWithinSection`, `pointBalanceGlobal`, `spacingPenalty`, `preLeavePenalty`, `crReward`, `dualEligibleIcuBonus`, `standbyAdjacencyPenalty`, `standbyCountFairnessPenalty`) MUST appear in `ScoreResult.components`, even when it contributes zero. Returning only `totalScore` is a contract violation.
  3. **`HIGHER_IS_BETTER` direction-guard invariant.** For any valid allocation `A` with score `S1`, converting one filled `AssignmentUnit` to unfilled produces allocation `A'` with score `S2` where `S2.totalScore ≤ S1.totalScore`. This invariant is property-testable and MUST be exercised in any implementation's test suite. It guards against silent direction inversions that would otherwise pass integration tests.
  4. **`crReward` diminishing marginal utility per doctor.** For any doctor `d`, the kth honored `CR` (k ≥ 2) contributes strictly less reward than the (k−1)th. The first honored `CR` for a doctor contributes the maximum. Exact curve shape is an implementation detail; the strict-monotonic-decrease property is contractually fixed.
  5. **Scorer-owned soft-effect reading.** Scorer reads `DailyEffectState` directly. The rule engine does not mediate soft-effect reads (see D-0024 and D-0027).
  6. **Operator-tuneable weights via sheet (v1 parity).** Component weights are extracted at parser boundary and flow as `scoringConfig.weights`, overriding template defaults at run time. Operator values MUST preserve per-component sign orientation (a reward component stays a reward regardless of the numeric weight).
  7. **Streaming/delta scoring permitted as optimization.** Implementations MAY ship a streaming implementation provided it produces byte-identical `ScoreResult` values to the pure-function evaluation within a single implementation on a single platform, and does not leak lifecycle state into the public contract. Byte-identical is the single normative parity criterion; cross-implementation or cross-platform parity is not guaranteed (see FW-0011).
- **Rationale:** A pure-function contract is the scorer's equivalent of the rule engine's statelessness: it makes the scorer the easiest thing to review, test, and re-implement, and it keeps it safe to call from any solver strategy (including future score-aware ones via the `scoringConsultation` extension clause). Mandating the component breakdown on every result converts "what does this score mean" from oral tradition into a contract-enforced artifact, which is the only way benchmark campaigns (FW-0015) will be able to compare strategies meaningfully. The direction-guard invariant operationalizes `HIGHER_IS_BETTER` against the specific class of bugs — sign errors, accidental penalty inversions, component swaps — that cost real engineering hours in v1. The `crReward` diminishing-marginal-utility curve solves the "one doctor hoards all honored `CR`" failure mode with a single well-specified component property, rather than requiring a separate fairness component to hold the line; the solver's `CR_MINIMUM_PER_DOCTOR` seeding (D-0026) complements this from the generation side. Permitting streaming as an optimization leaves a clear path to FW-0010 without making first-release implementations pay for it.
- **Consequences:** `docs/scorer_contract.md` is accepted as the normative boundary. Concrete weight values and `crReward` curve shape are explicitly deferred to the scorer implementation slice (FW-0014 queues the v1 reference pass). Operator-tuneable curve parameters beyond weights are deferred to FW-0007. Per-unit-position fairness (`unitPositionFairnessPenalty`) and `workloadWeight` on `SlotTypeDefinition` are queued as FW-0008 and FW-0009 respectively. Blueprint §16's current "routine variation" wording is narrower than this reality and receives a clarifying patch in the same change round as this decision (D-0028).
- **Follow-up actions:** During scorer implementation, author a direction-guard property test and a component-breakdown schema test so the two contract-level invariants are machine-enforced from day one. Reference v1's effective weights as FW-0014 describes.
- **Related docs:** `docs/scorer_contract.md`, `docs/domain_model.md` §11, `docs/request_semantics_contract.md` §10, `docs/blueprint.md` §5, §7.6, §16.

### D-0026: Solver architecture contract — scoring-blind, strategy-pluggable, whole-run failure, `maxCandidates` termination, retention moved downstream
- **Date:** 2026-04-23
- **Status:** Accepted
- **Context:** M2 needs a normative solver boundary alongside the rule-engine and scorer contracts (D-0024, D-0025) so implementation planning for the minimal local compute pipeline can proceed against a stable search-stage shape. Several design questions were live. First, whether the solver should be scoring-aware from day one (cheaper path to quality but harder to reason about and benchmark) or scoring-blind (matches blueprint §16 and isolates candidate generation from ranking). Second, whether the contract should admit one strategy or define a strategy-pluggability surface that can absorb future score-aware strategies (hill-climb, simulated annealing, beam search, parallel seeded-merge, constraint propagation, CP-SAT) without a contract rewrite. Third, how `CR` seeding and fill order should be specified: v1's effective behavior was "prefer to honor `CR`s first, then fill tightest slot first," and the checkpoint needed to make that explicit in contract language rather than relying on implementation lore. Fourth, how to compute the `CR` floor `X` — a fixed operator input, a smart default derived from the input distribution, or both. Fifth, termination: whether the first release admits wall-clock bounds (convenient for long runs but breaks byte-identical determinism) or candidate-count bounds only. Sixth, what happens when a slot is unfillable — partial allocation with unfilled markers, or whole-run failure. Seventh, where retention lives: in the solver (as v1 did) or downstream at a selector stage (which is what `docs/domain_model.md` §12 already anticipates). The answers to these are not independent; they interact with D-0024, D-0025, and D-0027.
- **Decision:** Adopt `docs/solver_contract.md` as the normative solver boundary with the following coherent set of decisions:
  1. **Scoring-blind public surface.** Solver is `(normalizedModel, ruleEngine, seed, fillOrderPolicy, terminationBounds, preferenceSeeding) → CandidateSet | UnsatisfiedResult`. The solver MUST NOT consume scorer, scoring config, or any soft-effect magnitude. Emitted `TrialCandidate` entries carry no score.
  2. **Strategy-pluggable interface with additive extension clause.** A strategy is identified by `strategyId` and described by a `StrategyDescriptor`. First release ships exactly `SEEDED_RANDOM_BLIND`. Future strategies MAY declare additional strategy-specific inputs and MAY opt into a read-only scoring-oracle handle (`scoringConsultation: "READ_ONLY_ORACLE"`), but the extension is additive only and never overrides scorer/rule-engine/retention-owned logic.
  3. **First-release composite `SEEDED_RANDOM_BLIND`.** Two phases: `CR_MINIMUM_PER_DOCTOR` preference seeding (Phase 1; best-effort; below-floor outcomes accepted; CR-vs-fixed-assignment conflicts skip the CR) + `MOST_CONSTRAINED_FIRST` fill (Phase 2; tightest-slot-first; tie-breaking seeded-deterministic).
  4. **`crFloor` computation.** `mode ∈ {"SMART_MEDIAN", "MANUAL"}`. `SMART_MEDIAN` default computes `X = floor(median(CR-count-per-doctor))` over the full doctor set (including zero-CR doctors). `MANUAL` uses the operator-supplied non-negative integer. `X = 0` effectively disables Phase 1. Computed `X` MUST be logged in `SearchDiagnostics` at run start — required for audit because `SMART_MEDIAN`'s value depends on input data.
  5. **Whole-run failure on any unfillable slot.** When any non-fixed demand unit cannot be filled under rule-engine validity within the declared termination bounds, the solver returns `UnsatisfiedResult` (with `unfilledDemand` identifying the offending units and `reasons` giving structured explanations). No partial allocations are emitted.
  6. **`maxCandidates`-only termination.** First-release `terminationBounds` is exactly `maxCandidates` (required, positive integer). No time budget. Strategies MUST NOT consult wall-clock time.
  7. **Byte-identical determinism within a single implementation on a single platform.** Randomized decisions derive exclusively from `seed`. Cross-implementation determinism is not guaranteed (FW-0011).
  8. **Retention moved downstream to the selector stage.** Solver does not implement retention modes. It emits every valid candidate up to `maxCandidates`. `TrialBatchResult` best-candidate fields are populated retroactively by the selector after scoring.
  9. **Operator-tuneable surface.** `crFloor.manualValue` is operator-tuneable (v1 parity); the scorer's component weights (D-0025) are separately operator-tuneable.
- **Rationale:** A scoring-blind solver with a strategy-pluggable interface is the cleanest way to keep first-release search reviewable while leaving room for the full slate of future strategies listed in FW-0004 and FW-0005 without a contract rewrite. The additive extension clause is specifically narrower than "strategies can do whatever they want": future score-aware strategies get a read-only oracle handle but never take over scorer-component responsibility, which is the specific coupling v1 had that made it hard to swap strategies cleanly. `CR_MINIMUM_PER_DOCTOR` plus `MOST_CONSTRAINED_FIRST` is the direct specification of what v1's effective seeded search did in practice — the contract is catching up to observed behavior and making it testable. `SMART_MEDIAN` as the default `crFloor` lets most operator inputs be self-tuning without a `MANUAL` override, while keeping `MANUAL` available when an operator has an informed preference. Whole-run failure on any unfillable slot is stricter than v1's behavior; it trades late-discovered operator frustration for upfront failure clarity, which aligns with blueprint §5 ("If no valid candidate exists, the system must report that clearly (not silently degrade)."). `maxCandidates`-only termination matches the determinism discipline in §16 — wall-clock bounds would make byte-identical reproducibility impossible, which would undermine both D-0025's direction-guard invariant testing and FW-0015's benchmark-campaign intent. Moving retention downstream to the selector is the v2 departure that matters most here: in v1, retention entangled with search made it hard to benchmark strategies on candidate-generation quality alone, and factoring it out lets FW-0013's richer retention modes land without touching solver code.
- **Consequences:** `docs/solver_contract.md` is accepted as the normative boundary. First-release implementation plan is `SEEDED_RANDOM_BLIND` only; FW-0004 and FW-0005 queue score-aware and parallel strategies behind the extension clause. The selector stage is now an explicit contract-closure item whose contract is not yet drafted; its first-release scope is `BEST_ONLY` as the default retention mode plus `FULL` retention as a per-run operator opt-in for ad-hoc auditability, with `TOP_K`, `FULL_WITH_DIAGNOSTICS`, and per-batch artifact export deferred to FW-0013 benchmark-campaign work. Benchmark campaign mode interactions with the solver are deferred to FW-0015. Blueprint §16 receives a clarifying patch (D-0028) so the operator-tuneable surface wording reflects `crFloor` alongside scorer weights.
- **Follow-up actions:** During solver implementation, author seeded-determinism tests that confirm byte-identical outputs under identical `(seed, normalizedModel, ruleEngine, fillOrderPolicy, terminationBounds, preferenceSeeding)` inputs. Implement the `SearchDiagnostics` logging for `strategyId`, `fillOrderPolicy`, `crFloorComputed`, `crFloorMode`, and `seed` at run start.
- **Related docs:** `docs/solver_contract.md`, `docs/rule_engine_contract.md`, `docs/scorer_contract.md`, `docs/domain_model.md` §10.1, §12, `docs/blueprint.md` §5, §7.5, §16.

### D-0027: Pipeline stage separation — three-stage `solver → scorer → selector`, retention with selector, scorer reads soft effects directly
- **Date:** 2026-04-23
- **Status:** Accepted
- **Context:** D-0024, D-0025, and D-0026 each fix a stage boundary, but the overall pipeline composition is itself a decision that deserves its own record. Specifically: is the pipeline two-stage (solver emits ranked candidates; retention with solver) or three-stage (solver emits unranked candidates; scorer ranks; selector applies retention)? And where do soft effects get read — inside the rule engine (symmetric with hard rules) or inside the scorer (direct read from `DailyEffectState`)? These two questions are not independent of D-0024/D-0025/D-0026 — they close out the last piece of the first-release pipeline shape.
- **Decision:** Adopt the following pipeline-composition decisions:
  1. **Three-stage pipeline `solver → scorer → selector`.** Solver emits unscored `CandidateSet` (or `UnsatisfiedResult`). Scorer produces one `ScoreResult` per candidate. Selector applies retention policy over scored candidates and produces the final `AllocationResult` (plus any retained artifacts).
  2. **Retention is owned by the selector stage, not by the solver.** The solver's retention boundary (`docs/solver_contract.md` §14) is specifically about this: first release ships `BEST_ONLY` as the default retention mode at the selector plus `FULL` retention as a per-run operator opt-in for ad-hoc auditability. `TOP_K` and `FULL_WITH_DIAGNOSTICS` modes, and per-batch artifact export formats, are deferred to FW-0013 benchmark-campaign work.
  3. **Solver is scoring-blind.** It does not consult scorer, scoring config, or soft-effect magnitude. See D-0026.
  4. **Scorer reads `DailyEffectState` directly from `normalizedModel`.** The rule engine does not expose a "what soft triggers are active on this date" query. The rule engine handles hard validity only; soft effects (`prevDayCallSoftPenaltyTrigger`, `callPreferencePositive`) are the scorer's concern. See D-0024 and D-0025.
- **Rationale:** The three-stage composition is what lets each stage be a pure function of its inputs (see the statelessness / pure-function discipline in D-0024 and D-0025) without any one stage needing to reach across boundaries. The specific retention placement is the v2 correction to v1's search-retention entanglement: putting retention with the selector lets benchmark campaigns (FW-0015) compare strategies on candidate-generation quality alone and then compare retention modes on scoring quality alone. The direct soft-effect read by the scorer avoids the specific drift where "what triggers apply here" ends up with two authoritative sources — if the rule engine also surfaced soft-effect information, scorers would eventually call into that surface and soft-effect evaluation would drift into a de facto shared responsibility. One authoritative soft-effect reader (the scorer) prevents that drift by construction.
- **Consequences:** The selector stage is an explicit contract-closure item for a later checkpoint; its contract is not drafted in this round. First-release selector scope is `BEST_ONLY` default retention plus `FULL` retention as a per-run operator opt-in over scored candidates (`docs/domain_model.md` §12.5), with `TOP_K`, `FULL_WITH_DIAGNOSTICS`, and per-batch artifact export formats deferred to FW-0013 benchmark-campaign work. All three stages in this round commit to pure-function / stateless public surfaces, which composes cleanly and is individually easier to test. Soft-effect evaluation is not split across the rule engine and scorer, so adding a new soft effect (beyond `prevDayCallSoftPenaltyTrigger` and `callPreferencePositive`) is a scorer change only.
- **Follow-up actions:** Author `docs/selector_contract.md` when the selector stage is scoped for implementation closure (likely in a subsequent checkpoint of M2 or under M3). Keep this decision as the anchor for why the selector exists as a distinct stage rather than as a solver sub-component or scorer add-on.
- **Related docs:** `docs/rule_engine_contract.md`, `docs/scorer_contract.md`, `docs/solver_contract.md`, `docs/domain_model.md` §8.2, §11, §12.

### D-0029: `unitIndex` operational-equivalence — same-`SlotType` units are equivalent in doctor admissibility and baseline workload weight
- **Date:** 2026-04-23
- **Status:** Accepted
- **Context:** During M2 C1 contract design (D-0024 / D-0025 / D-0026), the question came up of how to model multi-unit demand for departments where one `(dateKey, slotType)` has `requiredCount > 1` (for example, a future department needing 10 medical-officer call slots per day, operationally numbered MO1..MO10 on the sheet). An earlier draft suggested that semantically-distinct positions could be modeled as distinct `unitIndex` values on the same `SlotType` with potentially differential weight ("MO1 is 10% heavier than MO5"). That suggestion was rejected on operational grounds: assigning differential per-position weight is arbitrary and unjustifiable in real rostering practice; what operators actually want is even *distribution* across positions over time, not per-position difficulty differentiation. ICU/HD first release does not exercise this case at all (`requiredCount = 1` everywhere), but the contract direction needs to be settled before any future multi-unit department lands so the architecture does not drift into per-position weighting on the way in.
- **Decision:** All `AssignmentUnit` entries that share the same `(dateKey, slotType)` pair (i.e., differ only by `unitIndex`) are **equivalent for doctor-admissibility rules** — the four doctor-admissibility hard invariants (`BASELINE_ELIGIBILITY_FAIL`, `SAME_DAY_HARD_BLOCK`, `SAME_DAY_ALREADY_HELD`, `BACK_TO_BACK_CALL` per `docs/rule_engine_contract.md` §11) and for baseline workload weight. The fifth hard invariant, `UNIT_ALREADY_FILLED`, is explicitly NOT part of this equivalence: it is per-unit occupancy bookkeeping keyed on the full `(dateKey, slotType, unitIndex)` identity and correctly branches on `unitIndex` by construction. `unitIndex` is a stable operational identity that supports writeback labeling (for example, "MO1", "MO2", "MO3" mapping to `unitIndex = 0, 1, 2`) but MUST NOT carry implicit difficulty or workload differentiation in any contract. Concretely:
  1. **Doctor-admissibility equivalence.** For any `(dateKey, slotType, doctorId)` triple and any `ruleState`, the rule engine's doctor-admissibility hard invariants — `BASELINE_ELIGIBILITY_FAIL`, `SAME_DAY_HARD_BLOCK`, `SAME_DAY_ALREADY_HELD`, `BACK_TO_BACK_CALL` (`docs/rule_engine_contract.md` §11) — MUST evaluate identically regardless of which `unitIndex` the `proposedUnit` refers to. These four rules govern whether a given doctor can be placed on a given `(dateKey, slotType)` at all; they do not branch on `unitIndex`. The fifth hard invariant, `UNIT_ALREADY_FILLED`, is NOT a doctor-admissibility rule — it is per-unit occupancy bookkeeping keyed on the full `(dateKey, slotType, unitIndex)` identity, and it correctly branches on `unitIndex` by construction (that is exactly how `requiredCount > 1` multiplicity is tracked). The equivalence property does not relax per-unit occupancy; a unit that is filled is filled, and its `unitIndex` identity is how that occupancy is recorded.
  2. **Baseline workload-weight equivalence.** The scorer MUST treat all units of the same `SlotType` as carrying identical baseline workload weight. Per-position fairness across `unitIndex` values is a future additive scoring component (FW-0008), not a `unitIndex` weight; per-`SlotType` burden weighting is a future additive `SlotTypeDefinition` field (FW-0009 / `docs/domain_model.md` §7.6).
  3. **Modeling boundary.** Departments that genuinely need semantically-differentiated roles MUST model them as distinct `SlotType` identities (for example, separate `MO_PRIMARY_CALL` and `MO_BACKUP_CALL` slot types), not as distinct `unitIndex` values of the same `SlotType`. Adding semantic meaning to `unitIndex` itself is contract-prohibited and would constitute a breaking shape change to `docs/domain_model.md` §10.2.
  4. **Solver canonical fill-order for multi-unit demand is deferred.** The first multi-unit department will need a canonical determinism rule for which unfilled `unitIndex` to fill first within a `(dateKey, slotType)` (FW-0017 captures this). The equivalence in this decision does not by itself fix the determinism rule; it only fixes that the rule does not turn on weight differentiation.
- **Rationale:** Operationally, when a department writes "MO1, MO2, MO3" on the roster, those numbered positions are operationally identifiable but architecturally interchangeable — the doctor on MO1 is not doing a heavier or lighter shift than the doctor on MO5; the numbers are sequencing labels, not workload signals. Trying to encode per-`unitIndex` weight would create unjustifiable knobs (what is the actual weight ratio between MO1 and MO5? nobody can defensibly say) and would push the rostering problem into per-position fairness logic that no operator is asking for. Distribution fairness — "doctor X has been MO5 six times this month while doctor Y has been MO5 zero times" — is a real operator concern, but it belongs in a fairness scoring component (FW-0008), not in baseline weight differentiation. Forcing genuinely-different roles to be distinct `SlotType` identities also keeps the eligibility / hard-rule surface honest: if MO_PRIMARY and MO_BACKUP have different eligibility groups or different rules, those differences live in the template's slot/group declarations, not buried inside `unitIndex` semantics.
- **Consequences:** `docs/domain_model.md` §10.2 is patched in the same change round as this decision to state the equivalence property in its Notes section. `docs/rule_engine_contract.md` §11 inherits the equivalence — its four doctor-admissibility invariants (`BASELINE_ELIGIBILITY_FAIL`, `SAME_DAY_HARD_BLOCK`, `SAME_DAY_ALREADY_HELD`, `BACK_TO_BACK_CALL`) do not branch on `unitIndex`, while `UNIT_ALREADY_FILLED` correctly branches on `unitIndex` by construction as per-unit occupancy bookkeeping — and does not require a contract change. `docs/scorer_contract.md` baseline workload-weight components inherit the equivalence and likewise do not require a contract change. FW-0008 (per-unit-position fairness) and FW-0009 (`workloadWeight` on `SlotTypeDefinition`) remain the future-work surfaces for multi-unit fairness and per-`SlotType` burden weighting respectively. FW-0017 is added in the same change round to capture the still-open canonical-fill-order question for multi-unit demand once it materializes. ICU/HD first release is unaffected operationally — `requiredCount = 1` everywhere means `unitIndex` is always 0 — but the architectural direction is settled so future multi-unit departments do not have to relitigate it.
- **Follow-up actions:** When the first multi-unit department template is being authored, point template authors at this decision and at FW-0017. If a department case ever emerges where per-`unitIndex` differentiation appears genuinely necessary, the right move is to revisit by splitting into distinct `SlotType` identities (or by introducing the `workloadWeight` extension per FW-0009), not by reopening this decision.
- **Related docs:** `docs/domain_model.md` §7.7, §10.2; `docs/rule_engine_contract.md` §11; `docs/scorer_contract.md` §10, §11; `docs/future_work.md` FW-0008, FW-0009, FW-0017.

### D-0030: Selector architecture contract — pluggable, `HIGHEST_SCORE_WITH_CASCADE` first release, `BEST_ONLY` default + `FULL` operator opt-in retention, execution-layer-supplied `runId`
- **Date:** 2026-04-25
- **Status:** Accepted
- **Context:** M2 C1 closed the rule-engine, scorer, and solver boundaries (D-0024, D-0025, D-0026) and committed the pipeline-stage separation `solver → scorer → selector` (D-0027). The selector stage was named there but not contract-closed — D-0027 sub-decision 2 left first-release retention scope locked at `BEST_ONLY` default plus `FULL` operator opt-in but explicitly deferred selector-contract drafting to a subsequent checkpoint. Several selector-shape questions were live for M2 C2. First, whether the selector should expose a single fixed selection rule or a strategy-pluggable interface that can absorb future selection strategies (fairness-biased, multi-objective Pareto, operator-preferred-candidate) without a contract rewrite. Second, the first-release strategy's tie-break behavior — pure `totalScore` would deadlock on ties without a deterministic disambiguation rule, and the cascade choice (which components to consult, in what order) needed to be explicit so the resulting selection is auditable. Third, the precise output shapes for `BEST_ONLY` versus `FULL` retention, including whether `FULL` retention writes sidecar files and what those files contain. Fourth, how the failure branch (`UnsatisfiedResult` from the solver) interacts with retention mode — specifically whether `FULL` should produce a degenerate empty sidecar file or output identical content to `BEST_ONLY`. Fifth, how the `(runId, candidateId)` traceability identity composes — specifically whether `runId` is selector-synthesized or execution-layer-supplied, and how run-level metadata flows through the final result without per-candidate duplication. Sixth, how `TrialBatchResult` retroactive population (named in D-0026 sub-decision 8 but never operationalized) should be specified, including what per-batch score-distribution summary the selector emits.
- **Decision:** Adopt `docs/selector_contract.md` as the normative selector boundary with the following coherent set of decisions:
  1. **Pluggable strategy interface.** Selector strategies are pluggable from day one behind a stable `StrategyDescriptor` shape that mirrors the solver's pattern (`docs/solver_contract.md` §11). Future strategies register additively; new mutation channels into scorer or rule-engine state require a `contractVersion` bump but new strategy registrations conforming to the existing interface do not.
  2. **First-release strategy `HIGHEST_SCORE_WITH_CASCADE`.** Selects the candidate with the highest `totalScore`. On ties, applies a deterministic two-level cascade tie-break: first prefer higher `pointBalanceGlobal` (less-negative penalty), then prefer higher `crReward` (more-positive reward). Final fallback is the lowest `candidateId`, which guarantees full determinism in the (vanishingly rare) total-cascade-tie case. Cascade depth is exactly two; deeper components from `docs/domain_model.md` §11.2 are not consulted under this strategy.
  3. **Retention modes `BEST_ONLY` (default) + `FULL` (opt-in).** First-release retention surface is exactly two modes. `TOP_K` and `FULL_WITH_DIAGNOSTICS` are deferred to FW-0013 Phase 2 (benchmark-campaign work).
  4. **Per-mode output shapes.** `BEST_ONLY` carries `winnerAssignment` + `winnerScore` (full component breakdown by D-0025) + `searchDiagnostics` + the run envelope; no sidecar files. `FULL` carries everything from `BEST_ONLY` plus full `SearchDiagnostics` chain inline plus `candidatesSummaryPath` and `candidatesFullPath` sidecar pointers.
  5. **Sidecar file shapes.** `candidates_summary.csv` (one row per retained candidate; columns `candidateId`, `totalScore`, every named component score from `docs/domain_model.md` §11.2, plus run-level metadata `runId`, `seed`, `batchId`) and `candidates_full.json` (full `AssignmentUnit[]` per candidate indexed by `candidateId`, plus `ScoreResult` per candidate). Both files declare `schemaVersion: 1` and embed `runId` so retained artifacts are unambiguously traceable when found out of context. Both are cross-referenced by `candidateId`.
  6. **`UnsatisfiedResult` handling: identical output regardless of retention mode.** On the failure branch, the selector returns an `UnsatisfiedResultEnvelope` carrying `unfilledDemand`, `reasons`, and aggregated `searchDiagnostics`; no sidecar files are written under `FULL` because there are no candidates to retain (the solver did not emit any). Forcing a degenerate empty sidecar file under `FULL` would produce operator-facing noise without improving auditability.
  7. **Run envelope and traceability.** The full identity of any retained candidate is the pair `(runId, candidateId)`. `runId` is execution-layer-supplied and the selector MUST NOT synthesize it; absence at selector entry is a contract-breaking defect on the caller side. `candidateId` is a run-monotonic integer scoped to that run, assigned in solver-emission order. All run-level metadata (`snapshotRef`, `configRef`, `seed`, `fillOrderPolicy`, `crFloorMode`, `crFloorComputed`, `generationTimestamp`) flows once on the run envelope at the `FinalResultEnvelope` level, not repeated per candidate. `generationTimestamp` is execution-layer-supplied for the same reason as `runId`: the selector MUST NOT consume clocks (`docs/selector_contract.md` §18), so any timestamp embedded in retained sidecar artifacts (`candidates_full.json` per `docs/selector_contract.md` §14.2) must arrive on the run envelope rather than be synthesized at write time. Future expansion of run-level metadata fields is additive and does not require a `contractVersion` bump.
  8. **`TrialBatchResult` retroactive population.** Selector populates `TrialBatchResult.bestCandidate` (operationalizing D-0026 sub-decision 8) and a per-batch score-distribution summary. The summary covers `min`, `max`, `median`, `mean`, `stddev` for `totalScore` plus `min`, `max`, `median` for every first-release component identifier in `docs/domain_model.md` §11.2 — the "per-component min/max/median" option chosen over `totalScore`-only to support per-component retention diagnostics without requiring `FULL` retention.
  9. **Full-set ingest in first release.** Selector consumes the full `scoredCandidateSet` at once; streaming ingest (selector that processes candidates as they are scored) is deferred to FW-0018. First-release retention volumes are within memory bounds at ICU/HD scale (5,000 candidates × ~120 `AssignmentUnit` entries each is MB-scale).
  10. **Pluggable selector strategy registry; future strategies additive without contract bump.** Future strategies (fairness-biased, multi-objective Pareto, operator-preferred-candidate) register as new `StrategyDescriptor` entries; the registration is additive only and never overrides scorer-component logic, retention-mode semantics, or determinism guarantees.
- **Rationale:** A pluggable strategy interface from day one is the cheapest path to absorbing future selector strategies without reopening this contract — exactly the same reasoning that motivated the strategy-pluggable solver in D-0026. `HIGHEST_SCORE_WITH_CASCADE` operationalizes the obvious "pick the highest score" rule with a deterministic disambiguation rule (cascade + `candidateId` fallback) so byte-identical determinism per §18 is reachable end-to-end. The cascade choices (`pointBalanceGlobal` then `crReward`) reflect operationally-meaningful preferences — when total scores tie, prefer rosters with less-negative aggregate penalty distribution and stronger honored-`CR` reward signal — without consulting deeper components that would make the tie-break harder to explain to operators. Retention scope `BEST_ONLY` + `FULL` (rather than ship-the-full-FW-0013-set) tracks the locked first-release scope from D-0027 sub-decision 2 and explicitly leaves richer modes for benchmark-campaign work where they have an operator-facing justification. The no-sidecar-on-failure rule prevents the specific operator-facing failure mode of "I asked for FULL retention and got an empty CSV" — the failure branch already carries the diagnostics that matter (`unfilledDemand`, `reasons`, aggregated `SearchDiagnostics`); a degenerate empty CSV would mislead more than inform. Execution-layer-supplied `runId` is what makes `(runId, candidateId)` actually composable across pipeline stages — if the selector synthesized `runId`, then upstream stages would have a different `runId` (or none) and trace-back across the run boundary would silently fragment. Per-component min/max/median in the per-batch summary is the option chosen because the `totalScore`-only alternative would lose the ability to spot batches where (for example) `pointBalanceGlobal` distribution is anomalous even though `totalScore` distribution looks normal — a degraded-signal failure mode that retention-stage diagnostics specifically exist to catch. Full-set ingest in first release matches the actual memory footprint at ICU/HD scale; streaming would add implementation complexity without operator-facing benefit at the volumes this release will see (FW-0018 captures the trigger to revisit).
- **Consequences:** `docs/selector_contract.md` is accepted as the normative boundary and closes M2 C2. The selector stage now has a contractual home for everything D-0026 and D-0027 deferred to it (retention scope, retroactive `TrialBatchResult` population, sidecar artifact shape). FW-0013 is updated to cross-reference the new contract and to mark Phase 1 as locked-by-contract; Phase 2 (`TOP_K`, `FULL_WITH_DIAGNOSTICS`, per-batch artifact export formats) remains deferred. FW-0018 is added in this change round for streaming-selector implementation. Concrete sidecar file paths and naming conventions remain execution-layer concerns (this contract deliberately does not pin them), so the future execution layer is free to embed timestamps or run prefixes in file names without reopening the selector contract. First-release implementation plan ships `HIGHEST_SCORE_WITH_CASCADE` only; alternative selector strategies remain future-work surface registrations rather than contract-version bumps.
- **Follow-up actions:** During selector implementation, author byte-identical determinism tests under fixed inputs (mirroring the solver's seeded-determinism discipline in D-0026) to confirm both `BEST_ONLY` and `FULL` outputs are reproducible. Confirm that `candidates_summary.csv` and `candidates_full.json` round-trip cleanly through the chosen serialization library on a representative ICU/HD-sized candidate set so `schemaVersion: 1` is genuinely a stable artifact format and not a label on a serialization that drifts run-to-run.

### D-0031: Writeback architecture contract — boundary, branch-on-write semantics, idempotency discipline, doctor-identity resolution, failure-branch rendering, source-sheet identity propagation, stack ownership
- **Date:** 2026-04-26
- **Status:** Accepted
- **Context:** M2 C1 closed the rule-engine, scorer, and solver boundaries (D-0024, D-0025, D-0026), committed the pipeline-stage separation `solver → scorer → selector` (D-0027), and M2 C2 closed the selector boundary including the `FinalResultEnvelope` shape and the run-envelope additivity clause (D-0030). The selector's `FinalResultEnvelope` is the canonical pipeline output, but where it lands operator-facing — the writeback boundary onto the operator's Google Sheets shell that M1 generated — was named at `docs/domain_model.md` §14 only as "an adapter concern" and had no normative contract. Several writeback-shape questions were live. First, stack ownership: writeback IS sheet-facing (so D-0017 / D-0018 point at Apps Script) but consumes a Python-stage output (which would suggest direct Python-to-Sheets writeback if not for the OAuth-surface and stack-split discipline). Second, conflict policy when the operator has edited the source tab between run-start and writeback (detect-and-refuse vs detect-and-warn vs blind-overwrite vs branch-on-write). Third, doctor identity resolution from `AssignmentUnit.doctorId` (runtime identity) to displayed cell value, including how that resolution interacts with operator edits to column A mid-flow. Fourth, atomicity discipline if the writeback dies mid-write of the new tab. Fifth, idempotency on repeated envelope upload — strict skip-on-`runId`-match vs respect-operator-agency new-tab-per-upload. Sixth, run-envelope traceability content embedded in the writeback artifact. Seventh, the launcher-facing diagnostic surface for success / failure / runtime-error states. Eighth, source-sheet identity propagation — whether `sourceSpreadsheetId` and `sourceTabName` should land on the snapshot (touching `snapshot_contract.md`), on the run envelope (exercising `docs/selector_contract.md` §16.3 additivity), or on a writeback-only wrapper. Ninth, delivery-plan framing: M3's `Status: Planned` and stated `Dependencies: M2 complete` make activating M3 today (while M2 is mid-flight) a process-consistency problem, but the writeback contract surface is ready to draft scope-ahead while the selector contract is still fresh in context. Tenth, failure-branch rendering: whether a richer diagnostic surface (banners, snapshot reconstruction, `searchDiagnostics` aggregation tables) belongs in the failure tab or whether a deliberately-crude minimum content is the right first-release scope. The answers to these are coupled — branch-on-write semantics dissolve the conflict policy question; respect-operator-agency idempotency simplifies the writeback path; source-sheet identity on the run envelope simplifies launcher UX — and the eleven coupled answers below land as one architecture-level decision.
- **Decision:** Adopt `docs/writeback_contract.md` as the normative writeback boundary with the following coherent set of decisions:
  1. **Stack ownership: Apps Script consumes serialized envelope.** Writeback runs on the Apps Script platform alongside the M1 sheet-generation surface, consuming a serialized JSON envelope produced by the local-first Python compute pipeline upstream (`docs/writeback_contract.md` §6.1). Apps Script is the sole writer to operator-owned spreadsheets; the Python core does not hold Sheets credentials and does not write to operator-owned spreadsheets directly. This preserves the D-0017 / D-0018 stack split, inherits the M1.1 OAuth-scope discipline (D-0023), and keeps a clean credential-isolation boundary in place for any future cloud-orchestrated Python compute mode (forward-compatibility note, not a committed direction). Concrete transport mechanics for the JSON envelope (paste, file upload, Drive Picker, alternative) and the JSON's concrete schema layout are deferred as implementation-slice concerns (`docs/writeback_contract.md` §6.3, §8); the contract pins the categories of content the JSON MUST carry but does not pin schema layout, mirroring the `docs/selector_contract.md` §14.3 pattern.
  2. **Delivery-plan framing: scope-ahead, M2 stays Active, M3 stays Planned, telegraph M3 sub-bullet.** The writeback contract is drafted scope-ahead of M3 activation. M2 (`Minimal local compute pipeline`) remains the active milestone in `docs/delivery_plan.md` §6; M3 (`Safe result/output and writeback`) remains `Planned` in §5. A sub-bullet under M3's likely-checkpoints in §5 telegraphs that writeback contract closure is drafted scope-ahead in this PR and remains the natural M3 C1 once M3 activates. §10 deferred-list gains a clarifying line acknowledging the scope-ahead drafting; §12 change-log gains today's entry. §6 / §7 / §8 / §9 / §11 / §15 are untouched: M2 stays Active, no new active checkpoint is added, and the M2 C2 sign-off recording belongs to the in-flight selector closure rather than to this scope-ahead drafting. The cleanest precedent for scope-ahead contract drafting is how D-0027 named the selector stage on 2026-04-23 and PR #65 drafted `docs/selector_contract.md` on 2026-04-25; the writeback contract takes the same pattern at the M2 boundary.
  3. **Branch-on-write: always-new-tab semantics.** Every writeback invocation creates a new tab in the source spreadsheet (`docs/writeback_contract.md` §11). The source tab named by `runEnvelope.sourceTabName` MUST NOT be mutated by writeback under any circumstance. Tab name is `<sourceTabName>_RM<YYMMDDHHMMSS>` with second-resolution timestamp; auto-suffix `_2` / `_3` / … is required for collision uniqueness on the rare same-second collision. The writeback tab is read-only from the operator's perspective (whole-tab protection). The JSON envelope carries a literal cell-data snapshot of the source tab at run-start (column-A doctor names, request cells, call-point cells, prefilled fixed-assignment cells, plus shell parameters `department`, `periodStartDate`, `periodEndDate`, `doctorCountByGroup`) so Apps Script can regenerate the structural surfaces via existing M1 code. No bloat mitigation in first release (operator manages cleanup of accumulated writeback tabs). Run-envelope traceability content lands in BOTH a visible footer block AND hidden developer metadata on the tab (sub-decision 9 below).
  4. **Doctor identity resolution: column-A from snapshot via `doctorIdMap`.** Writeback resolves `AssignmentUnit.doctorId` to the displayed cell value via the snapshot bundle's column-A cell values, mediated by a `doctorIdMap` from `doctorId` to `(sectionGroup, rowIndex)` per source-tab structure (`docs/writeback_contract.md` §12). The map is built at the parser/normalizer boundary upstream from the source tab's column-A cells and rides on the JSON envelope. The column-A cell value the operator typed at run-start is the single source of truth for the displayed doctor name; writeback does not consult the parser/normalizer's canonical doctor table, a sidecar metadata block on the run envelope, or the live source-tab column A at writeback time. Operator-edited column A between run-start and writeback does not propagate to the writeback tab — the writeback tab faithfully reflects run-start column-A content, and the source tab faithfully reflects what the operator currently has. Both representations coexist on different tabs.
  5. **Unfilled-unit cell representation: deferred to FW-0021.** First-release upstream contracts (`docs/solver_contract.md` §10, §14; D-0026 sub-decision 5) guarantee that every `AssignmentUnit` in a success-branch `winnerAssignment` has non-null `doctorId`. Writeback MUST NOT receive a null-`doctorId` `AssignmentUnit` on the success branch in first release. The "what does writeback render for null `doctorId`" question is for a case that cannot trigger first-release; explicitly defer cell representation for null `doctorId` to FW-0021, triggered when a future score-aware solver strategy (FW-0004) emits partial allocations or a future template lands with `requiredCount > 1` and a partial-fill strategy. The contract makes no normative statement about null-`doctorId` rendering.
  6. **Failure-branch behavior: branch new tab; very simple and crude minimum content.** When `finalResultEnvelope.result` is an `UnsatisfiedResultEnvelope` per `docs/selector_contract.md` §10.2, writeback follows the always-new-tab pattern (sub-decision 3) and emits a failure-branch tab with deliberately minimum content: a "FAILED" indicator at the top, `unfilledDemand` rendered as plain-text rows, `reasons` rendered as plain-text rows, plus the run-envelope traceability footer and hidden developer metadata with `Status: FAILED` / `status: "FAILED"`. Snapshot reconstruction in the failure tab, rich formatting, banners, colored cells, and `searchDiagnostics` aggregation tables are explicitly out of first-release scope (`docs/writeback_contract.md` §13). Richer presentation is deferred to FW-0019, triggered by pilot-operator feedback that the crude format is insufficient.
  7. **Atomicity: cleanup-on-failure; "fully populated tab OR no tab."** Writeback MUST surface either a fully-populated writeback tab (success-branch winner allocation per sub-decision 3, or failure-branch content per sub-decision 6) OR no new tab at all (`docs/writeback_contract.md` §14). On any exception or write-stage error during tab population, the implementation MUST attempt to delete the partial writeback tab. If cleanup itself fails, the writeback diagnostic surface (sub-decision 10) MUST name the orphaned tab so the operator can manually delete it. The concrete implementation pattern (Apps Script `Range.setValues` batch + `try { populate } catch (e) { spreadsheet.deleteSheet(newTab); throw e; }` or equivalent) is implementation-slice; the contract pins the operator-facing semantics rather than the code path.
  8. **Idempotency: each upload creates new tab unconditionally; respect operator agency.** Writeback treats each envelope upload as a fresh invocation that creates a new tab unconditionally (`docs/writeback_contract.md` §15). There is no detect-and-skip path on repeated upload of the same `runId`. Repeated uploads accumulate writeback tabs (one per upload). The brief's "redo leaves sheet identical to single-run" intent is interpreted as **content-equivalent** (same allocation values land in equivalent cells of equivalent tabs from the same `runId`) rather than tab-list-identical. Operator agency takes precedence over duplicate-tab clutter at pilot scope; the operator manages cleanup by deleting tabs themselves. A future detect-and-skip mode that scans hidden developer metadata for a matching `runId` is captured in FW-0020.
  9. **Run-envelope traceability: 4-row visible footer + 6-key hidden developer metadata.** Every writeback tab MUST carry traceability content in two complementary surfaces (`docs/writeback_contract.md` §16). Visible footer at the bottom of the tab: four rows in fixed order — `Run ID:`, `Generated:`, `Source:`, `Status:` (with `SUCCESS` or `FAILED`). Hidden developer metadata via Apps Script `Sheet.addDeveloperMetadata(key, value)`: six keys — `runId` (string), `generationTimestamp` (ISO 8601 string), `sourceTabName` (string), `sourceSpreadsheetId` (string), `contractVersion` (string, constant `"1"` — Apps Script developer metadata API stores values as strings, so this contract's `contractVersion: 1` lands as the string `"1"` per `docs/writeback_contract.md` §16.2), `status` (enum-valued string, `"SUCCESS"` / `"FAILED"`). Per-cell metadata, score breakdowns, `searchDiagnostics` aggregations, and additional run-envelope fields beyond the six keys (`seed`, `fillOrderPolicy`, `crFloorMode`, `crFloorComputed`, `selectorStrategyId`) are explicitly NOT included — the score and diagnostics content are selector-stage audit artifacts (`docs/selector_contract.md` §14, §17), not writeback-traceability content; the additional run-envelope fields are available in the JSON envelope if needed. Future expansion of hidden developer metadata keys is additive per `docs/writeback_contract.md` §16.4.
  10. **Diagnostic surface: 3-state launcher minimum content.** The writeback launcher MUST surface a three-state diagnostic on every invocation (`docs/writeback_contract.md` §17). Success state: confirmation message + new tab name + spreadsheet link anchored to the new tab where the platform supports tab anchoring. Failure state (`UnsatisfiedResultEnvelope` input): brief failure indicator + new failure tab name + spreadsheet link anchored to the failure tab; the launcher MUST NOT duplicate the failure-branch tab's `unfilledDemand` and `reasons` content into the launcher diagnostic itself (that lives on the failure tab per sub-decision 6). Runtime-error state: human-readable error message + orphan-tab name if cleanup-on-failure left one (per sub-decision 7). Visual styling, button placement, and exact wording are implementation-slice concerns. The launcher diagnostic and the spreadsheet state MUST agree (no partial-state launcher commit).
  11. **Source-sheet identity propagation: required `runEnvelope` fields per selector contract v2.** The writeback contract requires `runEnvelope.sourceSpreadsheetId` (Google Sheets spreadsheet ID, normalized per `docs/sheet_generation_contract.md` §12.5) and `runEnvelope.sourceTabName` (string) (`docs/writeback_contract.md` §18). Both fields are added to the selector contract's `runEnvelope` required-fields list in `docs/selector_contract.md` §9 item 3, with selector `contractVersion` bumped from `1` to `2` in the same change round (see D-0032 for the selector bump record). The fields were initially proposed as additive per `docs/selector_contract.md` §16.3 (no selector bump); during PR #66 codex review the resulting contract-seam gap (selector compliance without these fields would not imply writeback compatibility) was surfaced as a P1 consistency flag, and the upstream-patch resolution was adopted instead. Selector compliance now implies writeback compatibility on this surface; no execution-layer composition gate is required. The fields are execution-layer-supplied at parser/normalizer ingestion time (when the source tab and spreadsheet are known) and flow through the pipeline alongside `runId`, `generationTimestamp`, and other run-level metadata. Writeback uses `sourceSpreadsheetId` to identify the target spreadsheet and `sourceTabName` as the basis for the new writeback tab name (sub-decision 3). The writeback launcher consumes the source identity from the run envelope rather than from a separate operator-supplied form field; first-release destination is source-only (the operator manually copies/moves the resulting tab if they want a different destination), with cross-spreadsheet/cross-tab override deferred to FW-0022.
- **Rationale:** A pure-adapter Apps Script writeback consuming a serialized envelope is the cheapest path that preserves the D-0017 / D-0018 stack split, inherits the M1.1 OAuth-scope discipline, and reuses the M1.1 launcher pattern for operator UX continuity. The alternative — Python writing directly to operator-owned spreadsheets — would expand the credential surface in three uncomfortable ways (operator OAuth flow in Python, service account with per-spreadsheet share, maintainer-mediated runs), and would break the stack split for marginal handoff savings. Always-new-tab branch-on-write is the surprising-but-correct simplification: it dissolves the broader conflict-handling problem (operator edits between run-start and writeback) at the writeback boundary, gives operators a side-by-side comparison surface for free, makes failure-branch consistent with success-branch (same code path, same mental model), and makes idempotency a content-level guarantee rather than a tab-list-identity one. The single source of truth for the displayed doctor name is the operator's run-start column-A cell value, mediated through a `doctorIdMap` rather than the parser/normalizer's canonical doctor table — this preserves the operator's mental model that column A IS the canonical doctor-name surface and avoids any hidden disagreement between parser-stage and writeback-stage doctor displays. Cleanup-on-failure atomicity matches the "fully populated tab OR no tab" operator-facing semantics with the simplest possible implementation pattern (`try { populate } catch (e) { delete; throw }`); orphan-tab surfacing on cleanup-failure preserves operator awareness without forcing the contract to pin a recovery code path. Respect-operator-agency idempotency reflects the user steering ("let users do what they want — if their action is dumb still allow — the bloated output is results of their own choice") and reinterprets the brief's idempotency requirement as a content-level guarantee rather than a tab-list-identity guarantee, which is the simpler property to specify and the one that actually matches what the operator cares about (no silent duplicate-allocation effects). The 4-row footer + 6-key hidden metadata is the minimum operator-facing audit surface that supports both human-readable trace-back ("which run produced this tab?") and programmatic consumption ("scan the spreadsheet for tabs from `runId X`") without spilling into score-breakdown or search-diagnostics territory that the writeback boundary does not own. Three-state launcher diagnostic minimum content is what the operator needs to see to know what just happened; richer styling is deferred so the contract surfaces what's normative ("you must show success/failure/runtime-error and what to do about each") without pinning UI choices that the implementation slice can iterate. Source-sheet identity on the run envelope (vs the snapshot or a writeback-only wrapper) keeps the fields at the right semantic home (run-level metadata) without forcing a snapshot-contract change or a wrapper-level audit fragment; the fields were initially proposed as additive per `docs/selector_contract.md` §16.3 to avoid bumping the just-closed selector contract, but PR #66 codex review surfaced the resulting contract-seam gap (selector compliance without these fields would not imply writeback compatibility) as a P1 consistency flag, and the cleaner upstream path of bumping the selector contract from v1 to v2 with the fields added to the §9 item 3 required-fields list was adopted instead so selector compliance implies writeback compatibility on this surface (see D-0032). Drafting scope-ahead of M3 activation matches the D-0027 / PR #65 precedent that worked cleanly between the M2 C1 close and M2 C2 activation; activating M3 today would violate M3's own §5-stated `Dependencies: M2 complete` while M2 is mid-flight.
- **Consequences:** `docs/writeback_contract.md` is accepted as the normative boundary, drafted scope-ahead of M3 activation. M2 stays the active milestone; M3 stays `Planned`. The writeback boundary previously named only at `docs/domain_model.md` §14 ("an adapter concern") now has a contractual home covering branch-on-write semantics, doctor identity resolution, failure-branch rendering, atomicity, idempotency, traceability, diagnostic surface, and source-sheet identity propagation. The selector contract is bumped from `contractVersion: 1` to `contractVersion: 2` in the same change round per D-0032 — `runEnvelope.sourceSpreadsheetId` and `runEnvelope.sourceTabName` are added to the §9 item 3 required-fields list so selector compliance implies writeback input compatibility. Concrete JSON envelope schema layout, launcher transport mechanics (paste vs upload vs Drive Picker), and Apps Script module decomposition remain execution-layer concerns deferred to the M3 implementation slice. Five new future-work entries land in this change round: FW-0019 (richer failure-tab presentation), FW-0020 (idempotency-via-`runId`-skip detection), FW-0021 (writeback rendering for partial-allocation strategies), FW-0022 (cross-spreadsheet/cross-tab destination override), FW-0023 (in-situ writeback optimization for the no-conflict case). `docs/delivery_plan.md` §5 M3 likely-checkpoints sub-bullet telegraphs the scope-ahead drafting; §10 deferred-list gains a clarifying line; §12 change-log records the writeback contract drafting and the selector v1 → v2 bump together. The selector contract is the only previously-settled contract reopened in this change round, and only for the v2 bump per D-0032; `docs/rule_engine_contract.md`, `docs/scorer_contract.md`, `docs/solver_contract.md`, `docs/sheet_generation_contract.md`, `docs/parser_normalizer_contract.md`, `docs/template_contract.md`, `docs/template_artifact_contract.md`, `docs/request_semantics_contract.md`, `docs/domain_model.md`, and `docs/blueprint.md` are all untouched.

Author content-equivalence determinism tests for the writeback adapter against fixed `(finalResultEnvelope, snapshot, doctorIdMap)` inputs, mirroring the byte-identical determinism discipline upstream. ### D-0032: Selector contract v2 — `runEnvelope` source-sheet identity required-fields expansion (`sourceSpreadsheetId`, `sourceTabName`)
- **Date:** 2026-04-26
- **Status:** Accepted
- **Context:** D-0030 closed the selector contract at `contractVersion: 1` on 2026-04-25 (PR #65), with `runEnvelope` required-fields list `runId`, `snapshotRef`, `configRef`, `seed`, `fillOrderPolicy`, `crFloorMode`, `crFloorComputed`, and `generationTimestamp` (`docs/selector_contract.md` v1 §9 item 3) and an additivity clause for future run-level metadata expansion (`docs/selector_contract.md` §16.3). D-0031 closed the writeback contract on 2026-04-26 (PR #66) and required `runEnvelope.sourceSpreadsheetId` and `runEnvelope.sourceTabName` at the writeback input boundary for source-sheet routing and tab naming. D-0031 sub-decision 11 initially proposed these fields as additive per `docs/selector_contract.md` §16.3 (no selector contract bump), reasoning that the additivity clause was explicitly designed for this kind of expansion. PR #66 codex review surfaced this as a P1 contract-seam consistency flag: selector contract compliance under v1 does not require either field, so a v1-compliant `FinalResultEnvelope` could legitimately reach writeback without these fields and break the writeback ingestion contract; the additivity clause exercise leaves the end-to-end contract chain inconsistent. Codex offered two resolutions: bump the selector contract upstream to require the fields, or define an explicit execution-layer composition gate at the writeback boundary. The composition-gate resolution was attempted first (writeback `§18.5` in commit 3308de3) and codex re-flagged the seam as still unresolved (the gate concept itself was the new ambiguity). The user directed adopting the upstream-patch resolution instead.
- **Decision:** Bump `docs/selector_contract.md` from `contractVersion: 1` to `contractVersion: 2` and add `sourceSpreadsheetId` and `sourceTabName` to the `runEnvelope` required-fields list in §9 item 3. Both fields are execution-layer-supplied at parser/normalizer ingestion time. The selector itself does not consume either field — they ride through unchanged per §16.4 — but they are required at the selector boundary so a downstream writeback adapter cannot receive a contract-compliant envelope that lacks them. Concretely:
  1. **`contractVersion` bump v1 → v2.** `docs/selector_contract.md` §2 is updated to declare `contractVersion: 2`. §2.1 ("Version history") is added to record the v1 baseline (PR #65, D-0030) and the v2 expansion (PR #66, this decision). The bump rule in §2 is left unchanged in spirit; a clarifying parenthetical distinguishes optional-additive fields (non-bumping per §16.3) from required-fields-add (bumping).

### D-0033: `dateKey` adapter normalization — snapshot adapter emits ISO 8601 `rawDateText`; parser uses it directly as `dateKey`
- **Date:** 2026-04-26
- **Status:** Accepted
- **Context:** Surfaced during M2 C3 T1 implementation (PR #70). The parser/normalizer pipeline emits a `dateKey` on multiple normalized entities (`SlotDemand.dateKey`, `Request.dateKey`, `FixedAssignment.dateKey`, `DailyEffectState.dateKey`, `RosterDay.dateKey`) so downstream stages can cross-reference days in O(1). `docs/domain_model.md` §7.2 names `dateKey` as "stable date identity" but does not pin its concrete format; `docs/snapshot_contract.md` §8 declares only `rawDateText` (raw cell text) plus `dayIndex` (ordered position). The real ICU/HD May 2026 source sheet stores dates as Excel serial numbers (`46146.0` = 2026-05-04); other operator-facing platforms could supply ISO strings, locale-formatted text, or other forms. Without a pinned format, any downstream stage that compares `dateKey`s across pipeline boundaries risks silent drift if upstream stages disagree on the canonical form.
- **Decision:** The snapshot adapter MUST normalize date identity to ISO 8601 (`YYYY-MM-DD`) before emitting `dayRecord.rawDateText`. The parser uses `rawDateText.strip()` directly as `dateKey` across all normalized entities. No additional date parsing happens at the parser stage. Concretely:
  1. **Adapter responsibility.** Whatever sheet-side date format the adapter encounters (Excel serial number, locale-formatted text, ISO string, ad hoc string) is normalized to `YYYY-MM-DD` form at adapter boundary, before the snapshot is emitted. Adapters that cannot deterministically normalize a cell's date identity MUST fail at adapter boundary rather than emit a malformed snapshot.
  2. **Parser responsibility.** The parser treats `rawDateText.strip()` as the canonical `dateKey`. Empty / whitespace-only `rawDateText` continues to be a §13 structural defect (`ISSUE_DAY_RAW_DATE_EMPTY`) per the parser admission pipeline. The parser does not reformat, reparse, or validate ISO compliance — that is adapter territory.
  3. **Cross-stage invariant.** Because every entity's `dateKey` is derived from the same `rawDateText`, equality across `SlotDemand.dateKey`, `Request.dateKey`, `FixedAssignment.dateKey`, `DailyEffectState.dateKey`, and `RosterDay.dateKey` is exact string equality on the ISO form.
- **Rationale:** Each layer's responsibility stays clean: the snapshot adapter knows the sheet world (Excel serial conversion, locale parsing, free-text cleanup); the parser knows the logical world (canonical day identity used by downstream stages). The adapter already has to do format-detection work to populate `rawDateText` at all, so codifying ISO output downstream costs nothing extra and prevents per-stage re-derivation drift. Two alternatives were considered and rejected: (a) parser-side normalization would pull sheet-format complexity into the parser and conflict with `parser_normalizer_contract.md` §17 ("rule engine evaluates normalized legality; it does not recover lost parser meaning") by analogy — parser should not recover lost adapter normalization either; (b) using `dayIndex` alone as the `dateKey` would lose calendar-day awareness needed by the scorer (weekend / public-holiday point weights per `docs/template_artifact_contract.md` §9 `pointRows.defaultRule`) and by the rule engine's `BACK_TO_BACK_CALL` invariant (which currently keys on `dateKey - 1` / `dateKey + 1` semantics per `docs/rule_engine_contract.md` §11 — date arithmetic on bare integers loses month-boundary correctness when `dateKey` is also exposed to writeback for operator-facing day labeling).
- **Consequences:**
  - The current PR #70 parser implementation already conforms — `_validate_structural`, `_interpret_template`, `_parse_requests`, and `_process_prefilled_assignments` all use `rawDateText.strip()` as `dateKey` directly with no reparsing.
  - All existing T1 test fixtures use ISO strings (`2026-05-01`, etc.) and remain valid.
  - The future Excel/Google Sheets adapter implementation slice (own checkpoint, not yet scoped) MUST handle Excel serial number conversion, locale text parsing, and any other format-detection work before snapshot emission. That checkpoint is the natural home for the adapter-side normalization implementation.
  - No contract document is reopened by this decision — `docs/snapshot_contract.md` §8 (which declares `rawDateText` as raw cell text) is consistent with this decision because the adapter contract leaves `rawDateText` content shape to the adapter; this decision pins the shape to ISO 8601 for ICU/HD first release without requiring a snapshot-contract change.
  - The decision is ICU/HD first-release scoped. Future departments operating on platforms with different date semantics (for example, fiscal-year-relative dates or non-Gregorian calendars) may revisit; if they do, the adapter contract for that platform will declare its own normalization rule and the parser side stays unchanged.
- **Follow-up actions:** When the snapshot adapter implementation slice begins, validate that the adapter emits `rawDateText` in ISO 8601 form and rejects malformed dates at adapter boundary. Add an adapter-side acceptance test against the May 2026 ICU/HD source (which carries Excel serial dates) confirming the adapter produces ISO output that the parser admits without reformatting.
- **Related docs:** `docs/snapshot_contract.md` §8; `docs/parser_normalizer_contract.md` §11, §14; `docs/domain_model.md` §7.2.

### D-0034: Doctor `displayName` matching algorithm for prefilled-assignment resolution — trim + collapse internal whitespace + casefold
- **Date:** 2026-04-26
- **Status:** Accepted
- **Context:** Surfaced during M2 C3 T1 implementation (PR #70). `docs/parser_normalizer_contract.md` §14 says the parser must "exactly deterministically resolve prefilled doctor text against names entered in doctor-entry sections" but does not specify the matching algorithm. Three candidate algorithms were live: strict character-equal (operator types `Dr foxtrot`, parser fails to match `Dr Foxtrot`, NON_CONSUMABLE — too brittle for non-technical operators); light tolerance (trim + casefold — shipped in T1 as a working default); heavy tolerance (also fold punctuation, fuzzy-match — risks colliding two distinct doctors when surnames stem-match). The real ICU/HD May 2026 source sheet has doctor names with trailing spaces (`"Loh Xinpeng "`) that the strict algorithm would reject; operators may also type variations in case or accidentally double-tap the spacebar between first and last name.
- **Decision:** Doctor `displayName` matching for prefilled-assignment resolution applies the following normalization to BOTH the prefilled cell text AND the doctor `displayName` before character-equal comparison:
  1. **Trim** leading and trailing whitespace.
  2. **Collapse** runs of internal whitespace to a single space (so `"Dr  Foxtrot"` and `"Dr Foxtrot"` compare equal).
  3. **Casefold** for Unicode case-insensitive comparison (so `"dr foxtrot"` and `"Dr Foxtrot"` compare equal).
  Punctuation, diacritics, and any other character classes are NOT folded. After normalization the comparison is character-equal. Ambiguity (two distinct `displayName`s collapsing onto the same normalized form, OR a prefilled cell matching multiple doctors) remains a §14 NON_CONSUMABLE case (`ISSUE_PREFILLED_DOCTOR_NAME_AMBIGUOUS`).
- **Rationale:** The chosen rule handles the realistic spreadsheet quirks observed on the May 2026 source (trailing whitespace, occasional internal whitespace differences) and the obvious operator typing variations (case differences) without admitting risky fuzzy matches that could confuse two distinct doctors. The strict alternative would force operators to character-perfectly retype names that already sit in column A — a non-trivial cost for non-technical pilot operators. The heavy-tolerance alternative would risk admitting a wrong fixed assignment silently — and fixed assignments are load-bearing for the roster (they bind the solver per `docs/domain_model.md` §10.1 and operator-prefilled cells override request-derived hard block per `docs/parser_normalizer_contract.md` §14), so a wrong-doctor admission has high blast radius. The §14 ambiguity check is the safety backstop: when the chosen normalization happens to collapse two genuinely distinct displayNames, the parser refuses with `ISSUE_PREFILLED_DOCTOR_NAME_AMBIGUOUS` rather than picking one. Diacritic folding is intentionally omitted because diacritic differences in clinical names are typically meaningful (and ICU/HD first release operates in an English-name environment); if a real case emerges where this is too strict, the rule can be loosened additively without breaking prior matches.
- **Consequences:**
  - The current PR #70 parser implementation is patched in this change round to apply the D-0034 normalization on both sides of the prefilled-cell match (`_normalize_doctor_name` helper in `python/rostermonster/parser/admission.py`); previous behavior was trim + casefold only (internal-whitespace runs left intact).
  - Test fixtures gain a positive case (`with_prefill_using_messy_whitespace_and_case`) verifying that `"   dr   foxtrot  "` resolves to `Dr Foxtrot`, locking the trim + collapse + casefold behavior.
  - The §14 ambiguity check (`ISSUE_PREFILLED_DOCTOR_NAME_AMBIGUOUS`) continues to fire on collapsed-name collisions; existing test (`test_ambiguous_prefill_doctor_is_non_consumable`) remains valid.
  - The rule is ICU/HD first-release scoped. Future departments with non-English names, hyphenation conventions, or punctuation-bearing displayNames may surface cases where this rule needs widening (or narrowing). T3 (hand-test against the ICU/HD May 2026 sheet) is the natural place to discover any operator-realistic case the chosen rule mishandles; if any such case emerges, this decision can be revised additively.
- **Follow-up actions:** When T3 lands, confirm the rule handles the May 2026 doctor list end-to-end without false negatives. If a future department template introduces non-English names or punctuation-bearing displayNames, surface for revisit; widen the rule additively rather than reopening this decision.
- **Related docs:** `docs/parser_normalizer_contract.md` §14; `python/rostermonster/parser/admission.py` (`_normalize_doctor_name`, `_process_prefilled_assignments`); `python/tests/test_parser.py` (`test_prefill_with_messy_whitespace_and_case_resolves_per_d0034`).

### D-0038: `pointRules` completeness — fail loudly on missing keys (revising D-0037 sub-decision 5)
- **Date:** 2026-04-27
- **Status:** Accepted
- **Context:** D-0037 sub-decision 5 declared that "Missing `(slotType, dateKey)` entries fall back to `1.0` per-call so the scorer remains sign-correct under partial overlay." That choice was motivated by sign-correctness preservation under operator-supplied partial overlays. The PR #82 codex P2 round-trip closed the construction-level gap (`ScoringConfig.pointRules` is now truly required at construction time, no `default_factory`), but the per-key fallback inside the scorer's `pointBalance*` components remained — `point_rules.get((slotType, dateKey), 1.0)`. Operator review surfaced the same fail-fast principle the construction-level fix applied: the per-key 1.0 fallback masks producer-side bugs. A parser implementation that accidentally fails to populate some `(slotType, dateKey)` keys (because of a layout regression, a snapshot extraction defect, a missing template-default-computation path, etc.) silently produces wrong scores — the missing entries are scored as if every call were worth `1.0` per-call, which is wrong whenever the template default for that day is non-1.0 (weekend = 1.75, weekend-to-weekend = 2.0, weekend-to-weekday = 1.5). That class of producer bug should fail loudly, not silently degrade.

  The "partial overlay" justification from D-0037 sub-decision 5 was based on a misreading of the architecture: the parser overlay is sheet-onto-template-defaults, not partial-onto-fallback-1.0. Once the parser ships per `docs/decision_log.md` D-0036, every `(call-slot, dateKey)` pair receives either an operator-edited value OR the template-derived default — never an unspecified value. A missing key in `pointRules` reaching the scorer therefore signals a producer bug, not a legitimate "no override" state.

- **Decision:** Reverse D-0037 sub-decision 5's fallback rule. Concretely:

  1. **Per-key lookup is fail-loud.** The scorer MUST raise on missing `(slotType, dateKey)` keys when looking up call-point weights inside `pointBalance*` components. No silent `1.0` fallback. Implementations are free to validate completeness once at `score()` entry (preferred — surfaces all missing keys in one error message) rather than per-key at component-call time, but the observable behavior is identical: any allocation referencing a `(call-slot, dateKey)` not present in `ScoringConfig.pointRules` causes `score()` to raise with a clear error message naming the missing key(s).

  2. **Producer obligation.** The producer (parser overlay path per `docs/parser_normalizer_contract.md` §9) MUST populate every `(slotType, dateKey)` pair where `slotType` is a call-slot per the model's `slotTypes[].slotKind == "CALL"` filter and `dateKey` ranges over the period's `dayRecords`. Coverage is total; partial pointRules is a parser-side defect. Sheet-derived values + template-default backstops together cover the cross-product — the parser combines them at overlay time and emits a complete `pointRules` map.

  3. **`scorer_contract.md` §11 wording reversal.** The "missing entries fall back to `1.0`" sentence in scorer v2 §11 is replaced with "missing entries cause `score()` to raise; producer obligation is total coverage of `(call-slot, dateKey)` pairs from the model". The §22 v2 follow-on subsection notes the wording revision under D-0038. No further `contractVersion` bump is taken — the public shape (`weights`, `pointRules`, `curves?`) stays scorer v2; this decision tightens behavioral semantics without changing the type signature, and the existing "v1-targeted callers that supply only `weights` are non-compliant against v2" rule already covered the field-presence requirement. The behavioral tightening is captured in the §11 wording + this decision-log entry.

  4. **`parser_normalizer_contract.md` §9 update.** The overlay rule is extended with the producer-coverage requirement: parser overlay MUST emit a `scoringConfig.pointRules` map covering every `(slotType, dateKey)` pair from the cross-product of call-slot slotTypes and the period's dateKeys. Missing entries on the parser side are admission-blocking per §14 (matching the existing sign-orientation discipline for component weights).

  5. **`first_release_defaults` semantics.** `ScoringConfig.first_release_defaults()` previously returned `pointRules={}`. Under fail-loud semantics, an empty `pointRules` only works for allocations with no call slots — which never happens in production but is a degenerate test scenario. The factory is updated to take a `model: NormalizedModel` argument and populate `pointRules` with a uniform `1.0` weight across every `(call-slot, dateKey)` pair. A new helper `uniform_point_rules(model)` exposes the cross-product construction for ad-hoc test fixtures and for future production callers (e.g., a "reset to no-override defaults" path post-parser-overlay if needed). The uniform-1.0 result is explicitly NOT what the production parser overlay produces — the parser computes per-day template defaults from `pointRows.defaultRule` weekday/weekend mapping (1.0 / 1.75 / 2.0 / 1.5), and `uniform_point_rules` is documented as a test/fixture helper, not a production "first-release defaults" path.

  6. **Test rewrite.** The PR #82 test `test_point_balance_falls_back_to_one_per_call_when_pointrules_empty` is removed (it was locking the now-reversed behavior). A new test `test_score_raises_on_missing_point_rules_key` is added, exercising the fail-loud behavior with an allocation that references a `(slotType, dateKey)` key absent from `pointRules`. Existing tests that previously used `ScoringConfig.first_release_defaults()` are updated to pass `_model()` so the factory builds complete `pointRules` for the test fixture's call-slot × day cross-product.

- **Bidirectional audit dogfooded** (per `docs/delivery_plan.md` §14):
  - **Forward**: scorer v2 §11 (consumer expects complete `pointRules`) → parser_normalizer §9 producer obligation (coverage rule) → declared in this decision. ✓
  - **Reverse**: parser_normalizer §9 (producer references producing complete `pointRules`) → scorer v2 §11 (consumer expects it complete; raises on missing) → declared in this decision. ✓
  - **Forward**: parser_normalizer §14 admission cases (mis-signed weights → NON_CONSUMABLE) → scorer v2 §10 / §15 sign-orientation classification (existing) — unchanged here. ✓
  - **Reverse**: scorer §11 helper `uniform_point_rules(model)` (test/fixture path) → no consumer in production parser overlay (parser computes per-day defaults from template, not uniform 1.0). The helper is correctly scoped as test-only, not on the parser → scorer production seam. ✓

- **Rationale:** The fail-fast principle the construction-level fix already enforced (PR #82) extends naturally to the per-key level: any missing `pointRules` key at score-time is a producer-side defect, not a legitimate state. Silent fallback to `1.0` masks the defect and produces scores that look reasonable but don't reflect the operator's actual intent (especially on weekends where template defaults are 1.75 / 2.0). Total-coverage producer obligation is the simpler discipline — the parser already has access to the model's call slots and period days at overlay time, and the cross-product computation is straightforward (model.slotTypes × model.period.days, filtered to call-slots). Tests use a clearly-named test helper (`uniform_point_rules`) to avoid the per-test cross-product boilerplate; the helper's name and docstring make clear it's not the production "first-release defaults" path.

### D-0039: `spacingPenalty` geometric-decay shape (port v1, replacing v2 binary at gap=2)
- **Date:** 2026-04-28
- **Status:** Accepted
- **Context:** First-release v2 `spacingPenalty` ships as a **binary** count of same-doctor call pairs with `gap < 3` days (`python/rostermonster/scorer/components.py` `_SPACING_MIN_GAP_DAYS = 3`). Since the rule engine's `BACK_TO_BACK_CALL` already rejects `gap = 1`, the only gap producing a soft penalty in v2 today is `gap = 2`. Pairs at `gap = 3, 4, 5, 6, 7+` all contribute zero. This is contract-compliant — `docs/scorer_contract.md` §4 lists `spacingPenalty` as a component identifier but does not pin the curve shape — but it loses the gradient v1's reference implementation provided. The legacy v1 implementation in `for_claude_code/old_repos/v1/scorer_main.js` (`getShortGapCallPenalty_`) computed `base / 2^(gap - 2)` for `gap` ∈ {2..6}, zero for `gap > 6`. The smooth curve gave the solver a continuous gradient to optimise against — closer-spaced calls hurt more than wider-spaced ones, even within the soft window — rather than treating "gap = 2 vs gap = 6" as binary.

  Operator review during M2 C7 UX iteration surfaced the gap: the v2 description on the operator-facing Scorer Config tab read "*Fires when the same doctor has two call placements within the 3-day minimum-gap window*" — accurate for v2 but misleading because it implied the operator could tune the curve via the weight, when in fact the weight only scales a binary on/off. The user requested porting v1's gradient back into v2.

- **Decision:** Replace v2's binary count with v1's geometric-decay shape, with one rationale-driven adjustment to v1's cutoff. Concretely:

  1. **Soft-window window.** Per-pair penalty contribution is non-zero for `gap ∈ {2, 3, 4, 5, 6}` and zero elsewhere. `gap = 1` remains hard-blocked by rule engine `BACK_TO_BACK_CALL` (no scorer involvement). `gap ≥ 7` contributes zero, matching v1's `MAX_SOFT_GAP_DAYS = 6`. Rationale: a 7-day gap corresponds to roughly once-per-week call cadence, which is the natural rhythm we want neither to encourage nor discourage. Penalising gap = 7 would push the solver to spread pairs even further apart, which has no operational benefit and risks pathological all-spacing-no-coverage trade-offs.

  2. **Decay shape.** Per-pair penalty contribution is `weight / 2^(gap - 2)` for `gap ∈ {2, 3, 4, 5, 6}`. At `gap = 2` the contribution is the full per-pair weight; each additional day halves it. Concrete progression at `weight = -2`: gap 2 contributes `-2.0`, gap 3 contributes `-1.0`, gap 4 contributes `-0.5`, gap 5 contributes `-0.25`, gap 6 contributes `-0.125`, gap ≥ 7 contributes `0.0`. Total `spacingPenalty.score` = sum across all same-doctor call pairs in the allocation.

  3. **Property pinning in `docs/scorer_contract.md` §12A** (new section, parallel to §12's `crReward` strict-monotonic-decrease pattern). The contract pins:
     - **Strict monotonic decrease** across the soft window when `weights[spacingPenalty] < 0`: contribution at `gap = k` is strictly less in absolute magnitude than contribution at `gap = k - 1`, for `k ∈ {3, 4, 5, 6}`. When `weights[spacingPenalty] = 0` (legitimate disable-the-component state per §10 / §15 — penalty weights MAY be zero), every contribution is zero and the strict-decrease property is vacuously satisfied.
     - **Zero past cutoff**: contribution at `gap ≥ 7` is exactly zero (cutoff is part of the curve definition; independent of weight).
     - **Fixed first-release shape**: geometric / halving (`weight / 2^(gap - 2)`); `MAX_SOFT_GAP_DAYS = 6`. Operator-tuneable curve parameters remain `docs/future_work.md` FW-0007 deferred (same discipline as `crReward`'s curve shape).

  4. **Constant rename in `python/rostermonster/scorer/components.py`.** Replace `_SPACING_MIN_GAP_DAYS = 3` with two constants: `_SPACING_BASE_GAP_DAYS = 2` (the gap at which contribution equals the full weight) and `_SPACING_MAX_SOFT_GAP_DAYS = 6` (the largest gap with non-zero contribution). The new names match v1's vocabulary and make the curve shape readable at the call site.

  5. **Test additions in `python/tests/test_scorer.py`.** Existing spacing tests are updated for the new curve. New `test_spacing_penalty_strictly_decreases_per_gap_step` property test mirrors `test_cr_reward_strictly_diminishes_per_doctor` from §12 — property-style, varies the gap directly, asserts strict-decrease across the soft window. New `test_spacing_penalty_is_zero_past_max_gap` pins the gap = 7 cutoff explicitly.

  6. **`contractVersion` bump v2 → v3.** Per the §2 version bump rule ("bump `contractVersion` only when scoring semantics, direction, component enumeration, or contract-level input/output shape changes"), this decision changes scoring SEMANTICS for valid configs: the same `ScoringConfig` produces different `spacingPenalty.score` under v3 vs v2 because gaps in `[3, 6]` now contribute non-zero where they previously contributed zero. v2-targeted ranking behavior is therefore not preserved, and consumers relying on the `contractVersion` signal for behavioral compatibility need the bump. This is the discriminator vs D-0038's "no bump" precedent — D-0038 tightened ADMISSION (the set of valid configs shrank, but valid-and-still-valid configs produced the same scores), whereas D-0039 tightens COMPUTATION (same valid config → different score). The `ScoringConfig` public shape itself is unchanged — `weights[spacingPenalty]` stays a single scalar.

  7. **No new operator-facing surface.** The Scorer Config tab description for `spacingPenalty` will be updated in the same change round (or shortly after) to describe the curve shape rather than the binary behavior, but that is a producer-side wording polish and does not require contract scope.

- **Bidirectional audit dogfooded** (per `docs/delivery_plan.md` §14):
  - **Forward**: scorer §12A (consumer pins curve) → no producer-side dependency (the curve is hard-coded inside the scorer; operator-tuneable curve params are FW-0007 deferred). ✓
  - **Reverse**: no contract references producing a curve shape at the parser boundary; `scoringConfig.weights[spacingPenalty]` stays a single number. ✓
  - **Forward**: scorer §13 direction-guard invariant (`HIGHER_IS_BETTER`) → spacingPenalty contributions remain strictly non-positive (weight ≤ 0 by §10 / §15 sign discipline; `2^-(gap-2) > 0` for all `gap ∈ {2..6}`); converting a filled unit to unfilled removes call placements and therefore can only reduce the magnitude of the spacing penalty (penalty becomes less negative or stays zero). Direction-guard invariant preserved. ✓

- **Rationale:** The smooth gradient gives the solver a continuous signal to optimise against, which matters because `SEEDED_RANDOM_BLIND` Phase 2 (`docs/solver_contract.md` §12) chooses doctors via tightest-slot-first with seeded tie-break — having spacingPenalty distinguish gap = 3 from gap = 6 lets the seeded tie-break reflect spacing preference even when both placements are rule-valid. v1's halving-each-day shape was empirically reasonable across the 2026 pilot cohorts and there is no specific operator complaint motivating a different curve. The cutoff at `gap = 7` instead of v1's "post-decay zero" wording embeds the once-per-week rhythm explicitly: above 7 days, spacing is not the optimisation pressure we want — coverage and fairness take over. Hard-pinning the curve in the contract (rather than treating it as implementation detail like §12 does for `crReward`) is justified by v1 already having pilot validation; we deliberately commit to this shape and defer operator tunability to FW-0007.

  - `docs/scorer_contract.md` §12A: new normative subsection pinning the spacingPenalty curve shape + cutoff + strict-decrease property.
  - `python/rostermonster/scorer/components.py`: `_SPACING_MIN_GAP_DAYS = 3` removed; `_SPACING_BASE_GAP_DAYS = 2` + `_SPACING_MAX_SOFT_GAP_DAYS = 6` added; `spacing_penalty()` rewritten to sum `weight / 2^(gap - 2)` across pairs in `[2, MAX_SOFT_GAP_DAYS]`.

### D-0040: Apps Script ↔ Python transport — browser download (operator-saved JSON file)
- **Date:** 2026-04-29
- **Status:** Accepted
- **Context:** `docs/decision_log.md` D-0036 sub-decision 4 commits to settling transport jointly with writeback in the same change round when the snapshot-extraction Apps Script implementation activates. `docs/writeback_contract.md` §6.3 left transport ("operator paste, file upload, Drive Picker, or alternative") as an implementation-slice concern with the explicit constraint that inbound (snapshot ingestion) and outbound (writeback envelope delivery) MUST adopt the same mechanic to avoid two divergent transports across the same Apps Script ↔ Python boundary. The transport question is now load-bearing for M2 C9 (snapshot extractor activation) and remains scoped-ahead for M3 (writeback).
- **Decision:** Transport on both directions is **browser download of a JSON file** that the operator saves locally and points the local Python CLI at:
  1. **Inbound (snapshot extractor, M2 C9):** Apps Script extractor builds a Snapshot-shape JSON object, wraps it in `Utilities.newBlob(jsonString, 'application/json', filename)`, and serves it as a download via the Apps Script Web App's HTML response (or a `HtmlService.createHtmlOutput` page with a `<a download>` link, depending on UX iteration). The operator clicks the download, saves the file (default name `snapshot_<spreadsheetId>_<timestamp>.json` per D-0042), and runs `python -m rostermonster.run --snapshot path/to/snapshot.json`.
  2. **Outbound (writeback, M3):** Python writes a `FinalResultEnvelope`-shape JSON file to disk under the local run output directory. The operator drags the file into the launcher's writeback form (or pastes its content into a textarea — exact UX is M3 implementation-slice). Apps Script ingests the JSON and applies the writeback per `docs/writeback_contract.md` §10.3 branch-on-write semantics.
  3. **Filename convention.** Snapshot files: `snapshot_<spreadsheetId>_<extractionTimestamp>.json` per D-0042 (see that decision for `snapshotId` field separately). Writeback envelopes: `writeback_<runId>.json` (M3 implementation-slice; recorded here because the symmetric mechanic is being settled now).
  4. **Wire format.** UTF-8 JSON. Pretty-printed (2-space indent) for operator-grokkable inspection — first-release pilot is small enough that file size is not a concern; debugging benefit dominates. No envelope wrapping, no compression, no base64.
  5. **Auth surface scoped to a single bounded expansion on the Apps Script side; Python side gets no new auth.** The transport mechanic itself (browser download blob via `Utilities.newBlob`) requires no new OAuth scope on top of what the launcher already has. The launcher's OAuth scope IS expanding from `drive.file` to `drive` for the M2 C9 architecture (per D-0041 sub-decision 4 — the `getFileById + makeCopy` template path can't piggyback on `drive.file`), but that expansion is driven by the bound-template architecture, NOT by this transport decision. The bound shim and central library stay minimal (`spreadsheets.currentonly` + matching scopes per D-0041 sub-decision 7). Python side gets no new auth — it reads a local file, that's it. Drive Picker / Drive API ingestion on the Python side are explicitly NOT taken; they would require a Drive OAuth grant on the Python side that currently does not exist.
- **Bidirectional audit dogfooded** (per `docs/delivery_plan.md` §14):
  - **Forward**: extractor produces JSON (consumer = Python pipeline) → Python expects to read JSON from disk via the new CLI's `--snapshot` flag → declared in this decision and in `docs/snapshot_adapter_contract.md`. ✓
  - **Reverse**: writeback consumer (Apps Script) expects JSON envelope arrival → Python writeback emitter produces JSON file at known path → captured here for the symmetric M3 work. ✓
- **Rationale:** Browser download is the simplest mechanic that satisfies all constraints: operator-grokkable (file lives on their disk, can be inspected with any editor); no new OAuth scopes; produces a stable on-disk artifact useful for debug and audit; symmetric on both directions of the seam. Drive Picker / Drive API would have lower friction but adds a Drive OAuth grant on the Python side (currently zero auth). HTTP push to a local Python server requires the operator to start a server (operationally awkward for a single-machine pilot). Clipboard paste breaks at large file sizes (current dev fixture: ~150 KB pretty-printed; tolerable for clipboard but already past the threshold where an editor is more reliable). Two-step manual flow (download → CLI) is acceptable for pilot scale where the operator already runs the local CLI explicitly.
- **Consequences:**
  - `docs/snapshot_adapter_contract.md` (new in this change round) — declares browser-download as the adapter's output surface and pins the filename convention.
  - `docs/writeback_contract.md` §6.3 — receives an addendum citing this decision as the resolved transport (the §6.3 deferral language stays for forward-compat, but the joint-scoping with the inbound side is now resolved). M3 writeback implementation will follow the same mechanic.
  - `apps_script/central_library/` (new in M2 C9 Phase 2) — implements the JSON-build + blob-download mechanic.
  - `python/rostermonster/run.py` (new in M2 C9 Phase 2) — `--snapshot path/to/snapshot.json` CLI entrypoint reading the file from disk.
  - The test-only xlsx → JSON bridge (`python/tests/extract_icu_hd_may_2026.py`) is unaffected; it remains the fixture-derivation tool, not the live ingestion path.
- **Follow-up actions:**
  1. If pilot operator surfaces friction with the manual two-step flow (download → CLI), revisit toward Drive Picker as the next step up. Do NOT add HTTP push without revisiting the auth-surface question.
  2. If the snapshot file size grows past ~5 MB (large rosters with rich provenance), revisit toward gzip compression to keep the operator's download manageable.

### D-0041: Extractor placement — bound template spreadsheet + central Apps Script library
- **Date:** 2026-04-29
- **Status:** Accepted
- **Context:** The launcher (M1.1) is a standalone Apps Script Web App at `apps_script/launcher/`. It creates spreadsheets via `SpreadsheetApp.create()`. Sheets created this way are not container-bound to the launcher project — simple `onEdit(e)` / `onOpen(e)` triggers declared in the launcher do NOT fire on those sheets (this is the architectural finding that reverted FW-0024 in PR #89 and was captured in `docs/future_work.md` FW-0024). For M2 C9 the snapshot extractor must run on operator-edited spreadsheets, and the natural UX is a per-sheet menu ("Roster Monster → Extract Snapshot") rather than a separate launcher URL where the operator pastes a spreadsheet ID. A standalone Web App cannot deliver a sheet menu; only container-bound scripts can.
- **Decision:** Adopt a **bound template spreadsheet + central Apps Script library** architecture for the extractor. Concretely:
  1. **Template spreadsheet.** A single empty spreadsheet (`[INTERNAL] Roster Monster Template`) is created manually in Drive once. Inside it, Extensions → Apps Script creates a container-bound script project. This bound script project is a thin shim — typically <30 lines — that declares `onOpen(e)` to install the operator menu and exposes one menu handler per public action, each delegating to the central library.
  2. **Central library.** `apps_script/central_library/` is a new clasp-managed Apps Script project carrying the extractor logic (Snapshot-build, DeveloperMetadata reading, blob-download wiring, etc.). Published as a versioned Apps Script Library; the bound shim depends on it.
  3. **Library version mode = HEAD (development mode).** The bound shim's manifest pins the library at version `0` (Apps Script's "always-latest" / development-mode alias). Pushing a fix to the central library via clasp propagates immediately to every operator sheet's next menu click. **Apps Script's development-mode library access requires the runtime user to have Drive *Editor* permission on the library script** (viewers cannot load HEAD; they can only load published numbered versions). This constraint forces the library Drive-sharing decision in sub-decision 6 to "Anyone with link → Editor", which was an architectural defect in the original draft of this decision (caught by Codex P1 on PR #95) and is corrected here. Pilot scale (one operator, a few sheets) does not warrant versioned-deployment + bound-script-version-bump ceremony; the trade-off is that an in-flight broken merge to the library reaches operators on next click — same Codex-review-before-merge discipline already in place applies.
  4. **Launcher generation flow.** The launcher's existing `SpreadsheetApp.create(name)` is replaced with `DriveApp.getFileById(TEMPLATE_FILE_ID).makeCopy(name)`. The copy inherits both the bound shim AND the bound shim's library dependency, so every launcher-generated spreadsheet has the menu from the moment it is opened. The launcher's existing layout / cell-population / DeveloperMetadata wiring runs on the copy as it does today. **Two access requirements** (both must be satisfied; sub-decision 4 covers OAuth, the manual template-sharing step in setup (a) covers Drive ACL):
     - **OAuth scope:** `DriveApp.getFileById(...)` against a file NOT created by the app is exactly the scenario D-0023's closing note anticipated ("Future features that need to read or modify files not created by the app cannot piggyback on `drive.file` alone and would need to revisit this decision"). The launcher's `apps_script/launcher/src/appsscript.json` MUST be updated to add `https://www.googleapis.com/auth/drive` to `oauthScopes` (the existing `drive.file` is insufficient for `getFileById` against a manually-created template; Apps Script's static analysis routes any `getFileById` call to the broader `drive` scope per the precedent recorded in D-0023's "Considered alternatives" item 1). The trade-off: the OAuth consent screen widens (operator sees a broader Drive permission prompt on first launcher use); CASA assessment cost re-emerges if the app ever leaves Testing-user mode (deferred to post-pilot per follow-up action 5 below). Pilot scope (named operators in Test User pool) absorbs the trade-off cleanly.
     - **Drive ACL:** Because the launcher executes as `USER_ACCESSING` per `apps_script/launcher/src/appsscript.json`, the operator's own OAuth identity governs runtime Drive access — broadening OAuth scope alone is insufficient if the operator account cannot see the template file at all. The template MUST be Drive-shared at Viewer level (recorded in setup step (a) below), parallel to the library's Editor sharing (sub-decision 6). Without the share, even an operator with the broader OAuth grant hits a Drive permission-denied error on `getFileById`. **Pilot trade-off:** the template + library Drive shares are set to "Anyone with link → Viewer" / "Anyone with link → Editor" respectively rather than per-email allowlists, to keep one-time setup simple. Tightening to per-email allowlists is captured as `docs/future_work.md` FW-0025 — see sub-decision 6 + that entry for the documented risk and revisit triggers.
  5. **Three Apps Script projects in the repo.**
     - `apps_script/launcher/` (existing, unchanged) — the launcher Web App.
     - `apps_script/bound_shim/` (new) — clasp-managed against the bound script that lives inside `[INTERNAL] Roster Monster Template`. Carries `onOpen` menu wiring + delegating shim functions.
     - `apps_script/central_library/` (new) — clasp-managed Apps Script Library carrying the extractor logic.
  6. **Library Drive sharing = "Anyone with link → Editor" (pilot-scope simplification; tightening tracked as FW-0025).** Apps Script's development-mode library access (sub-decision 3) requires the runtime user to have *Editor* (not just viewer) permission on the library script — viewers can only load published numbered versions, not HEAD. The strict-discipline option is per-email Editor allowlist matching the Test User pool (D-0021 / D-0022); first-release pilot adopts the simpler "Anyone with link → Editor" instead, accepting the documented risk that any URL holder could mutate the library and have the change propagate to every operator's next click via HEAD mode. **Justification for accepting the risk at pilot scale:** (a) the library URL is not published anywhere external — it lives in the maintainer's Drive and is only known to the maintainer + the named pilot operator(s); (b) the library carries no secrets, just code, so a leaked URL doesn't expose credentials; (c) a malicious edit would have to be made by someone who acquired the URL AND wanted to compromise the extractor specifically — a narrow threat model for a one-operator pilot; (d) per-email allowlist setup ceremony is non-trivial across three Drive-shared assets (template + library + future writeback assets), and getting that wrong locks operators out. **Tightening this to per-email allowlists is captured as `docs/future_work.md` FW-0025**, with revisit triggers when pilot scope expands or production-readiness gates require it.
  7. **OAuth scope union.** First menu click on a freshly-generated sheet prompts the operator with the union of the bound shim's scopes + the library's scopes. Scopes MUST be kept minimal — `spreadsheets.currentonly` for the bound shim (read+write the active spreadsheet only), `script.scriptapp` if installable triggers are ever added (deferred to FW-0024 follow-up), and matching scopes on the library. The **launcher** itself (separate Apps Script project from the bound shim + library) carries the broader scopes per sub-decision 4: `spreadsheets`, `drive` (broadened from `drive.file` to enable template `getFileById` + `makeCopy`), `userinfo.email`. Pilot operators are inside the existing Test User pool, so unverified-app warnings are bypassed despite the broader scope.
  8. **Migration: existing M1.1-generated sheets without the bound shim.** Operators who already have launcher-generated sheets predating this checkpoint will not have the menu — their sheets were created via `SpreadsheetApp.create()`, not `makeCopy(template)`. Those operators must regenerate to gain the extractor. The pilot's small footprint (one operator, a few sheets) makes this a one-message-to-operator concern. No backward-compat shim, no automated migration tool.
  9. **What this enables that's currently blocked.** This architecture also unlocks FW-0024 (auto-repopulate empty call-point cells via `onEdit`) — a bound shim's `onEdit(e)` fires natively, no installable triggers required. **Apps Script simple triggers only fire on functions declared directly in the bound script, NOT on functions imported from a library**, so FW-0024's implementation pattern is: declare the `onEdit(e)` trigger entrypoint in the bound shim (e.g., `function onEdit(e) { RMLib.handleCallPointEdit(e); }`) and put the actual auto-fill logic in the library. FW-0024 is NOT shipped in M2 C9 (out of scope) but is no longer architecturally blocked once the bound + library pattern is in place.
- **Bidirectional audit dogfooded** (per `docs/delivery_plan.md` §14):
  - **Forward**: launcher produces template-copy spreadsheets (consumer = operator) → operator opens sheet → bound shim's `onOpen(e)` runs → menu visible → operator clicks "Extract Snapshot" → bound shim delegates to library → library reads sheet via DeveloperMetadata anchors per D-0043 → library emits Snapshot-shape JSON per `docs/snapshot_contract.md` § 5 + §11A → blob-download per D-0040 → operator saves file → Python CLI ingests. Library indirection makes the bound shim trivially small (delegate-only), so freezing the shim at copy time costs nothing — the shim's single responsibility is `MyLib.<func>(arg)`, and `MyLib` lives at one address that updates centrally. The Apps Script Library system is the correct primitive for this exact pattern. Manual one-time setup ceremony (template spreadsheet + bound script project + library project + library sharing settings) is acceptable because it happens once per environment, not per operator or per checkpoint.

### D-0042: Snapshot identity scheme — `snapshot_<spreadsheetId>_<extractionTimestamp>`
- **Date:** 2026-04-29
- **Status:** Accepted
- **Context:** `docs/snapshot_contract.md` §6 declares `metadata.snapshotId` as adapter-required and "always present and non-empty"; the field's concrete generation rule was deferred to the snapshot adapter implementation. M2 C9 needs to settle the rule. The launcher (M1.1) already uses a parallel `runId` pattern (`<vMMDDHHMMSS>` style) for tab-name pairing per `docs/sheet_generation_contract.md` §11A; reusing the same shape keeps the launcher and extractor identifiers conceptually aligned.
- **Decision:** Adopt **`snapshot_<spreadsheetId>_<extractionTimestamp>`** as the `snapshotId` generation rule:
  1. **Format.** Literal prefix `snapshot_`, followed by the source spreadsheet's File ID (Drive `FileResource.id`), followed by an underscore, followed by the extraction timestamp in ISO 8601 `YYYY-MM-DDTHH-MM-SS` form (colons replaced with hyphens for filename-safety).
  2. **Example.** `snapshot_1abc...XYZ_2026-04-29T14-30-15`.
  3. **Filename of the downloaded JSON file.** `<snapshotId>.json` per D-0040's filename convention. The on-disk artifact is therefore self-identifying — the operator can grep their downloads folder by spreadsheet ID or by extraction date.
  4. **Uniqueness.** Combination of source spreadsheet ID + extraction timestamp is unique per extraction run (two extractions of the same spreadsheet at sub-second resolution are not anticipated under operator workflow). If sub-second collisions emerge, append an integer suffix (`_2`, `_3`, …); not pinned in first release.
  5. **Stability across re-extraction.** `snapshotId` changes on every extraction (timestamp moves), even from the same source spreadsheet with no operator edits. This is intentional: the snapshot is a *point-in-time* extraction; same source + later timestamp is a new snapshot.
  6. **Relationship to `runEnvelope.runId`.** `snapshotId` and `runId` are NOT the same field. `runId` is execution-layer-supplied (selector boundary, per `docs/selector_contract.md` §9 item 3) and identifies a Python compute run; `snapshotId` is adapter-supplied and identifies a snapshot extraction. A given Python run consumes a snapshot whose `snapshotId` shows up in `metadata.snapshotId` and gets propagated through `runEnvelope` per the existing pipeline. They share a parallel timestamp-ish format but are distinct surfaces.
- **Bidirectional audit dogfooded** (per `docs/delivery_plan.md` §14):
  - **Forward**: snapshot consumer (Python parser) reads `Snapshot.metadata.snapshotId` per `docs/snapshot_contract.md` §6 → adapter produces it per the format above → declared in this decision and `docs/snapshot_adapter_contract.md`. ✓
  - **Reverse**: snapshot producer (Apps Script extractor) emits `snapshotId` → consumer's input shape (`metadata.snapshotId` field on the top-level Snapshot object) declares it required → matches `docs/snapshot_contract.md` §6 today. ✓
- **Rationale:** Embedding source identity (`spreadsheetId`) and time (`extractionTimestamp`) into the ID makes snapshots greppable and their filenames self-documenting, which is the right operator UX for a small-scale local-CLI pilot. Random UUIDs would be shorter but break the operator's ability to tell apart a snapshot by inspecting the filename. The launcher's `runId` pattern is similar in spirit; aligning them avoids two different conventions across the same architectural boundary.
- **Consequences:**
  - `apps_script/central_library/` (M2 C9 Phase 2) implements `_buildSnapshotId(spreadsheetId, timestamp)` per the format.
  - `docs/snapshot_adapter_contract.md` documents the format.
  - Downloaded filenames default to `<snapshotId>.json` (consequence of D-0040's filename convention pointing at this field).
- **Follow-up actions:**
  1. If sub-second extraction collisions are observed (currently not anticipated), append a numeric suffix and update this decision rather than reformatting the prefix.
  2. If the snapshot file is ever hosted somewhere shared (Drive folder, S3 bucket, etc.) instead of operator-local-only, revisit whether the spreadsheet ID embedded in the filename is appropriate to expose at that layer (Drive ID is not a credential but is a stable identifier).
- **Related docs:** `docs/snapshot_contract.md` §6 (declares `metadata.snapshotId`); `docs/decision_log.md` D-0040 (filename convention pointing at `snapshotId`), D-0036 (M2 C9 commitment), D-0041 (extractor placement); `docs/sheet_generation_contract.md` §11A (launcher's parallel `runId` pattern); `docs/snapshot_adapter_contract.md` (new — codifies the format).

### D-0043: Layout-drift handling — DeveloperMetadata-anchored extraction (no visible-text fallback)
- **Date:** 2026-04-29
- **Status:** Accepted
- **Context:** The Apps Script extractor must locate sheet regions (section headers, day-axis row, doctor rows, request cells, prefilled-assignment cells, call-point cells, Scorer Config rows) when reading a launcher-generated spreadsheet. Two anchoring strategies are available: (a) read the cell at known (row, col) coordinates derived from the template artifact; (b) read DeveloperMetadata tags attached to specific rows/cells at generation time and locate them via `Range.findDeveloperMetadata(...)`. Coordinate-based anchoring is brittle against operator edits (inserting / deleting rows shifts everything). DeveloperMetadata is row-anchored — when a row is inserted or deleted, the metadata travels with the row, and `findDeveloperMetadata` returns the new range. The launcher already uses DeveloperMetadata for the Scorer Config tab's per-row componentId tags (M2 C7). Earlier M2 C9 planning considered a hybrid "metadata first, fall back to visible-text lookup" approach to preserve backward compatibility with sheets generated before this checkpoint. D-0041's adoption of the bound-template + library pattern moots that fallback need: pre-D-0041 sheets do not have the bound shim either, so they cannot run the extractor regardless of anchoring strategy — they have to be regenerated. The fallback was carrying complexity for a backward-compat scenario that doesn't exist.
- **Decision:** **DeveloperMetadata-anchored extraction, with NO visible-text fallback.** Concretely:
  1. **Generation-time metadata.** The launcher (M1.1, extended in M2 C9 Phase 2) attaches DeveloperMetadata to:
     - **Sheet level** (already attached today): `rosterMonster:tabType` (`requestEntry` | `scorerConfig`), `rosterMonster:templateVersion`, `rosterMonster:runId`.
     - **Sheet level** (NEW — drives sub-decision 3 cardinality validation): `rosterMonster:expectedDoctorCount.<sectionKey>` per declared section (variable cardinality — depends on operator-supplied doctor counts at launch); `rosterMonster:expectedDayCount` (variable — depends on period length).
     - **Section header rows** (NEW): `rosterMonster:section` with value matching the template-declared `sectionKey` (`MICU` | `MICU_HD` | `MHD` for ICU/HD first release).
     - **Day-axis row** (NEW): `rosterMonster:dayAxis` (value: `true` — single-row anchor).
     - **Doctor rows** (NEW): `rosterMonster:doctorRow` with value `<sectionKey>:<doctorIndexInSection>` so the extractor can locate by both section and ordinal.
     - **Call-point rows** (NEW): `rosterMonster:callPointRow` with value matching the template-declared `pointRows.rowKey` (`MICU_CALL_POINT` | `MHD_CALL_POINT`).
     - **Prefilled-assignment shell rows** (NEW): `rosterMonster:assignmentRow` with value `<surfaceId>:<rowOffset>` for output-mapping locator alignment per `docs/snapshot_contract.md` §10.
     - **Scorer Config rows** (already attached today): `rosterMonster:componentId` per row.
  2. **Extraction algorithm — sheet-scoped + runId-paired finder.** All metadata-finder calls MUST be sheet-scoped (`Sheet.createDeveloperMetadataFinder()`), NOT spreadsheet-scoped (`Spreadsheet.createDeveloperMetadataFinder()`), because a single spreadsheet may carry multiple request-entry tabs (operator-duplicated for history, regenerated for a new period via the existing `generateIntoExistingSpreadsheet` path, etc.) and a workbook-scoped finder would silently mix anchors across tabs. Procedure: (a) read `SpreadsheetApp.getActiveSheet()`, validate sheet-level `tabType = requestEntry`, read the sheet's `runId`; (b) iterate `Spreadsheet.getSheets()` to find the unique sheet with `tabType = scorerConfig` and the matching `runId` (paired per `docs/sheet_generation_contract.md` §11A's tab-name pairing rule); (c) on each chosen tab, run sheet-scoped finders for each per-row anchor key and read cells at column offsets relative to the anchored row. Column offsets are stable across operator edits because the launcher locks column structure at generation time per `docs/sheet_generation_contract.md` §6 (operators may not add/remove columns). The full algorithm specification lives in `docs/snapshot_adapter_contract.md` §6.
  3. **Per-anchor completeness validation: cardinality + uniqueness + value-coverage.** Key-presence-only checks (a sheet-scoped finder returning *some* matches) are insufficient — partial-loss cases where one row's metadata gets deleted while siblings remain would silently emit an incomplete snapshot the parser/solver would treat as valid input (a real silent-data-loss path flagged by Codex on PR #95). The extractor MUST validate every multi-row anchor against the launcher-recorded expected counts before emitting any record. Concretely: read the sheet-level `expectedDoctorCount.<sectionKey>` + `expectedDayCount` anchors at extraction time; for each multi-row anchor (`doctorRow`, `dayAxis` cells, `callPointRow`, `componentId`, `assignmentRow`, `section`), check (a) cardinality matches expected, (b) values are unique (no duplicates), (c) the value set covers the expected set with no missing or unexpected entries. Any deviation surfaces a per-anchor `EXTRACTION_ERROR` naming the specific gap (`doctor row coverage mismatch on section MICU — expected 5, got 4`). The expected-set source-of-truth is the launcher-recorded sheet-level metadata for variable-cardinality anchors (doctor rows per section, day axis) and the template artifact + `docs/domain_model.md` §11.2 for fixed-cardinality anchors (call-point rows, component ids, assignment shell rows, sections). The full algorithm specification lives in `docs/snapshot_adapter_contract.md` §6.
  4. **NO visible-text fallback.** If a metadata anchor key returns zero matches entirely (the absent-anchor case, distinct from sub-decision 3's partial-loss case), the extractor surfaces a hard `EXTRACTION_ERROR` and refuses to emit a snapshot. The error message names which anchor was missing (`section:MICU not found`, `dayAxis row not found`, etc.) so the operator can identify the broken sheet and regenerate. Sheets predating this M2 C9 architecture (no bound shim, no new metadata) cannot be extracted by definition; this is the migration boundary captured in D-0041 sub-decision 8.
  5. **Operator row-edit drift tolerance.** Operators may insert or delete rows above / below an anchored row (e.g., adding personal notes); DeveloperMetadata follows the row, so `findDeveloperMetadata` returns the new location and extraction continues correctly. Column inserts / deletes are still disallowed per the existing `docs/sheet_generation_contract.md` §6 / §7 protection posture. Note: deleting a row that *carries* a metadata anchor removes the anchor along with the row — the cardinality validation in sub-decision 3 catches this case.
  6. **Extractor → snapshot → parser admission discipline.** A snapshot that is structurally complete (all metadata anchors found and pass cardinality validation) but carries operator-edited cells violating parser-stage rules (mis-signed weights, malformed dates, etc.) is admission-blocked at parser stage per `docs/parser_normalizer_contract.md` §14, NOT at extraction stage. The extractor's job is fidelity (capture what was seen) — semantic interpretation is parser territory per `docs/snapshot_contract.md` §3. This decision affects extractor-side error surfaces only; parser-side admission is unchanged.
- **Bidirectional audit dogfooded** (per `docs/delivery_plan.md` §14):
  - **Forward**: snapshot consumer (Python parser) reads `Snapshot.doctorRecords` / `dayRecords` / etc. with `sourceLocator.path = { sectionKey, doctorIndexInSection }` etc. per `docs/snapshot_contract.md` §10 → extractor produces those locators by reading metadata anchors per the algorithm above → declared in this decision and `docs/snapshot_adapter_contract.md`. ✓
  - **Reverse**: extractor reads metadata anchors → launcher MUST attach those metadata anchors at generation time → declared in this decision (sub-decision 1) and updated in `docs/sheet_generation_contract.md` (this change round adds the new anchors to the contract's section on launcher-attached DeveloperMetadata). ✓
- **Rationale:** Coordinate-based extraction worked in the test bridge because the bridge is one-shot and the dev-copy xlsx is known to have a specific layout. Production must tolerate operator row inserts (a real workflow: adding a "PERSONAL NOTES" row above the section), and DeveloperMetadata is the only Apps Script primitive that survives row drift. The visible-text fallback was a hedge against pre-D-0041 sheets that's no longer needed once the bound-shim pattern itself requires regeneration. Hard-failing on missing anchors is preferable to silent fallback because a missing anchor signals a real defect (corrupted spreadsheet, manual deletion of an anchor row, extracting from a wrong-template sheet) that a fallback would mask.
- **Consequences:**
  - `apps_script/launcher/src/Layout.gs` (existing) — extended in M2 C9 Phase 2 to attach the new DeveloperMetadata at generation time.
  - `apps_script/central_library/extractor.gs` (new in M2 C9 Phase 2) — implements the metadata-finder algorithm and the hard-fail-on-missing-anchor surface.
  - `docs/sheet_generation_contract.md` — updated in this change round to declare the new launcher-side metadata anchors as part of the structural surface (parallel to §11A's existing Scorer Config tab metadata).
  - `docs/snapshot_adapter_contract.md` (new in this change round) — declares the metadata-anchored extraction algorithm + the no-fallback discipline.
  - Pre-D-0041 sheets cannot be extracted; operators receive a clear migration message via D-0041 sub-decision 8 and `docs/snapshot_adapter_contract.md` Migration section.

  1. If a real operator workflow surfaces where a metadata anchor legitimately disappears (e.g., operator deletes a section the roster doesn't use), revisit toward "soft-missing anchor → empty record collection" rather than "hard-fail extraction" — recorded as future-work if it appears.

### D-0044: Writeback transport mechanic — file upload via launcher form
- **Date:** 2026-04-30
- **Status:** Accepted
- **Context:** `docs/decision_log.md` D-0040 settled the snapshot-extraction (inbound) transport at "browser-download of a JSON file" and noted the writeback (outbound) side would adopt the same mechanic in reverse. D-0040's wording — "the operator drags / pastes it into the launcher's writeback form" — was deliberately vague between drag-and-drop, copy-paste textarea, and `<input type="file">` upload. M3 C1 (writeback implementation) needs a concrete UI choice. `docs/writeback_contract.md` §6.3 / §8 / §22 leave the concrete launcher transport as implementation-slice, so settling it here doesn't require a writeback `contractVersion` bump.
- **Decision:** **File upload via the launcher web form** is the operator-facing transport for writeback. Concretely:
  1. The launcher Web App (`apps_script/launcher/`) gains a writeback route exposing an HTML form with an `<input type="file" accept=".json,application/json">` field.
  2. Operator selects the JSON file produced by the local Python CLI (`python -m rostermonster.run --snapshot ... --output result.json`) — see D-0047 for the rule that the CLI's output IS the writeback envelope.
  3. The form reads the file content client-side (standard browser `FileReader` API) and passes the JSON string to a server-side handler. The dispatch mechanism — `google.script.run.<handlerName>(jsonString)` for the embedded-HtmlService-form pattern OR a real HTTP `<form action="..." method="post">` captured by the launcher's `doPost(e)` for the standalone-form pattern — is implementation-slice (M3 C1 Phase 2). The contract pins only that the file content reaches the server side as a JSON string the server parses; the two Apps Script dispatch mechanisms are mutually exclusive within a single form's submit path and the implementation MUST pick one.
  4. Maximum file size is bounded by Apps Script's request-payload limit (currently 50 MB compressed / 30 MB uncompressed for HTML form posts; first-release writeback envelope at ICU/HD scale is well under 100 KB so this is not a concern).
  5. The form does not accept drag-and-drop or clipboard paste in first release; those are deferred as UX-polish entries if pilot operator feedback warrants.
  6. No new OAuth scopes. The launcher's existing `drive` scope (broadened in M2 C9 per D-0041 sub-decision 4) is sufficient — the file upload is operator-mediated and doesn't require Apps Script to read the operator's Drive directly.
- **Bidirectional audit dogfooded** (per `docs/delivery_plan.md` §14):
  - **Forward**: writeback consumer (Apps Script) expects to receive a JSON envelope per `docs/writeback_contract.md` §9 → operator uploads via file picker → server `JSON.parse`s → consumer's input shape satisfied. ✓
  - **Reverse**: writeback producer (Python CLI's output JSON file per D-0047) → file upload widget consumes the file → no producer-side changes needed. ✓
- **Rationale:** File upload is the simplest mechanic that:
  - matches the operator's existing mental model from M1.1 (forms with file inputs are commonplace),
  - avoids the size constraints of clipboard paste (operator workflow shouldn't break if envelope grows past clipboard limits),
  - reuses the launcher's existing form infrastructure rather than building a new transport surface,
  - mirrors the symmetry of D-0040's browser-download for the inbound side (operator manages a single local file; no Drive API on either side).
  Drag-and-drop adds JavaScript complexity for marginal UX gain; can be added later as polish. Drive Picker would require additional OAuth scopes.
- **Consequences:**
  - `docs/writeback_contract.md` §6.3 / §22 receive an addendum citing this decision as the resolved transport (the §6.3 / §22 deferral language is retained for forward-compat — drag-and-drop and Drive Picker remain implementation-slice options for future polish — but the joint-scoping commitment from D-0036 is now fully resolved on both directions).
  - M3 C1 Phase 2 implementation extends the launcher Web App with: (a) a new HTML view for the writeback form, (b) a route or query-parameter dispatch from `doGet(e)` to render either the existing generation form or the new writeback form, (c) a server-side handler accepting the JSON content + invoking the writeback library — exposed either as a named function reachable via `google.script.run` (embedded-form pattern) OR as a `doPost(e)` entrypoint (HTTP-POST-form pattern) per D-0044 sub-decision 3, exactly one of which is selected at implementation time.
  - The launcher's existing `drive` OAuth scope (per D-0041) covers the writeback path; no new scope addition.
  - File-size limit is not a first-release concern (envelopes are ≪50 MB); revisit only if envelope shape grows to include large sidecar content (currently excluded per `docs/writeback_contract.md` §8 + §9).
- **Follow-up actions:**
  1. If pilot operator surfaces friction with the file-upload-then-form-submit two-step flow, revisit toward drag-and-drop or auto-submit-on-file-pick. Captured as future-work polish if it surfaces.
  2. If the writeback envelope ever needs to carry FULL-retention selector sidecars (currently prohibited per `docs/writeback_contract.md` §8 + §9), revisit the file-size limit and potentially the transport mechanic.

### D-0045: Writeback envelope shape — single JSON wrapper carrying FinalResultEnvelope + snapshot bundle + doctorIdMap
- **Date:** 2026-04-30
- **Status:** Accepted
- **Context:** `docs/writeback_contract.md` §9 declares the writeback consumer's three input categories (`finalResultEnvelope`, `snapshot`, `doctorIdMap`) but explicitly leaves "concrete JSON envelope schema layout" deferred (§6.3, §8, §22). M3 C1 needs a concrete shape so the Python CLI knows what to emit and the Apps Script writeback library knows what to ingest. The simplest operator-facing UX (D-0044's file-upload mechanic) requires the operator to upload a SINGLE file, not three; the envelope MUST therefore wrap all three categories into one JSON document.
- **Decision:** Writeback envelope is a single JSON wrapper at top level with the following shape categories (concrete key ordering / nesting / Unicode normalization remain implementation-slice per `docs/writeback_contract.md` §22; this decision pins the categories of content):
  ```
  {
    "schemaVersion": 1,
    "finalResultEnvelope": { ...selector v2 FinalResultEnvelope per docs/selector_contract.md §10... },
    "snapshot": {
      "columnADoctorNames": [...],
      "requestCells": [...],
      "callPointCells": [...],
      "prefilledFixedAssignmentCells": [...],
      "shellParameters": { department, periodStartDate, periodEndDate, doctorCountByGroup }
    },
    "doctorIdMap": [{ "doctorId": "...", "sectionGroup": "...", "rowIndex": N }, ...]
  }
  ```
  Concrete properties:
  1. **`schemaVersion`**: integer, currently `1`. Distinct from `docs/writeback_contract.md` `contractVersion` per §20.2 (the writeback artifact's `contractVersion` IS the schema-version surface for the *writeback tab*; this `schemaVersion` is for the *upload envelope* the operator uploads). Bumps when the envelope's required-categories set changes in a v1-targeted-reader-breaking way.
  2. **`finalResultEnvelope`**: the unmodified `FinalResultEnvelope` produced by the selector, per `docs/selector_contract.md` §10.
  3. **`snapshot`**: a SUBSET of the original snapshot the operator extracted in M2 C9 — only the categories the writeback library needs to reconstruct shell content per `docs/writeback_contract.md` §10.1. Full snapshot fields (e.g., `requestRecords` raw text, `prefilledAssignmentRecords` raw text per `docs/snapshot_contract.md` §9 / §11) are repackaged into the simpler operator-cell shapes the writeback contract names. The Python CLI is responsible for projecting from the full snapshot to this writeback-subset.
  4. **`doctorIdMap`**: array of `{doctorId, sectionGroup, rowIndex}` triples derived by Python from the original snapshot's doctor records.
- **Bidirectional audit dogfooded** (per `docs/delivery_plan.md` §14):
  - **Forward**: writeback consumer (Apps Script library) reads `envelope.finalResultEnvelope`, `envelope.snapshot.*`, `envelope.doctorIdMap` → Python CLI emits exactly those fields → declared in this decision and `docs/writeback_contract.md` §9. ✓
  - **Reverse**: writeback producer (Python CLI) emits the wrapper shape → operator uploads to launcher → consumer's input shape (writeback contract §9 categories) is satisfied → declared here and in D-0047. ✓
- **Rationale:** Single wrapper file is the cleanest operator UX (one upload, not three), avoids partial-upload corruption (operator can't accidentally upload only result.json without the snapshot), and puts assembly responsibility on the Python CLI (which has all three inputs at hand naturally). Pinning categories without pinning concrete schema layout matches `docs/selector_contract.md` §14.3's pattern and `docs/writeback_contract.md` §6.3 / §22's existing deferral discipline.
- **Consequences:**
  - `python/rostermonster/run.py` is extended in M3 C1 Phase 2 to emit the wrapper envelope as its output instead of the bare `FinalResultEnvelope`. The CLI gains a `--writeback-ready` flag (default `true` in first release) controlling this; pure `FinalResultEnvelope`-only output remains available via `--writeback-ready=false` for callers who don't intend to writeback.
  - Apps Script writeback library validates the wrapper's required categories at ingestion time and surfaces validation failure through the runtime-error state per `docs/writeback_contract.md` §17.3.
  - `docs/writeback_contract.md` §9 receives an addendum subsection (provisionally §9.1 or §9-extension) declaring the wrapper-shape categories normatively, distinct from the concrete schema layout which remains implementation-slice.
- **Follow-up actions:**
  1. If a future selector strategy emits a meaningfully different `FinalResultEnvelope` shape that requires writeback to consume new categories, bump the wrapper's `schemaVersion` per §1 of this decision.
  2. The `snapshot` subset's exact field shapes are pinned in M3 C1 Phase 2 alongside the Python projection helper that produces them; document the shape in `docs/writeback_contract.md` at that point.
- **Related docs:** `docs/writeback_contract.md` §9 (input shape — primary contract surface); `docs/selector_contract.md` §10 (FinalResultEnvelope shape); `docs/snapshot_contract.md` §5..§11 (snapshot shape Python projects from); `docs/decision_log.md` D-0044 / D-0046 / D-0047 (companion M3 C1 Phase 1 decisions); `python/rostermonster/run.py` (CLI extended in Phase 2 to emit the wrapper).

### D-0050: Dual-track Python architecture — shared compute core + thin CLI wrapper + thin HTTP wrapper
- **Date:** 2026-04-30
- **Status:** Accepted
- **Context:** D-0049 reframes M4 to deliver both a local CLI (maintainer-side dev velocity) AND a Cloud Run HTTP service (operator-side one-click UX). Both surfaces must run the SAME compute code path — solver-strategy experiments performed locally must be deployable to cloud without a translation layer, and any production-side defect must be reproducible from the local CLI. This decision pins the Python module structure that enforces the "one compute core, two thin wrappers" discipline.
- **Decision:** Refactor `python/rostermonster/run.py` to factor compute orchestration into a shared core function callable from both wrappers:
  1. **Shared compute core** (provisional path: function within `python/rostermonster/run.py` or moved to `python/rostermonster/pipeline.py`) — accepts a parsed `Snapshot` + a `RunConfig` (carrying `maxCandidates`, `seed`, `runEnvelope` overrides, retention mode, etc.) and returns the wrapper envelope per `docs/decision_log.md` D-0045. Contains all parser → solver → scorer → selector orchestration. Pure-function, testable in isolation.
  2. **CLI wrapper** (`python/rostermonster/run.py` `main()`) — argparse + JSON file I/O around the shared core. Existing flags (`--snapshot`, `--output`, `--max-candidates`, `--seed`, `--retention`, `--sidecar-dir`, `--writeback-ready`) preserved; behavior unchanged for existing callers.
  3. **HTTP wrapper** (provisional path: new module `python/rostermonster_service/app.py` + `Dockerfile` at repo root or a new `cloud_compute_service/` subdirectory) — Flask single-endpoint service receiving `{snapshot, optionalConfig?}` per `docs/cloud_compute_contract.md` §6, calling the same shared core, returning the wrapper envelope (or structured error envelope).
- **Bidirectional audit dogfooded** (per `docs/delivery_plan.md` §14):
  - **Forward**: cloud HTTP wrapper consumes the shared core's output (wrapper envelope per D-0045) → CLI wrapper produces the same shape from the same core → consumer's input shape is the writeback envelope per `docs/writeback_contract.md` §9. ✓
  - **Reverse**: shared core's output is consumed by both wrappers → both wrappers emit the same shape → declared in this decision and in D-0045's wrapper schema. ✓
- **Rationale:** Two surfaces sharing one compute core is the only way to keep solver-strategy experiments safely portable between local development and cloud production. Splitting the implementation across two modules (one for CLI, one for HTTP) would create drift opportunities — bug fixes or strategy changes might land in one wrapper but not the other. The thin-wrapper discipline mirrors the established `docs/blueprint.md` separation between adapter layers and core compute (D-0017 / D-0018 stack split): both wrappers are adapters; the core is invariant.
- **Consequences:**
  - M4 C1 Phase 2 work refactors `run.py` to expose the shared core, then adds the HTTP wrapper alongside.
  - Both wrappers depend on the same `rostermonster` package — no compute-logic duplication. Changes to the parser, solver, scorer, or selector modules propagate to both wrappers automatically.
  - The Cloud Run container builds the entire `python/rostermonster/` module; the wrapper is a thin Flask app on top.
  - HTTP-wrapper-specific concerns (request validation, error-envelope formatting, observability hooks) live in the HTTP wrapper module, NOT in the shared core. The CLI wrapper similarly keeps argparse and file-I/O concerns out of the core.
- **Follow-up actions:**
  1. If a future use case emerges where cloud and local compute paths need to genuinely diverge (e.g., cloud-only batch parallelism, local-only debug instrumentation), revisit whether the dual-track architecture needs a third "service-specific" layer or whether divergence belongs in configuration.
  2. If the HTTP wrapper grows beyond a thin Flask layer (e.g., needs background workers, queue dispatch, streaming responses), reconsider whether the current "single Cloud Run service" deployment model still fits or whether a more elaborate orchestration is needed.
- **Related docs:** `docs/delivery_plan.md` M4 C1 (active checkpoint deploying the dual-track architecture); `docs/cloud_compute_contract.md` (new contract introduced in this PR pinning the HTTP wrapper's shape); `docs/decision_log.md` D-0049 (M4 reframing this architecture serves), D-0051 (cloud platform / auth), D-0052 (Apps Script library reorganization on the consumer side); `python/rostermonster/run.py` (CLI module to be refactored).

### D-0051: Cloud platform — Cloud Run + Apps Script IdentityToken auth + consolidated GCP project
- **Date:** 2026-04-30
- **Status:** Accepted
- **Context:** M4 C1 needs a cloud-side compute platform for the HTTP wrapper per D-0050. Several options exist (Cloud Run, Cloud Functions Gen 2, App Engine, Compute Engine, third-party FaaS). The choice determines deployment mechanics, cold-start behavior, request timeout, scaling semantics, and crucially the auth model between Apps Script (bound shim) and the cloud service. The Apps Script side must invoke the cloud service from operator OAuth context without introducing a new credential surface.
- **Decision:** Cloud Run is the M4 C1 platform. Auth model: bound shim authenticates to Cloud Run via Apps Script's `ScriptApp.getIdentityToken()` → Cloud Run IAM (`roles/run.invoker`). All Apps Script projects (launcher, bound shim, central library) and the Cloud Run service share **one consolidated GCP project: `RosterMonsterV2`**. Concretely:

IAM binding `roles/run.invoker` granted to the launcher's OAuth test-user list (the same emails currently allowlisted on the OAuth consent screen). The `muteHttpExceptions: true` flag is required so the bound shim can dispatch on HTTP status codes (401/403 for auth failures per `docs/cloud_compute_contract.md` §7.5; 200 for application-level state per §10) rather than have Apps Script throw on every non-2xx response. No shared secret, no API key, no service account JSON.

     This sub-decision was revised twice during M4 C1 Phase 2: first on PR #103 Codex review (the original D-0051 draft under-listed `script.external_request` / `openid` / `userinfo.email`), and second during live testing where `spreadsheets.currentonly` proved insufficient for `openById` even on the bound spreadsheet itself.
  4. **Container**: Python 3.12-slim base + `requirements.txt` installing `flask` + the local `rostermonster` package. Build via `gcloud builds submit` or `gcloud run deploy --source` (Cloud Build under the hood). Versioning: each deploy bumps a Cloud Run revision; rollback is one-click via Cloud Run console.

### D-0053: Solver seed default = random per invocation; explicit `--seed` / `optionalConfig.seed` opts into fixed reproducibility
- **Date:** 2026-04-30
- **Status:** Accepted (amends D-0051 sub-decision 6 + `docs/cloud_compute_contract.md` §9.3)
- **Context:** D-0051 sub-decision 6 + `docs/cloud_compute_contract.md` §9.3 (settled in PR #103) pinned both surfaces' default seed to `_DEFAULT_SEED = 20260504` from `python/rostermonster/run.py`, hand-picked in M2 as the May 2026 fixture's period start date. Two purposes for the fixed seed at the time: (a) test reproducibility — tests that don't pass `--seed` explicitly land on a deterministic RNG path; (b) maintainer-side debugging velocity — re-running the CLI against the same snapshot returns identical results so changes are isolated to code, not RNG. Surfaced 2026-04-30 during M4 C1 Phase 1 closeout discussion: the same rationale that motivates `--seed` for reproducibility also produces a counter-intuitive operator-facing behavior — the bound shim's "Solve Roster" menu, on the same snapshot, would return the byte-identical roster on every click, with no path for the operator to "try again" and explore the search space without knowing to pass a different seed via the optionalConfig surface. For a one-click operator UX surface (M4 C1's quality bar), a fresh attempt on each invocation is the natural expectation.
- **Decision:** **Default seed when omitted is randomly chosen per invocation; explicit `--seed` / `optionalConfig.seed` opts into the prior fixed-seed reproducibility behavior.** Concretely:
  1. **Default mechanism**: when neither the CLI's `--seed` flag nor the cloud `optionalConfig.seed` field is passed, the implementation picks a fresh seed via `random.randint(0, 2**31 - 1)` at the start of the run. The chosen seed is recorded in `RunEnvelope.seed` (per `docs/selector_contract.md` v2 §9 item 3 — already required field) so the operator can reproduce the run later by passing `--seed <that-value>`.
  2. **Both surfaces share this mechanism**: the CLI and the cloud HTTP wrapper both call into the same shared compute core per D-0050; the random-default behavior lives in the shared core, not in either wrapper's argparse/Flask layer.
  3. **Reproducibility opt-in**: passing `--seed N` on the CLI or `optionalConfig.seed=N` in the cloud request both pin the seed to `N` and produce byte-identical results on re-runs. The prior `_DEFAULT_SEED = 20260504` constant is removed from `run.py`; tests and benchmarks that rely on byte-identical determinism MUST pass `--seed` explicitly (most already do; see Consequences).
  4. **D-0050's dual-track parity claim is preserved with a refined precondition**: "same `(snapshot, optionalConfig)` produces byte-identical envelopes" continues to hold WHEN `optionalConfig.seed` is explicitly set to the same value across both surfaces. With omitted seed, each surface picks its own random seed, so envelopes differ — that's expected behavior, not a parity violation. The parity claim never required default behavior to match across re-runs; it required behavior at same input to match.
- **Bidirectional audit dogfooded** (per `docs/delivery_plan.md` §14):
  - **Forward**: CLI consumer expects `RunEnvelope.seed` to be the actual seed used for the run → CLI's shared-core call captures the resolved seed (random or explicit) into the envelope → declared in `docs/selector_contract.md` v2 §9 item 3 as a required field. ✓
  - **Reverse**: cloud HTTP wrapper produces a wrapper envelope whose `runEnvelope.seed` MUST be reproducible → CLI consumer accepts that seed via `--seed N` to replay → bound by D-0050's shared-core invariant. ✓
- **Rationale:** Random-by-default + explicit-opt-in matches operator mental models on both sides:
  - Operator clicks "Solve Roster" → gets a roster. Clicks again → gets a *different* roster (a fresh attempt at the search space). Both clicks' chosen seeds are persisted in the writeback tab's traceability footer (already wired per `docs/writeback_contract.md` §16.1 / §16.2 — the run envelope flows through unchanged); operator can pin a specific roster by reading the seed off the tab and replaying with `--seed N`.

### D-0054: Cloud Run auth model — public service + Flask-side ID-token validation + email allowlist (amends D-0051 sub-decision 2)
- **Date:** 2026-04-30
- **Status:** Accepted (amends D-0051 sub-decision 2 + `docs/cloud_compute_contract.md` §7)
- **Context:** D-0051 sub-decision 2 pinned the auth model as IAM-protected Cloud Run with `Authorization: Bearer <ScriptApp.getIdentityToken()>`. M4 C1 Phase 2 live testing surfaced two blockers with that design:
  1. **Audience mismatch.** `ScriptApp.getIdentityToken()` issues an OIDC token whose `aud` claim equals the Apps Script project's linked GCP OAuth client ID, NOT the Cloud Run service URL. Cloud Run's IAM check requires `aud = <service URL>` exactly; with mismatched audience, every request is rejected `401 Unauthorized` before the service code runs. Confirmed in Cloud Run logs: "The request was not authorized to invoke this service." (https://cloud.google.com/run/docs/troubleshooting#401).
  2. **Authorization header stripping.** Cloud Run drops the standard `Authorization` header on `--allow-unauthenticated` services to prevent token-leakage to public services (https://cloud.google.com/run/docs/securing/managing-access#allow-unauthenticated). So switching to public Cloud Run + app-level token validation can't reuse the standard header — the bound shim's token never reaches Flask code.
- **Decision:** **Cloud Run runs `--allow-unauthenticated`. Operator's ID token travels in a custom `X-Auth-Token` header. Flask validates the token via `google.oauth2.id_token.verify_oauth2_token` (signature + expiry + email_verified) and gates by checking the `email` claim against a comma-separated `ALLOWED_EMAILS` env var.** Concretely:
  1. **Cloud Run deploy flag**: `--allow-unauthenticated` instead of the previous `--no-allow-unauthenticated`. Service is public at the platform layer; the auth check happens entirely in Flask.
  2. **Token transport**: bound shim sends `X-Auth-Token: <ScriptApp.getIdentityToken()>` instead of `Authorization: Bearer ...`. Cloud Run does NOT strip non-`Authorization` headers, so the custom header arrives at Flask intact.
  3. **Token validation**: `google.oauth2.id_token.verify_oauth2_token(token, requests.Request(), audience=None)`. `audience=None` deliberately skips the `aud` check (the whole reason this decision exists); signature, expiry, and `email_verified` are all still validated by Google.
  4. **Email allowlist**: Flask reads `ALLOWED_EMAILS` env var (comma-separated, lower-cased), checks the validated token's `email` claim against it. Mismatch returns a structured `INPUT_ERROR` per `docs/cloud_compute_contract.md` §10 with `code: "EMAIL_NOT_ALLOWLISTED"`.
  5. **Bound shim manifest**: `script.external_request` + `userinfo.email` + `openid` scopes from D-0051 sub-decision 3a are unchanged. The `openid` scope is what makes `ScriptApp.getIdentityToken()` return a valid OIDC token; `userinfo.email` populates the `email` claim Flask validates against; `script.external_request` is what `UrlFetchApp.fetch()` requires.
  6. **Local-test scaffolding**: Flask app honors a `DISABLE_AUTH_FOR_LOCAL_TESTING` env var that skips the auth path entirely. Set in `python/tests/test_service.py` so unit tests don't need to mock Google's token-verification endpoint. Production Cloud Run never sets this env var.
  7. **Service account flow remains future-work**: the cleanest long-term architecture is a service account with `roles/run.invoker` + Apps Script OAuth2 library minting OIDC tokens with the correct audience. That's the Google-recommended Cloud Run + Apps Script pattern. Deferred for first release because (a) it requires storing service-account JSON in Apps Script Script Properties (a credential surface), (b) it requires importing the apps-script-oauth2 library, (c) operator-identity gating becomes service-account-mediated rather than direct (operator-attribution lost in Cloud Run logs without app-level email surfacing). Captured as future-work entry. The Flask-side allowlist approach in this decision delivers operator-identity gating with simpler setup, fitting the M4 C1 "minimum demo" quality bar from D-0049.
- **Bidirectional audit dogfooded** (per `docs/delivery_plan.md` §14):
  - **Forward**: Cloud Run consumes `X-Auth-Token` carrying an ID token → bound shim sends it via `UrlFetchApp.fetch(..., headers: {'X-Auth-Token': token})` → declared in this decision and in `docs/cloud_compute_contract.md` §7. ✓
  - **Reverse**: Flask validates the email claim via the `ALLOWED_EMAILS` env var → bound shim's token includes the operator's email per `userinfo.email` scope (D-0051 sub-decision 3a) → operator-identity gating is correctly delegated to the Flask-side allowlist. ✓
- **Rationale:** App-level operator-allowlist via X-Auth-Token + verified Google ID token + email check delivers operator-identity gating without requiring service-account credential management. Tradeoffs vs the original IAM-based design:
  - **Loses**: platform-level (Cloud Run IAM) auth gate. Anyone can hit the Flask code path; auth check happens in app, not at the load balancer.
  - **Keeps**: per-operator identity gating (the original D-0051 intent — verified Google identity + allowlist check).
  - **Pilot-scope-acceptable**: at one-operator pilot scale with internal-URL distribution, the small DDoS / spam surface from a public service is acceptable. Cloud Run's free-tier covers far more requests than pilot needs; cost-runaway protection from `max-instances=5` is unchanged.
  - **Reversibility**: when pilot scope grows or formal least-privilege posture is required, switch to the service-account flow per future-work — a backwards-compatible change (token transport stays, just becomes service-account-mediated).
  Service account would have been "more correct" but introduces credential management complexity and obscures operator identity in Cloud Run logs. The pilot trade-off favors the simpler app-level check.
- **Consequences:**
  - `python/rostermonster_service/app.py`: adds `_check_operator_allowlist()` that reads `X-Auth-Token` header, verifies via google-auth, checks email against `ALLOWED_EMAILS` env var. Auth failures surface as `INPUT_ERROR` with structured `error.code` per cloud-compute-contract §10.
  - `cloud_compute_service/requirements.txt`: gains `google-auth` + `requests` runtime deps (transitive: cryptography, etc.). Image size grows by ~5 MB; cold start latency impact negligible (~50ms).
  - `apps_script/bound_shim/src/Menu.gs`: `UrlFetchApp.fetch` headers change from `{Authorization: 'Bearer ' + token}` to `{'X-Auth-Token': token}`. Same `ScriptApp.getIdentityToken()` source — token stays unchanged, only transport header changes.
  - Cloud Run service redeployed with `--allow-unauthenticated --set-env-vars ALLOWED_EMAILS=<email1>,<email2>,...`. The IAM `roles/run.invoker` binding from the original D-0051 setup is now functionally redundant (Cloud Run is public) but kept as defense-in-depth: if the service is later switched back to `--no-allow-unauthenticated`, the bindings are still there.
  - `docs/cloud_compute_contract.md` §7 (auth) rewritten to reflect the public-service + app-level-allowlist model. §7.5 (no fallback to public access) is replaced with §7.5 (X-Auth-Token transport rationale). §10.1 INPUT_ERROR codes gain auth-related codes (MISSING_AUTH_TOKEN, INVALID_TOKEN, EMAIL_NOT_VERIFIED, EMAIL_NOT_ALLOWLISTED).

The email allowlist constrains identity, not service. ### D-0056: Analyzer top-K selection — pure score-rank, no diversity heuristic at the analyzer stage
- **Date:** 2026-05-04 *(amended in-PR after Codex round-4 review surfaced that the original draft tiebreak — totalScore desc, then candidateId asc — was NOT equivalent to `HIGHEST_SCORE_WITH_CASCADE`'s two-level cascade per `docs/selector_contract.md` §12.2, which goes through `pointBalanceGlobal` → `crReward` BEFORE `candidateId`. Sub-decision 1 now mirrors the selector cascade exactly so §11.1's equivalence claim actually holds. Round-2 had also tightened "lexicographic ASCII" → "numeric ascending" on the `candidateId` fallback; the round-4 fix adds the two preceding cascade levels.)*
- **Status:** Accepted (M5 C1 design thread closure)
- **Context:** D-0055 sub-decision 4 places top-K + diversity in the analyzer (not the selector) but explicitly defers the diversity criterion to the M5 C1 design thread. Two candidate dispositions surfaced in the thread: (a) **pure score-rank** — sort candidates by `totalScore` descending, take the top K with `candidateId` ascending tiebreak, no diversity selection; (b) **greedy score-then-Hamming-distance** — pick highest-score candidate first, then for each subsequent slot pick the highest-score candidate whose assignment matrix differs from every already-picked candidate by at least D day-doctor cells, with D operator-tunable. Disposition (b) was the initial draft; on review the maintainer challenged it on architectural grounds (diversity-aware subset selection literature — Determinantal Point Processes per Kulesza & Taskar 2012, submodular maximization per Lin & Bilmes 2011, max-min diversification in recommender systems — applies in contexts where the scoring function is acknowledged-imperfect AND there is no separate exploration mechanism upstream; in our setup the solver IS the exploration mechanism, so layering diversity at the analyzer mixes two concerns that should stay separate).
- **Decision:** **Pure score-rank top-K, no diversity heuristic at the analyzer stage.** Concretely:
  1. **Selection rule**: sort the FULL sidecar's candidates by `ScoreResult.totalScore` descending; tiebreak ties on equal `totalScore` by **mirroring `HIGHEST_SCORE_WITH_CASCADE`'s two-level cascade per `docs/selector_contract.md` §12.2 exactly** — (a) higher `pointBalanceGlobal`, (b) higher `crReward`, (c) lowest numeric `candidateId` as final fallback (`candidateId` is a run-monotonic dense integer per `docs/selector_contract.md` §16.1; numeric comparison, NOT lexicographic ASCII). The full cascade (NOT just the `candidateId` fallback in isolation) is what guarantees the analyzer's rank-1 always equals the selector's BEST_ONLY winner under `docs/analysis_contract.md` §11.1's equivalence claim. Take the first `min(requested, candidatesAvailable)` entries.
  2. **K bounds**: tunable via Python CLI flag (planned `--top-k N` on the analyzer subcommand); default `5`; bounds `[1, 20]` inclusive; `requested > 20` or `requested < 1` is fail-loud (analyzer raises a structured rejection rather than silently clamping). If the K candidates are near-duplicates, that is signal — the operator is meant to see "the solver thinks these are all equivalent" rather than have the analyzer fabricate spread the solver did not produce. Tier 5's `pairwiseHammingDistance` matrix (`docs/analysis_contract.md` §10.7) is the operator's diagnostic for this.
  5. **Failure-branch + BEST_ONLY are out of scope**: analyzer requires FULL retention (`docs/analysis_contract.md` §9.1) and rejects the failure branch fail-loud (§9.2). No silent degradation to "K=1, return the winner" under BEST_ONLY.

- **Rationale:** Pure score-rank cleanly separates concerns: the **solver is the exploration mechanism**; the **analyzer is the passive observer**. For today's `SEEDED_RANDOM_BLIND` solver, candidates are diverse by construction (different seeds explore different regions of the assignment space), so the diversity heuristic would solve a problem that does not yet exist. For future LAHC (M6 territory), if the solver produces near-duplicate top-K, that itself is signal — operator should see "solver thinks these are all equivalent" and that surfaces a solver-strategy concern, not an analyzer-side concern. The fail-loud K-bounds discipline is consistent with the architecture's broader fail-loud preference (parser admission, scorer pointRules per D-0038, etc.) — silent clamps create a "did I get what I asked for" ambiguity the operator should never have to investigate.
