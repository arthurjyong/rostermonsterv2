### 7) Parallel solver (Cloud Batch + intra-request K-fanout, async UX) — *Active (activated 2026-05-10; UX pivoted sync → async on 2026-05-11 per D-0071; architecture amended to single-VM 2026-05-11 per Codex P1.7 fix)*
Parallelize K-trajectory LAHC execution via Cloud Batch to escape the GIL-bound single-thread bottleneck identified during M6 C4 cloud benchmarking. **Deployment shape pivoted twice during M7 C3:** (1) D-0070's original framing was 13-VM dense pack (`c3-highcpu-8` × 13, `multiprocessing.Pool(8)` × 13 = K=104) inside a sync request with Cloud Run timeout 250s; (2) D-0071's first framing kept the 13-VM dense pack but moved UX to async with a 2nd task group running the finalizer after the workers via `SUCCEEDED_OR_FAILED` dependency; (3) **Codex P1.7 amendment** caught that Cloud Batch v1 only supports ONE task group (`submitJob` rejects multi-task-group jobs), forcing a SECOND pivot to **single-VM dense pack** — one `c3-highcpu-88` VM, `multiprocessing.Pool(88)` × 1 task = K=88; the finalize step (aggregate + score + select + analyzer + POST callback) runs INLINE in the same Python process after `Pool.close() + .join()`. The sync→async pivot was driven by M7 C2 perftest evidence (real production roster at K_approved=104 returned 5/13 tasks unfinished at the 240s sync deadline, partial K'=64); the single-VM pivot was driven by Cloud Batch v1's single-task-group restriction. K reduces 104 → 88 (statistically equivalent — both 4-9× past best-of-K plateau at K=10-20 per FW-0036; quality impact negligible). 10-min operator-facing cap enforced via the finalize step's first-action self-check at 510s elapsed since `RM_SUBMIT_TIMESTAMP_MS` (reserves ~90s for finalize-step work so operator-facing total wall stays ≤ 600s); task `maxRunDuration: 660s` is a Batch-level safety net intentionally above the 600s self-check ceiling so the self-check (not Batch) determines when the timeout email fires; `maxRetryCount: 0` (single attempt — retry × 660s would blow the 10-min cap). FW-0037 elbow tuple locked: `L=50` / `idleThreshold=3500` / `swapProbability=0.5`. K_target=2,500 stays as the original design center but is quota-limited at pilot scope; K=88 reflects the largest c3-highcpu-X size (88 vCPU) that fits the current C3_CPUS=108 quota — future quota bump to C3_CPUS ≥176 unlocks `c3-highcpu-176` for K=176 (parameterized as FW-0040 dial). On-demand pricing chosen over Spot for predictable wall-time. Cloud Run per-instance concurrency, threading, and Spot pricing explicitly excluded per D-0070 sub-decisions. **M7 C1 closed 2026-05-10** with asia-southeast1 quota approvals (CPUS=300 / INSTANCES=350 / C3_CPUS=108 — binding); historical closure-K=104 per D-0070 sub-decision 7's three-quota rule (recorded against the 13-VM design that M7 C2 implemented). **M7 C2 closed 2026-05-11** — full Cloud Batch parallel-LAHC chain delivered (worker module + orchestrator wiring + Batch job spec + GCS adapter + per-call attempt-id race closure) with the §12A.4 byte-identity exit criterion proven (5/5 audit tests; local-CLI ↔ Cloud-Batch produce byte-identical writebackEnvelope at the same `(masterSeed, K)` including the load-bearing negative-seed `_UINT64_MASK` case). **M7 C3 (async architecture decision per D-0071 + Codex P1.7 single-VM amendment)** is the active checkpoint — docs-only: locks D-0071 (17 sub-decisions covering scope pivot, Cloud Run thin-front-door + Cloud Batch single-VM with inline finalize architecture, B-prime callback delivery, GCP ID-token auth, always-email-on-failure with `[RosterMonsterV2]` subject prefix + AnalyzerOutput JSON attachment, operator email source = bound shim user, 10-min cap, concurrent rejection by spreadsheet ID via Cloud Batch labels, in-flight toast UX, gcloud CLI status, finalize-runs-always semantics, SRB hidden from menu pending FW-0035-parity, debug `/compute-lahc-test` route kept, bound shim Solve Roster menu LAHC-only, FW-0027 absorbed into M7 C3+C4, FW-0039 finalizer-crash + callback-unreachable watchdog gap captured, new FW-0040 K=88→K=176 future quota bump dial) + sweeps delivery_plan / blueprint / README / AGENTS / cloud_compute_contract §9 + §10 + new async-callback contract section. **M7 C4 implements the async architecture** (Cloud Run shrinks to thin front door — validate + submit + return SUBMITTED; Cloud Batch single-task job spec with `c3-highcpu-88` + `Pool(88)` + inline finalize step in `worker.py`; bound shim `Solve Roster` shows in-flight toast then returns; launcher gains a SECOND Web App deployment with `executeAs: USER_DEPLOYING` for the `?action=async-render-callback` route that invokes existing `RMLib.applyWriteback + RMLib.renderAnalysis` from the POSTed envelope and sends `MailApp.sendEmail` to the operator;
