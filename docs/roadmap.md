### 7) Parallel solver (M7 candidate) — *Held in study before activation*
Parallelize K-trajectory LAHC execution to escape the single-thread bottleneck identified during M6 C4 cloud benchmarking (SRB cloud baseline — the cloud wrapper ships `SEEDED_RANDOM_BLIND` only in v1 per FW-0035; 4-vCPU bump only +10% throughput on the SRB baseline; signal generalizes to LAHC because both share GIL-bound pure-Python compute, and the LAHC cloud projection of ~80s/trajectory × K=10 ≈ ~13 min already overshoots the 5-min Cloud Run timeout before any parallelism). Goal: 64 K-trajectory LAHC runs in <5 min on cloud. LAHC's K-trajectory independence per `docs/solver_contract.md` §12A.2 makes parallelism structurally permitted at the algorithm level — K trajectories share no state past seed-derivation + final aggregation. Deployment shape held for in-detail study before activation: multiprocessing-spawn (Python `multiprocessing.Pool` over K trajectories — likely lowest-friction); Cloud Run job parallelism (separate per-trajectory invocations + result aggregation — linearly scalable but high cost overhead); Cloud Run per-instance concurrency (multiple K-trajectories per container — sync-budget interaction with the existing 6-min `UrlFetchApp` cap); hybrid multi-process within Cloud Run (one container, N=4-8 worker processes). Threading explicitly rejected (GIL + CPU-bound work). Determinism preservation under parallel execution needs re-audit per §12A.4 before activation. Captured as **FW-0038** with full deferral context + revisit triggers; activation precedes implementation per the M5 / M6 / M4 same-day-direction-setting pattern. Not currently scheduled — maintainer studies first, then activates.

## Intentional later work (not near-term)
The roadmap intentionally defers some work until concrete drivers surface:
- Parallel operational search and orchestration (FW-0027) — re-promotable to a milestone when scale or reliability drivers surface. **Distinct from FW-0038 / M7-candidate** which is solver-internal intra-cycle parallelism (parallelize K trajectories within one solver invocation); FW-0027 is operational parallelism (extract + compute + writeback multiple cycles concurrently).
- Observability and benchmark hardening (FW-0028) — re-promotable when long-term reliability or benchmark-comparison drivers surface. M5's analysis tooling overlaps the operator-facing slice but does not subsume the broader observability surface.
- Cloud-side FULL retention support (FW-0030) — prerequisite for the future Deep Solve auto-included analyzer path; not in M5 scope and explicitly carved off from M6 LAHC-only scope per the M5 closure thread.
- Cloud-mode LAHC integration (FW-0035) — stays parked through M6 closure; co-promotion target with FW-0036 / FW-0037 / FW-0038 (cloud LAHC defaults need multi-fixture-validated K + L AND parallelism-aware deployment shape before pinning).
- Systematic weight elicitation / tuning (FW-0033) — re-promotable when cycle-over-cycle weight churn or multi-operator priority disagreement surfaces concrete demand. M5 C4 verdict surfaced this thread but the maintainer's sequencing call was "search next, formulation rework later"; explicitly NOT rolled into M6 scope, stays parked through M6 closure.
- Broad generalization to additional departments before ICU/HD-first learning is closed.
- Pilot rollout to non-maintainer operators (broader pilot is a future milestone or operational rollout step, not currently scheduled).
