# Snapshot Adapter Contract (First-Pass Working Draft)

## Contract status and current scope
This is a **first-pass contract draft** for the snapshot-adapter boundary — the Apps Script side of the inbound Apps Script ↔ Python seam.

`docs/snapshot_contract.md` declares **what** a snapshot looks like (the shape Python ingests). This contract declares **who produces the snapshot, from what source, via what placement, at what cost** — the production-time obligations of the Apps Script extractor running on an operator-edited spreadsheet.

Intended to be useful immediately for M2 C9 implementation work, while keeping only genuinely unresolved details explicit. This draft separates settled decisions (anchored to `docs/decision_log.md` D-0036, D-0040, D-0041, D-0042, D-0043) from deliberate deferrals.

## 1. Purpose of the snapshot-adapter contract
The snapshot-adapter contract pins the Apps Script extractor's responsibilities for converting an operator-edited spreadsheet into a Snapshot-shape JSON file consumable by the Python parser/normalizer per `docs/snapshot_contract.md` and `docs/parser_normalizer_contract.md`.

The goals are:
- preserve fidelity (capture what was seen on the spreadsheet, untransformed)
- keep extraction stable across operator-tolerated edits (row inserts inside a section, request-cell content edits, weight cell edits, blank cells)
- fail loudly and visibly on extraction-blocking defects (missing layout anchors, malformed source spreadsheet)
- avoid expanding the auth surface (Apps Script side stays within the existing launcher OAuth grant; Python side gets no new auth)

## 2. Boundary position in the v2 pipeline
- **Sheet** (operator-edited spreadsheet generated by M1.1 launcher) — the input.
- **Apps Script extractor** (this contract) — reads the sheet via `SpreadsheetApp` + DeveloperMetadata anchors, builds a Snapshot-shape JS object, serializes it to JSON, hands it to the operator as a downloaded file.
- **Operator-mediated transport** — operator saves the downloaded file locally and points the Python CLI at the file path.
- **Python parser/normalizer** — reads the JSON file from disk, deserializes via `Snapshot.from_dict`, runs the existing parser admission pipeline per `docs/parser_normalizer_contract.md`.

The adapter's output IS a Snapshot per `docs/snapshot_contract.md` §5 + §11A. This contract does NOT redefine that shape; it pins how the shape is produced.

## 3. Architectural placement (settled per D-0041)
The extractor lives in **two new Apps Script projects** alongside the existing M1.1 launcher:

- `apps_script/m1_template_bound_script/` — a thin (<30 lines) script container-bound to a single template spreadsheet. Carries `onOpen(e)` to install the operator menu (`Roster Monster → Extract Snapshot`) and one menu handler per public action that delegates to the central library. Bound-script identity means simple `onEdit` / `onOpen` triggers fire natively (the architectural constraint that reverted FW-0024 in PR #89).
- `apps_script/m1_extractor_library/` — a clasp-managed Apps Script Library carrying the extractor implementation. Published as a versioned Apps Script Library; the bound shim depends on it at version `0` (HEAD / always-latest).

**Launcher generation flow change.** `apps_script/m1_sheet_generator/src/GenerateSheet.gs`'s `generateIntoNewSpreadsheet` path replaces `SpreadsheetApp.create(name)` with `DriveApp.getFileById(TEMPLATE_FILE_ID).makeCopy(name)`. The copy inherits the bound shim and its library dependency, so every launcher-generated spreadsheet has the menu from the moment it is opened. The existing `generateIntoExistingSpreadsheet` path remains supported but the resulting tab does NOT receive the bound shim — a known limitation captured in D-0041 sub-decision 8.

**OAuth-scope consequence.** Calling `DriveApp.getFileById(...)` against a manually-created template file is the scenario `docs/decision_log.md` D-0023's closing note anticipated — `drive.file` scope only covers files the app itself created or that were opened via the Drive Picker. The launcher's `apps_script/m1_sheet_generator/src/appsscript.json` MUST therefore broaden its `oauthScopes` from `drive.file` to `https://www.googleapis.com/auth/drive` to enable the template `getFileById + makeCopy` path. Trade-off: wider OAuth consent screen on first operator use; CASA assessment cost re-emerges if the app ever leaves Testing-user mode (deferred to post-pilot per `docs/decision_log.md` D-0041 follow-up action 5). Pilot scope (named operators in Test User pool per D-0021 / D-0022) absorbs the trade-off cleanly.

**One-time manual setup.** Some setup steps cannot be automated (Apps Script does not expose programmatic creation of bound script projects). Setup is performed once per environment:
1. Create the `[INTERNAL] Roster Monster Template` spreadsheet in Drive. Record its File ID. **Share the template explicitly per-email with each named pilot operator at Drive *Viewer* level** (sufficient for `makeCopy`; the resulting copy lands in the operator's own Drive). Required because the launcher executes as `USER_ACCESSING` per `apps_script/m1_sheet_generator/src/appsscript.json` — each operator's own OAuth identity determines actual Drive ACL access at runtime, so the broader `drive` OAuth scope (per `docs/decision_log.md` D-0041 sub-decision 4) is necessary but not sufficient. Without per-email template sharing, an operator with the OAuth grant still hits a Drive permission-denied error on `getFileById`. Update the Viewer allowlist as the pilot operator pool changes; mirrors the library Editor allowlist discipline in step 3.
2. Inside that spreadsheet, Extensions → Apps Script creates the bound script project. Record the bound script project ID.
3. Create the central library Apps Script project. Record its script ID. Share Drive **Editor** permission **explicitly per email** with each named pilot operator (NOT "Anyone with link → Editor"). Apps Script's development-mode (HEAD / version `0`) library access requires the runtime user to have Editor permission on the library script (viewers can only load published numbered versions). Per-email sharing keeps the code-tampering surface restricted to the same named operators who already have OAuth Test User status under D-0021 / D-0022, matching the existing Test User pool allowlist discipline. Link-wide Editor would expose the library to drive-by mutation by any URL holder, and because every bound shim loads HEAD a single edit would propagate to every operator — that path is explicitly avoided per `docs/decision_log.md` D-0041 sub-decision 6.
4. In the bound script's manifest (Resources → Libraries in the Apps Script editor, or `appsscript.json`), add the central library as a dependency at version `0` (HEAD / development mode).
5. In the launcher's Script Properties, set `TEMPLATE_FILE_ID` to the template spreadsheet's File ID.

After step 5 the system is operational; subsequent updates flow through the normal `clasp push` workflow.

## 4. Adapter responsibilities (extends `docs/snapshot_contract.md` §3)
The adapter owns:
- **Extraction fidelity.** Preserve raw cell text untrimmed, undivided, unnormalized at the snapshot layer. Cells with leading/trailing whitespace, mixed case, or unusual punctuation are emitted as raw `rawValue` strings; parser interprets per `docs/parser_normalizer_contract.md` §9 / §14.
- **Day-axis order.** Preserve extracted day order exactly, with `dayIndex` aligned to the template-declared day-axis. Do NOT repair duplicate dates or broken day ordering.
- **DateKey normalization at adapter (per D-0033).** Excel serial dates and locale-formatted date strings are normalized to ISO 8601 (`YYYY-MM-DD`) at extraction; emitted as `rawDateText` on `dayRecords`. Parser uses `rawDateText.strip()` directly as `dateKey` per D-0033.
- **DeveloperMetadata anchor reading (per D-0043).** All sheet-region location uses DeveloperMetadata via **sheet-scoped** finders (`Sheet.createDeveloperMetadataFinder().withKey(...)`) on the active request-entry tab + its `runId`-paired Scorer Config tab. Workbook-scoped finders (`Spreadsheet.createDeveloperMetadataFinder()`) MUST NOT be used because they silently mix anchors across multiple request-entry tabs in the same spreadsheet (operator-duplicated history, regenerated periods). No visible-text fallback. Missing anchors are extraction-blocking errors per §6 / §10.
- **Locator emission.** Each emitted record carries `sourceLocator` per `docs/snapshot_contract.md` §10 (typed by record kind) and `physicalSourceRef` per §12 (sheet name, sheet gid, A1 ref list).
- **Snapshot identity emission (per D-0042).** Adapter generates `metadata.snapshotId` as `snapshot_<spreadsheetId>_<extractionTimestamp>` where `<spreadsheetId>` is the source spreadsheet's File ID and `<extractionTimestamp>` is ISO 8601 with colons replaced by hyphens.
- **Output transport (per D-0040).** Adapter wraps the JSON in `Utilities.newBlob(jsonString, 'application/json', '<snapshotId>.json')` and serves it as a browser download via the bound shim's menu handler.

The adapter does NOT own:
- semantic interpretation of cell content (parser territory per snapshot §3)
- normalization of doctor names, request codes, slot identities
- repair of structural defects in the spreadsheet
- direct write to Drive or Cloud Storage (transport is browser-download only per D-0040)

## 5. Source surfaces on the operator spreadsheet (settled)
The extractor reads two tabs declared by the launcher per `docs/sheet_generation_contract.md`:

### 5.1 Request-entry tab (existing M1 surface; metadata extended in M2 C9 per D-0043)
Contains:
- Section header rows (one per `inputSheetSections` declared in the template artifact; ICU/HD first release: `MICU`, `MICU_HD`, `MHD`).
- Day-axis row carrying date headers spanning the period.
- Doctor rows under each section header carrying doctor `displayName` cells in column A and request cells across the day-axis columns.
- Per-day call-point rows (`MICU_CALL_POINT`, `MHD_CALL_POINT` per `pointRows` in the template artifact) carrying numeric cells.
- Lower-shell prefilled-assignment rows (output-mapping surfaces per `docs/template_artifact_contract.md`).

DeveloperMetadata anchors **attached at generation time** (this contract extends the existing M1.1 metadata posture):
- Sheet level (existing): `rosterMonster:tabType` = `requestEntry`, `rosterMonster:templateVersion`, `rosterMonster:runId`.
- Per row (NEW, D-0043 sub-decision 1):
  - `rosterMonster:section` on each section header row, value = `sectionKey` (`MICU` | `MICU_HD` | `MHD`).
  - `rosterMonster:dayAxis` on the day-axis row, value = `true`.
  - `rosterMonster:doctorRow` on each doctor row, value = `<sectionKey>:<doctorIndexInSection>` (zero-indexed).
  - `rosterMonster:callPointRow` on each call-point row, value = `pointRows.rowKey` (`MICU_CALL_POINT` | `MHD_CALL_POINT`).
  - `rosterMonster:assignmentRow` on each prefilled-assignment shell row, value = `<surfaceId>:<rowOffset>`.

### 5.2 Scorer Config tab (M2 C7 surface)
Contains:
- 9 component-weight rows (one per `componentId` per `docs/domain_model.md` §11.2), 4 columns (Component / Weight / Description / Sign per M2 C7 UX polish).

DeveloperMetadata anchors (existing per M2 C7):
- Sheet level: `rosterMonster:tabType` = `scorerConfig`, `rosterMonster:templateVersion`, `rosterMonster:runId`.
- Per row: `rosterMonster:componentId` per row, value = canonical componentId.

## 6. Extraction algorithm (settled)
The extractor MUST scope all metadata-finder calls to specific tabs (NOT spreadsheet-wide), to avoid silently mixing rows when a single spreadsheet contains multiple request-entry tabs (operator-duplicated for history, regenerated for a new period via the existing `generateIntoExistingSpreadsheet` path, etc.). Procedure:

1. **Identify the active request-entry tab.** Read `SpreadsheetApp.getActiveSheet()`. If `Sheet.createDeveloperMetadataFinder().withKey('rosterMonster:tabType').find()` does not return a match with value `requestEntry`, surface `EXTRACTION_ERROR: not a request-entry tab — open the period's request-entry tab and retry`. This guarantees the operator's currently-focused tab is the one being extracted.
2. **Read the active tab's `runId`.** Sheet-scoped metadata finder for `rosterMonster:runId`. The runId pairs the request-entry tab with its Scorer Config tab per `docs/sheet_generation_contract.md` §11A's tab-name pairing rule.
3. **Locate the paired Scorer Config tab in the same spreadsheet.** Iterate the spreadsheet's sheets; for each sheet, read its sheet-level `rosterMonster:tabType` + `rosterMonster:runId`. Pick the unique sheet where `tabType = scorerConfig` and `runId` matches the value from step 2. If zero or more-than-one match, surface `EXTRACTION_ERROR: paired Scorer Config tab not found / ambiguous for runId <X>`.
4. **For each surface kind on each chosen tab** (request-entry tab from step 1; Scorer Config tab from step 3), call `Sheet.createDeveloperMetadataFinder().withKey(<anchorKey>).find()` — sheet-scoped, NOT `Spreadsheet.createDeveloperMetadataFinder()` which would return matches across all tabs and silently mix runs.
5. If the sheet-scoped finder returns zero matches for a required anchor, the extractor surfaces an `EXTRACTION_ERROR` and refuses to emit a snapshot. The error message names which anchor was missing on which sheet (e.g., `dayAxis row not found on tab "<name>" — sheet may predate the M2 C9 metadata extension`).
6. For each anchored row, reads cells at column offsets relative to the metadata-anchored row. Column offsets are stable because the launcher locks column structure at generation time per `docs/sheet_generation_contract.md` §6 / §7 (operators cannot add/remove columns).
7. Emits one record per cell of interest, carrying:
   - the raw cell value (untrimmed),
   - the typed `sourceLocator` per snapshot §10 with `path` populated from the metadata-anchored row's value,
   - `physicalSourceRef` with `sheetName`, `sheetGid`, and the A1 reference of the source cell(s).
8. After all surfaces are read, builds the `metadata` block (snapshotId per D-0042, extraction summary counts per snapshot §6, periodRef from template-declared period identity), assembles the top-level snapshot object per snapshot §5, serializes to UTF-8 JSON with 2-space indent.

The extractor never repairs:
- duplicate `dayIndex` in the day-axis (parser decides if duplicates are admission-blocking per `docs/snapshot_contract.md` §3 / §13)
- duplicate `(sourceDoctorKey, dayIndex)` in request cells
- broken date ordering on the day-axis
- mis-signed weight values
- malformed numeric content

These are admission cases handled at parser stage per `docs/parser_normalizer_contract.md` §14.

## 7. Output transport (settled per D-0040)
- **Wire format.** UTF-8 JSON, pretty-printed (2-space indent), no envelope wrapping, no compression, no base64.
- **Filename.** `<snapshotId>.json` per D-0040 + D-0042 (e.g., `snapshot_1abc...XYZ_2026-04-29T14-30-15.json`).
- **Mechanic.** `Utilities.newBlob(jsonString, 'application/json', filename)` returned via the bound shim's `HtmlService.createHtmlOutput(...)` page or equivalent download-trigger pattern. Operator clicks the download, saves the file locally.
- **No new auth surface.** Apps Script side reuses the existing launcher OAuth grant. Python side reads the file from disk via the local filesystem; no Drive API, no HTTP server, no clipboard handling.
- **Operator workflow:** open spreadsheet → menu `Roster Monster → Extract Snapshot` → save downloaded JSON → `python -m rostermonster.run --snapshot path/to/snapshot.json`.

The transport mechanic applies symmetrically to writeback per `docs/writeback_contract.md` §6.3 (M3 territory, settled-jointly per D-0036). Outbound (writeback) follows the same pattern in reverse: Python emits a JSON file to disk; operator drags / pastes it into the launcher's writeback form; Apps Script ingests and applies per `docs/writeback_contract.md`.

## 8. Snapshot identity (settled per D-0042)
- **Format.** `snapshot_<spreadsheetId>_<extractionTimestamp>` where:
  - `<spreadsheetId>` = the source spreadsheet's Drive File ID.
  - `<extractionTimestamp>` = ISO 8601 `YYYY-MM-DDTHH-MM-SS`, colons replaced by hyphens for filename safety.
- **Example.** `snapshot_1abc...XYZ_2026-04-29T14-30-15`.
- **Stability.** Each extraction generates a fresh `snapshotId` (timestamp moves), even from the same source spreadsheet with no operator edits. Snapshots are point-in-time facts; same source + later timestamp is a new snapshot.
- **Relationship to `runEnvelope.runId`.** `snapshotId` is adapter-supplied and identifies an extraction event; `runId` is execution-layer-supplied (selector boundary per `docs/selector_contract.md` §9 item 3) and identifies a Python compute run. Distinct surfaces with parallel timestamp-ish formatting.

## 9. Migration: pre-D-0041 sheets do not carry the bound shim
Operator spreadsheets generated by the launcher BEFORE this checkpoint do NOT have the bound shim attached (they were created via `SpreadsheetApp.create()` rather than `makeCopy(template)`). Those sheets:
- have no `Roster Monster → Extract Snapshot` menu when opened,
- have no DeveloperMetadata anchors per §5.1's NEW set (only the M1.1 sheet-level anchors and the M2 C7 Scorer Config tab anchors),
- cannot be extracted by the production extractor.

**Operator workaround:** regenerate the period via the launcher to obtain a fresh sheet with the bound shim and the full metadata anchor set. The pilot's small footprint (one operator, a handful of sheets) makes this a one-message-to-operator concern. No backward-compat shim is provided. The test-only xlsx → JSON bridge (`python/tests/extract_icu_hd_may_2026.py`) remains usable for fixture-derivation against pre-D-0041 sheets but is NOT the production ingestion path.

## 10. Extraction-error surface
Errors surface to the operator via the bound shim's menu handler. Categories:
- **Wrong tab type (per §6 step 1).** Operator triggered Extract on a sheet whose sheet-level `rosterMonster:tabType` is missing or not `requestEntry`: `EXTRACTION_ERROR: not a request-entry tab — open the period's request-entry tab and retry.`
- **Paired Scorer Config tab not found / ambiguous (per §6 step 3).** Active tab's `runId` does not match exactly one Scorer Config tab in the same spreadsheet: `EXTRACTION_ERROR: paired Scorer Config tab not found / ambiguous for runId <X>.`
- **Missing anchor (per D-0043 sub-decision 3).** Sheet-scoped metadata finder returns zero matches for a required anchor on a chosen tab: `EXTRACTION_ERROR: <anchor key> not found on tab "<name>" — sheet may predate the M2 C9 metadata extension. Regenerate via the launcher.`
- **Spreadsheet identity gap.** Operator's session lacks the OAuth scopes to read the spreadsheet (rare; library scope mismatch): `EXTRACTION_ERROR: insufficient permissions; reauthorize the menu.`
- **Library load failure.** Bound shim cannot reach the central library at runtime (Drive sharing change, library project deleted): `EXTRACTION_ERROR: extractor library unavailable; contact maintainer.`

Errors do NOT emit a partial snapshot. Either a complete extraction succeeds and the operator receives a download, or extraction fails and the operator sees the error message.

## 11. Python ingestion CLI (M2 C9 Phase 2 deliverable)
The Python entrypoint `python -m rostermonster.run --snapshot path/to/snapshot.json` reads the file from disk, deserializes via `Snapshot.from_dict`, runs the existing parser admission pipeline per `docs/parser_normalizer_contract.md`, and pipes through the existing solver / scorer / selector stages per `docs/solver_contract.md` / `docs/scorer_contract.md` / `docs/selector_contract.md`. CLI signature, output format, and exit codes are M2 C9 Phase 2 implementation-slice concerns; the contract pins only that the CLI consumes a Snapshot-shape JSON file and respects the existing pipeline contracts.

## 12. Open questions / deferred decisions (intentionally narrow)
- **Multi-operator scaling of the library.** First release pins library version mode at `0` (HEAD) per D-0041 sub-decision 3. If pilot scope expands to multiple operators (currently one), revisit toward versioned library deployments to insulate operators from in-flight broken merges.
- **`generateIntoExistingSpreadsheet` lacks the bound shim.** Per D-0041 sub-decision 8, operators using the launcher's "add tab to existing spreadsheet" path do not get the bound shim and therefore do not get the extractor menu. Currently no demand to fix; would be either a manual bound-shim-attachment helper or a one-shot menu installer. Tracked as future work if demand surfaces.
- **`onEdit`-based behaviors (FW-0024).** D-0041 makes simple `onEdit` triggers viable for the first time; FW-0024 (auto-repopulate empty call-point cells) is no longer architecturally blocked. Implementation remains deferred to its own checkpoint when pilot-operator feedback warrants the work — this contract only declares that the architecture supports it.
- **Sub-second snapshot ID collisions.** D-0042 sub-decision 4 leaves the suffix-on-collision rule unpinned. Currently no observed need; revisit if real workflow emerges.

Any future changes to extractor responsibilities, transport, placement, snapshot identity, or layout-drift handling should be treated as explicit contract/version updates here, not implicit reopenings.

- `docs/template_artifact_contract.md` — declares `inputSheetSections`, `pointRows`, output-shell surfaces whose `sectionKey` / `pointRows.rowKey` / `surfaceId+rowOffset` values appear inside the metadata anchor values per §5.1 of this contract.
- `docs/writeback_contract.md` — symmetric outbound side. Transport mechanic settled jointly per D-0040; bound-shim-driven menu pattern likely to apply on the writeback ingestion form too (M3 implementation-slice concern).
- `docs/decision_log.md` — D-0036 (M2 C9 commitment), D-0040 (transport), D-0041 (placement), D-0042 (snapshot identity), D-0043 (layout-drift handling).
