# Delivery Plan

## 1. Purpose of this document
This is the **living execution guide** for Roster Monster v2.

It defines what is actively being delivered now (active milestone, active checkpoint, and supporting tasks) and what is intentionally next.

It does **not** replace:
- architecture truth in `docs/blueprint.md`
- milestone-level sequencing in `docs/roadmap.md`
- normative technical boundaries in contract docs
- issue-level implementation tracking

## 2. Planning vocabulary
- **Product**: the full end-to-end capability being built.
- **Milestone**: a major delivery state that advances product readiness.
- **Checkpoint**: a bounded, reviewable step inside a milestone.
- **Task**: a concrete work item used to close a checkpoint. Typical decomposition inside a checkpoint follows the **Task 1 (docs) → Task 2 (code, with optional Task 2A / 2B / 2C sub-letters when work has logically distinct chunks) → Task 3 (closure)** cadence per D-0064. Not a contract — a docs-only checkpoint may be just Task 1; an unusual checkpoint may have a different shape. Historical "Phase 1/2/3" labels in §11 closure entries (M2..M5 C2) are preserved as-is.

Working rule: this repo should normally maintain **one active milestone** and **one active checkpoint** at a time.

## 3. Product definition
Roster Monster v2 builds a reusable roster-allocation core with department-specific template control, while preserving Google Sheets as the operational front end. ICU/HD is the first concrete implementation target.

## 4. Product-level delivery principles
- Preserve Google Sheets as the operational front end.
- Keep architecture-first and contract-first boundaries where needed.
- Prioritize work that directly unblocks real operator workflow.
- Avoid reopening settled contracts unless there is a strong consistency reason.
- Avoid spending effort on items that do not support the active checkpoint.

### M6 — Solver-side score-aware search (LAHC)
- **Goal:** Deliver Late Acceptance Hill Climbing (LAHC) as the first alternative solver search strategy alongside today's `SEEDED_RANDOM_BLIND` per `docs/solver_contract.md` §11.1, addressing FW-0003's empirical score plateau on `SEEDED_RANDOM_BLIND`.
- **Why it matters:** Today's solver explores assignment space without score awareness; quality plateaus past ~100-200 candidates per FW-0003. LAHC's history-list accept criterion (move accepted iff score ≥ score from `L` iterations ago) escapes local optima while preserving determinism under fixed seed per D-0053. Strategy choice is recorded in the wrapper envelope so the M5 analyzer (per D-0066 sub-decision 7) makes LAHC's quality improvement (or lack thereof) visible — operator renders each strategy's `AnalyzerOutput` separately into the source spreadsheet (two comparison tabs land per D-0062 always-new-tab discipline) and manually cross-references between them. Single-tab side-by-side renderer enhancement is parked as FW-0034 (NOT in M6 scope).
- **Status:** **Active** *(activated 2026-05-07 alongside M6 C1 per D-0066)*
- **Checkpoints:** C1 solver-strategy contract extension + LAHC design (closed 2026-05-07 per D-0067); C2 LAHC core impl in Python (active); C3 envelope / CLI / cloud integration; C4 live operator validation via M5 analyzer.
- **Dependencies:** M5 closed (analyzer + renderer end-to-end so LAHC quality is measurable from the operator's seat); current `SEEDED_RANDOM_BLIND` implementation in `python/rostermonster/solver/` (M2 C4 closure).
- **Exit criteria:** All four checkpoints closed; LAHC implementation passes unit + integration tests; live operator validation via the M5 analyzer's comparison tab confirms LAHC's behavior is measurable + interpretable from the operator's seat. M6 closure does NOT require LAHC to strictly dominate `SEEDED_RANDOM_BLIND` — surfacing the comparison and learning what the data shows is the load-bearing exit criterion (per the M5-as-calibration-framework discipline).
