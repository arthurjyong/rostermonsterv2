# Future Work

## Purpose
Parking lot for hypothetical future directions and open ideas that are not yet committed work.

This document is intentionally **non-normative**:
- It is not a roadmap. Committed milestone-level delivery order lives in `docs/roadmap.md`.
- It is not a decision. Accepted directional decisions live in `docs/decision_log.md`.
- It is not an active execution tracker. Active milestone/checkpoint/task truth lives in `docs/delivery_plan.md`.
- It is not a contract. Normative technical boundaries live in the contract docs under `docs/`.

Entries here are ideas we want to remember, not commitments to build. If an entry is promoted to committed work, migrate it into the appropriate authoritative doc (roadmap / delivery plan / decision log / contract) and delete the entry from here. If an entry is deliberately rejected, record a short rationale in `docs/decision_log.md` and delete the entry from here.

## Entry format
Each entry should be short and concrete:
- **Title**
- **Date noted**
- **Context** — what we noticed, on which surface
- **Idea / direction** — what we might do later
- **Why deferred now** — usually: out of active checkpoint scope or not yet justified
- **Trigger to revisit** (optional) — what would make this worth picking up
- **Related surfaces** — files / docs the idea would touch

Keep entries implementation-facing. Do not restate contract meaning here. Do not turn this into a vague-ideas dumping ground.

## Maintenance
- Add entries as ideas surface during planning or implementation.
- Keep entries terse; detail belongs in the authoritative doc once the idea is committed.
- When an idea is promoted, move it and delete the parking-lot entry.
- When an idea is rejected, record a short rationale in `docs/decision_log.md` and delete the parking-lot entry.
- This doc must not grow into a parallel roadmap. If it starts dictating sequencing, promote the relevant entries and trim.

### FW-0001 — Launcher title should be department-driven, not hardcoded to ICU/HD
- **Date noted:** 2026-04-21
- **Context:** `apps_script/launcher/src/LauncherForm.html` renders a static `<h1>` reading "CGH ICU/HD Roster Launcher". This is harmless today because the Department dropdown has a single ICU/HD option and ICU/HD is the only supported template (blueprint §16, D-0005, D-0011). The title is effectively baked into the page rather than derived from the selected department.
- **Idea / direction:** When the launcher gains more than one department template, derive the title from the currently selected department / template rather than hardcoding it. Possible shapes: a generic "Roster Launcher" heading plus a department-driven subtitle, or a fully template-driven title string sourced from the template artifact. The title should follow the dropdown selection rather than lag behind it.
- **Why deferred now:** M1.1 scope is ICU/HD-only pilot; there is no second department template yet, so the single-value dropdown and fixed title are consistent with actual supported behavior.
- **Trigger to revisit:** First additional department template lands, or the Department dropdown gains a second option.
- **Related surfaces:** `apps_script/launcher/src/LauncherForm.html` (title literal and dropdown), `apps_script/launcher/src/Launcher.gs` (any server-side template listing surfaced to the form), `docs/sheet_generation_contract.md` §12 (launcher surface).

### FW-0002 — Launcher doctor-count groups should be template-driven, not hardcoded ICU/HD groups
- **Date noted:** 2026-04-21
- **Context:** `apps_script/launcher/src/LauncherForm.html` hardcodes three doctor-count inputs — `ICU only`, `ICU + HD`, `HD only` — mirroring the ICU/HD template's declared doctor groups (`ICU_ONLY`, `ICU_HD`, `HD_ONLY`). The number of inputs, their labels, and their identifiers are all ICU/HD-specific. The submission payload shape (`icuOnly`, `icuHd`, `hdOnly`) is equally ICU/HD-specific. Other departments will declare different doctor-group structures (for example, an Emergency Department template would not have ICU/HD groupings at all).
- **Idea / direction:** Drive the doctor-count fieldset from the selected department template's declared doctor groups rather than from hardcoded form fields. Render one numeric input per declared group, with labels sourced from the template. Corresponding consequence: the launcher payload and `normalizeAndValidateConfig_` consumption shape become template-driven rather than fixed on ICU/HD keys. The blueprint already treats slot/group structure and group-based eligibility as template-declared (blueprint §6, §16), so the direction is consistent with the architecture; only the launcher UI and payload have not yet been generalized.
- **Why deferred now:** ICU/HD is the sole first-release template. A template-driven form surface is not needed until a second department with a different group structure exists, and generalizing prematurely would add surface without operator benefit during the pilot.
- **Trigger to revisit:** First additional department template lands, or ICU/HD's group declarations change, or the launcher payload contract is revisited in `docs/sheet_generation_contract.md` §12.
- **Related surfaces:** `apps_script/launcher/src/LauncherForm.html` (doctor-count fieldset and payload shape), `apps_script/launcher/src/Launcher.gs` (`submitLauncherForm` payload mapping), `apps_script/launcher/src/GenerateSheet.gs` (`normalizeAndValidateConfig_` group handling), `docs/template_contract.md` (doctor-group declaration), `docs/sheet_generation_contract.md` §3A and §12.4.

### FW-0003 — Incremental rule engine implementation as an optimization
- **Date noted:** 2026-04-23
- **Context:** `docs/rule_engine_contract.md` §13 permits internal caching/indexing as long as the public contract remains stateless, and §14 defines the equivalence-test discipline required for any non-stateless implementation to ship. First release ships only the stateless reference implementation and is not required to carry an equivalence-test corpus.
- **Idea / direction:** When profiling shows the stateless reference to be the search-rate bottleneck during solver runs, implement an incremental-state rule engine that maintains indexed occupancy, call-adjacency structures, and eligibility bitsets across solver `tryAdd`/`undo` operations. Ship it only behind a contract-owned corpus of `(normalizedModel, ruleState, proposedUnit) → expected Decision` fixtures that cover at least each hard invariant in §11 and the `FixedAssignment` scoped admission cases in §15, per §14. Byte-identical decisions including canonical violation ordering are non-negotiable.
- **Why deferred now:** First-release correctness target is "works and is auditable," not "works at solver's peak search rate." The stateless reference is the easier implementation to review and is the contractual baseline. Incremental state makes sense only once the solver is exercising it hard enough for validity checks to become the dominant cost.
- **Trigger to revisit:** Profiling on realistic ICU/HD-sized inputs shows rule-engine time dominating a full `SEEDED_RANDOM_BLIND` run, or a later score-aware strategy's search rate requires faster validity adjudication than the stateless reference can supply.
- **Empirical baseline (measured 2026-04-29 against the production CLI on the maintainer's Mac mini, real ICU/HD May 2026 snapshot — 22 doctors × 29 days × 638 requests; raw data + full benchmark script in `experimental/timing/results.md`):**
  - 1 candidate: 1.73 s wall time, 76,682 rule-engine placement attempts (~76 k attempts/candidate)
  - 10 candidates: 16.3 s, 759,385 attempts (linear)
  - 100 candidates: 162.5 s ≈ 2.7 min, 7,614,358 attempts (linear)
  - Linear projection past the measured ceiling: 1,000 candidates ≈ 27 min; 10,000 ≈ 4.5 hours; 100,000 ≈ 45 hours.
  - Per-candidate cost is dominated by the inner-loop rule-engine dispatch (each candidate makes ~76 k stateless `evaluate` calls, the vast majority of which are repeated rejections like `SAME_DAY_HARD_BLOCK`/`BACK_TO_BACK_CALL` against doctor sets the engine has already classified within the same partial allocation).
  - The score-quality plateau on this dataset arrives quickly: 1 → 10 candidates yields ~220-point winner-score improvement; 10 → 100 yields a comparable ~227-point gain. Diminishing returns beyond ~100-200 candidates at this scale.
  - **Practical pilot ceiling: ~100-200 candidates.** Anything larger is impractical wall-time-wise on a developer machine without this incremental-rule-engine optimisation.
  - Validates the original speculation: profiling DOES show rule-engine time dominating once `--max-candidates` rises past the small-pilot range, and the score-quality gradient flattens just past that same range — meaning the solver's appetite for more candidates is now bottlenecked at exactly the point this entry anticipated.
- **Related surfaces:** `docs/rule_engine_contract.md` §13 and §14, future rule-engine test-corpus files (not yet created), solver Phase 2 rule-engine call sites; `experimental/timing/run_timing_benchmark.py` (the harness that produced the empirical baseline above) + `experimental/timing/results.md` (the markdown summary).

### FW-0004 — Score-aware solver strategies
- **Date noted:** 2026-04-23
- **Context:** `docs/solver_contract.md` §11.2 defines an additive extension clause (`scoringConsultation: "READ_ONLY_ORACLE"`) that future strategies MAY opt into for read-only access to a scoring oracle. First release ships exactly `SEEDED_RANDOM_BLIND` (§11.1), which is scoring-blind end-to-end.
- **Idea / direction:** Add one or more score-aware strategies behind the existing `StrategyDescriptor` interface — candidate directions include hill-climb, simulated annealing, beam search, constraint propagation, and CP-SAT (as a native ILP/CP encoding of the problem). Each new strategy declares its own `additionalInputs` (neighborhood size, temperature schedule, beam width, propagator set, etc.) and, when it consults scoring, sets `scoringConsultation: "READ_ONLY_ORACLE"` per §11.2. The scorer stays the single owner of component logic and direction.
- **Why deferred now:** First release needs a correctness baseline before it can judge whether score-aware search actually improves ICU/HD outputs vs. `SEEDED_RANDOM_BLIND` + scoring + selector. Landing a score-aware strategy prematurely would make it hard to isolate whether quality gains came from the solver change, the scorer weight choices, or the selector retention decisions.
- **Trigger to revisit:** The minimal local compute pipeline (M2) has landed and benchmark runs show `SEEDED_RANDOM_BLIND` is consistently leaving meaningful score headroom on realistic inputs, or a specific quality regression against v1 cannot be closed by scorer-weight tuning alone.
- **Related surfaces:** `docs/solver_contract.md` §11.2, §20; `docs/scorer_contract.md` §6 (scorer boundary); future strategy-descriptor registrations.

### FW-0005 — Parallel solver strategy
- **Date noted:** 2026-04-23
- **Context:** `docs/solver_contract.md` §20 explicitly defers parallel execution transport, worker coordination, and cross-worker merge semantics. First release is single-worker and single-process.
- **Idea / direction:** Add a parallel strategy — for example, independent-workers-with-seeded-merge, where N workers each run a seeded variant of the base strategy under distinct seeds and a merge step produces the final `CandidateSet`. Preserve byte-identical determinism under fixed `(seed, workerCount)` inputs. The parallel strategy is an additive `StrategyDescriptor` registration per §11.2 and does not change the solver's single-invocation return shape.
- **Why deferred now:** Single-worker throughput has not been measured against realistic ICU/HD inputs yet. Parallelism only matters if single-worker search time becomes operationally constraining, and it introduces transport/coordination complexity that cuts against M2's minimal-pipeline framing.
- **Trigger to revisit:** Single-worker runs exceed acceptable operator-facing latency on realistic inputs, or benchmark campaigns require running many seeds in parallel for comparative study.
- **Related surfaces:** `docs/solver_contract.md` §11, §20; future worker/orchestration milestones (M4 in `docs/delivery_plan.md` §5).

### FW-0010 — Streaming/delta scoring implementation
- **Date noted:** 2026-04-23
- **Context:** `docs/scorer_contract.md` §16 permits a streaming/delta scoring implementation as an optimization, provided it produces byte-identical `ScoreResult` values to the pure-function evaluation within a single implementation on a single platform, and does not leak lifecycle state into the public contract. Byte-identical is the single normative parity criterion; cross-implementation or cross-platform parity is FW-0011 territory.

- **Trigger to revisit:** Profiling on realistic ICU/HD-sized runs shows scorer time dominating, or a score-aware solver strategy (FW-0004) needs higher-frequency scoring than the pure-function implementation supports.

### FW-0011 — Cross-implementation determinism
- **Date noted:** 2026-04-23
- **Context:** `docs/solver_contract.md` §16 and `docs/scorer_contract.md` §17 and `docs/rule_engine_contract.md` §17 each scope determinism to a single implementation on a single platform. Cross-implementation determinism is explicitly not guaranteed today because RNG choices, floating-point ordering, and container iteration order differ across runtimes.

- **Trigger to revisit:** A second-runtime port of any stage in the pipeline is seriously proposed.

### FW-0017 — Canonical lowest-`unitIndex`-first fill-order rule for `requiredCount > 1`
- **Date noted:** 2026-04-23
- **Context:** ICU/HD first release uses `requiredCount = 1` for every `(dateKey, slotType)`, so `unitIndex` is always 0 and the question of "which `unitIndex` to fill first within a multi-unit `(dateKey, slotType)`" never arises in practice. `docs/decision_log.md` D-0029 settles that all units of the same `SlotType` are equivalent for doctor admissibility and baseline workload weight (with per-unit occupancy via `UNIT_ALREADY_FILLED` explicitly carved out), but it does not yet specify the solver's canonical fill order for multiple unfilled `unitIndex` values within one `(dateKey, slotType)`.
- **Idea / direction:** When the first department template lands with `requiredCount > 1` for any slot type, extend `docs/solver_contract.md` §12.2 (Phase 2 tie-breaking) with an explicit rule that within a single `(dateKey, slotType)`, the solver MUST fill the lowest unfilled `unitIndex` first. This preserves seeded-determinism guarantees in the multi-unit case (different `unitIndex` orderings under the same `seed` would otherwise produce different but equally-valid candidates, which would silently break `docs/solver_contract.md` §16 byte-identical determinism). The rule is mechanical and independent of `fillOrderPolicy` choice — it applies after the policy has selected the `(dateKey, slotType)` to fill next. Pair the contract change with an equivalence-test fixture covering multi-unit demand to lock the behavior in test.
- **Why deferred now:** No first-release department exercises the multi-unit case, so adding the rule today is normative wording for a code path that cannot execute. Landing it alongside the first multi-unit department keeps the contract change scoped to a setting where it can be validated end-to-end.
- **Trigger to revisit:** First additional department template lands with `requiredCount > 1` for any slot type, or ICU/HD changes to require multi-unit days.
- **Related surfaces:** `docs/solver_contract.md` §12.2, §16; `docs/domain_model.md` §7.7, §10.2; `docs/decision_log.md` D-0029.

### FW-0018 — Streaming-selector implementation
- **Date noted:** 2026-04-25
- **Context:** `docs/selector_contract.md` §17.3 specifies full-set ingest as the first-release default: the selector consumes the full `scoredCandidateSet` at once rather than processing candidates as they are scored. Memory footprint is acceptable at first-release ICU/HD scale (5,000 candidates × ~120 `AssignmentUnit` entries each is MB-scale). Streaming is permitted as a future optimization but not specified or shipped in first release.
- **Idea / direction:** When retention volumes grow past comfortable in-memory size — for example, benchmark campaigns that retain across many runs (FW-0015), or future departments with much larger candidate spaces — implement a streaming selector that processes candidates as they are scored rather than holding the full set in memory. Preserve byte-identical determinism by guaranteeing identical output to the full-set reference under matched inputs (analogous to FW-0010's streaming-scorer parity discipline). Both `BEST_ONLY` winner selection and `FULL` retention sidecar emission would need to operate on the streaming path without changing the contract-level output shape (`docs/selector_contract.md` §10) or sidecar artifact shape (`docs/selector_contract.md` §14).
- **Why deferred now:** First-release retention volumes are well within memory bounds at ICU/HD scale. Streaming adds implementation complexity (incremental cascade tracking, incremental per-batch score-distribution accumulation, partial-state byte-identical-output discipline) without operator-facing benefit at current volumes.

### FW-0026 — Fill-order policy: CALL slots before STANDBY slots
- **Date noted:** 2026-04-29
- **Context:** First-release solver Phase 2 fills via `MOST_CONSTRAINED_FIRST` per `docs/solver_contract.md` §12, which deliberately treats CALL and STANDBY slots symmetrically — picks the demand unit with fewest eligible-and-available doctors regardless of slot kind. The maintainer's operational intent (recorded 2026-04-29) is "fill all CALL slots before any STANDBY." Verified during the same conversation that the scorer side already filters every CALL-specific component (`crReward`, `spacingPenalty`, both `pointBalance*`, `preLeavePenalty`) to `slotKind == "CALL"` placements only, so standby placements correctly do NOT contribute to those scores; the gap is only on the solver-side fill order. The current symmetric treatment can produce candidate rosters where standby slots get filled at the expense of call-slot eligibility (a doctor placed on standby that day might have been a better fit for the still-unfilled call), even though the scorer wouldn't reward that standby placement against any CALL component.
- **Idea / direction:** Add a new fill-order policy (provisional name `CALL_FIRST_THEN_STANDBY`) that partitions Phase 2's demand into a CALL pass + a STANDBY pass: run `MOST_CONSTRAINED_FIRST` over the CALL-only demand to exhaustion first, then run it again over the STANDBY-only demand. Within each tier the existing constraint-satisfaction heuristic + seeded tie-break is preserved, so the determinism guarantee under `docs/selector_contract.md` §18 holds. Declared as a new fillOrderPolicy enum value alongside the existing `MOST_CONSTRAINED_FIRST`; first-release default may shift to the new policy or stay symmetric with the new one as opt-in (open design question, see below).
- **Open design questions:**
  1. **Strict precedence vs preferred-with-constraint.** Strict ("ALL call filled before any standby") is the simplest mental model but raises a question on unfillable CALL slots: does the solver report `UnsatisfiedResult` immediately, or proceed to STANDBY anyway? Current behavior is "fill what you can; report leftover as `UnsatisfiedResult`"; the strict version may surface more `UnsatisfiedResult` outcomes earlier. The preferred-with-constraint variant ("CALL gets priority, but within that the constraint heuristic still applies symmetrically") is more nuanced and may avoid extra `UnsatisfiedResult` cases at the cost of being harder to explain to operators.
  2. **First-release default.** Does the new policy *replace* `MOST_CONSTRAINED_FIRST` as the default, or live alongside it as opt-in? Operator-facing implication: existing benchmark seed/score baselines (e.g., the 2026-04-29 -59.97 winner on the May 2026 fixture) will shift if default fill order changes. Worth running a comparison run before promoting the new policy to default.
  3. **Compatibility with FW-0004 (score-aware solvers).** Future score-aware strategies (hill-climb, simulated annealing, etc.) may want to override fill-order entirely to start from a chosen seed roster. The fill-order-policy enum needs to leave room for "fill-order is strategy-internal" so a score-aware strategy isn't forced to honor `CALL_FIRST_THEN_STANDBY` if it doesn't make sense for that strategy.
  4. **Phase 1 (CR seeding) interaction.** Phase 1 already only places CALL-slot CR honors per `docs/solver_contract.md` §12.1 (CR is a CALL-only request), so Phase 1 is naturally compatible with the new Phase 2 ordering. Worth verifying explicitly during implementation.
- **Why deferred now:** Real contract-level change touching `docs/solver_contract.md` (new fillOrderPolicy + §12 phase 2 description), `docs/decision_log.md` (a D-NNNN entry recording the design choice), `python/rostermonster/solver/strategy.py` (Phase 2 partitioning), and tests. Needs a fresh design pass + Codex round; not a tonight-before-bed change.
- **Trigger to revisit:** Maintainer is back fresh and ready to design + implement; OR pilot operator reports a roster outcome where a "call slot was left unfilled while a standby slot got filled with a doctor who could have taken the call" pattern shows up, which would empirically motivate the fix beyond the current intent-based justification.
- **Related surfaces:** `docs/solver_contract.md` §12 + §12.1 + §12.2 + §12A (the fillOrderPolicy declaration); `python/rostermonster/solver/strategy.py` `_phase2_fill` (the existing MOST_CONSTRAINED_FIRST loop that would be partitioned); `python/rostermonster/scorer/components.py` (already filters all CALL-relevant components to `slotKind == "CALL"` — verified 2026-04-29); `docs/future_work.md` FW-0004 (score-aware solvers — interaction noted in design question 3 above); `docs/selector_contract.md` §18 (byte-identical determinism — preserved by the partitioning since each tier's seeded tie-break stays deterministic).

### FW-0037 — LAHC two-step tune-then-run deep-solve pattern with statistically-justified L-tuning protocol
- **Date noted:** 2026-05-08; updated 2026-05-09 with M6 C4 dev-copy evidence (idleThreshold sweep + move-mix sweep)
- **Context:** Post-M6 C3 closure empirical characterization on the May 2026 ICU/HD dev-copy fixture (scripts in `experimental/m6_lahc_comparison/` + `experimental/m6_c4_dryrun/`, PDFs synced to OneDrive `Claude_remote/`) surfaced two related findings the contract's §12A.5 defaults didn't anticipate:
  - **(i) `L=1000` (the §12A.5 default) sits PAST the cliff in our iter-budget regime.** Across 30 seeds × 8 L values, the score-vs-L curve plateaus at L ∈ {1, 10, 50, 100} (mean +90 to +91, sd ~1) and falls off catastrophically at L ≥ 500 (L=1000 mean +35.5 sd 13.2; L=10000 mean +2.8 sd 10.4). Wilcoxon adjacent paired tests confirm the cliff at L=100 → L=500 (p≪0.001, median Δ −9.6).
  - **(ii) Mechanism: classic "L too large for iter budget" failure mode** (Bazargani & Lobo 2017, GECCO; framing also implicit in Burke & Bykov 2017 EJOR's L-vs-runtime trade-off). Iter-by-iter trajectory trace at seed=1001 confirms: at L=10000 with `idleThreshold=5000` cutting trajectories at ~8k iters, the circular history list barely refreshes — most history slots still hold the initial seed-roster score (~−28). The dual-clause accept rule (`proposed ≥ current OR proposed ≥ history[i mod L]`) then degenerates to "accept anything not catastrophically worse than seed" (~77% accept rate). `current_score` random-walks; `best_so_far` rarely strictly improves before idle-termination; emitted score plateaus low. Best_so_far is correctly preserved (monotone-up by §12A.1.f construction) — the trajectory simply never reaches the optimum.
  - **(iii) Statistical analysis of "minimum seeds for the tuning step"** (`seed_count_analysis.py` on the 30-seed dataset) pinned N=5 via three triangulating tests:
    - **TOST equivalence**: all pairwise comparisons within the plateau are statistically equivalent at margin δ = ±2 score points (paired t-test, n=30, p_TOST < 0.05). L=50 vs L=100 are equivalent even at the tighter δ = ±1 margin — operationally interchangeable.
    - **Empirical Wilcoxon power for cliff detection** (L=100 vs L=1000, one-sided, α=0.05, 5000 bootstraps per N): 0% power at N ≤ 4 (test mechanics — discrete null distribution can't reach p < 0.05 with n ≤ 4 paired observations regardless of effect size); 100% power at N ≥ 5.
    - **Bootstrap stability of the argmax-of-mean pick rule**: P(picked L ∈ plateau {1, 10, 50, 100}) = 100% from N=1 (cliff is so dramatic the wrong-side answer is mechanically impossible); P(picked L ∈ best-decade {50, 100}) climbs monotonically — 80% at N=1, 93% at N=5, 96% at N=6, 98% at N=10.
  - **Adaptive (parameter-less) LAHC** (Bazargani & Lobo 2017) was considered as an alternative and rejected for our deployment context: per-trajectory adaptive overhead (warm-up phases at L=1, 2, 4, ..., L_optimal) costs ~2–3× per trajectory and doesn't amortize when the maintainer plans to run K trajectories or repeat across many seeds. Two-step tune-then-run amortizes the tuning cost across all subsequent runs; tuning becomes a once-per-problem-class cost, not a per-trajectory tax. (Adaptive LAHC remains useful as a *generic-library* fallback for new problem classes — kept as a candidate Tier-3 algorithm variant.)
  - **(v) M6 C4 update — idleThreshold sweep + move-mix sweep refined the sweet-spot tuple beyond `L`** (uncommitted scripts `idle_threshold_sweep.py` + `move_mix_sweep.py` in `experimental/m6_c4_dryrun/`): **idleThreshold sweep** (multiple seeds × ~6 log-spaced values) identified the score-vs-idle elbow at **`idleThreshold ≈ 3500`** — `idleThreshold=500` cuts trajectories before they reach the L=10 plateau's optimum (still climbing); below the elbow the score curve is still rising; above ~3500 the curve flattens (`idleThreshold=10000+` wastes budget without quality gain). M7 activation per D-0070 pins `idleThreshold=3500` as the elbow on the operator-side direct read of the sweep PDFs synced to OneDrive `Claude_remote/`. (An earlier internal summary called the sweet-spot ≈ 2000 — that was a quality-vs-cost minimum-cost point; 3500 is where the curve's bend actually sits.) **Move-mix sweep** across `swapProbability ∈ {0.0, 0.25, 0.5, 0.75, 1.0}` confirmed the §12A.7 default `0.5` is statistically equivalent to `0.25` and `0.75` (within the L=50/idle=3500 plateau); extreme values `0.0` (reassign-only) and `1.0` (swap-only) work via §12A.1.a fallback but show no quality lead and slightly worse plateau-time variance. Confirms maintaining `swapProbability=0.5` as the default while preserving the `--lahc-swap-probability` knob (added in M6 C4 PR-A) for future move-mix work. **M7 C1 (D-0070) elbow tuple**: `L=50`, `idleThreshold=3500`, `swapProbability=0.5` — picked from the FW-0037 plateau as the M7 production config; `L=50` chosen over `L=10`/`L=100` to slightly shorten per-trajectory wall time while staying in plateau (smaller `L` → history list refreshes faster → idle counter increments faster → fewer iters before cutoff).
- **Idea / direction:** Promote LAHC's deep-solve usage to a two-step pattern:
  - **Step 1 — Tune L (one-time per problem class).** Run a small L sweep with N=5 paired seeds × ~6–8 log-spaced L values × per-run cost. On May 2026 ICU/HD scale that's ~5 × 8 × ~16s ≈ ~11 minutes one-time. Identify the picked L via argmax-of-mean (or, more robustly, the smallest L that's not Wilcoxon-significantly worse than the best — the "elbow" rule). Cache the picked L keyed by a problem signature (snapshot shape hash + scoring-config hash + LAHC params other than L).
  - **Step 2 — Production run at the tuned L.** Operator's actual K-trajectory deep-solve uses the cached L. No per-trajectory tuning overhead. K can be operator-set to whatever fits their budget (FW-0036's K-default discussion lives on top of this).
  - **Operator-facing CLI surface (M7-scale):**
    - `python -m rostermonster.tune-l --snapshot S.json --strategy LAHC --seeds 5 --output l_tuning.json` — runs the L sweep, dumps the picked L + statistical evidence (Wilcoxon table, score-vs-L curve, TOST verdict). Maintainer-only per D-0067 sub-decision 6.
    - `python -m rostermonster.run --strategy LAHC` — when a cached `l_tuning.json` matches the problem signature, auto-load picked L. Otherwise warn "no tuning found — using §12A.5 default L=1000 (likely suboptimal); run `tune-l` first".
  - **Pinned methodology constants** (subject to multi-fixture validation before contract-level pinning):
    - L sweep grid: `L ∈ {1, 10, 50, 100, 500, 1000, 5000, 10000}` (8 log-spaced values covering greedy → past-cliff)
    - Tuning seeds: N=5 paired (Wilcoxon power floor + bootstrap stability target)
    - Equivalence margin: δ = ±2 score points (within scoring rounding for our scale)
    - Pick rule: argmax of mean across N seeds (or Wilcoxon-elbow detection — both pin to the plateau)

- **Context:** During M6 C4 cloud benchmarking on the live `roster-monster-compute` Cloud Run service (1 vCPU / 512 MB / 5-min timeout / scale-to-zero per `docs/cloud_compute_contract.md` §9), the cloud wrapper ships `SEEDED_RANDOM_BLIND` only in v1 per D-0068 / FW-0035 (cloud-mode LAHC integration parked through M6 closure), so the cloud runs measured here are **SRB cloud baselines**, not LAHC cloud runs. **SRB-K=32 cloud baseline**: warm ~270s for the full request (Phase 1 + Phase 2 × K=32 + scorer + selector + writeback assembly) — substantially slower than local SRB-K=32 (which completes in ~10s including the same downstream stages). **4-vCPU / 2 GiB hardware bump tested and reverted** on the SRB cloud baseline — only +10% throughput, demonstrating that hardware bumps yield diminishing returns under a GIL-bound single-thread workload. The single-thread finding generalizes to LAHC because LAHC's inner loop is also pure-Python sequential work (move generator → rule-engine call → scorer call → accept/reject decision, all GIL-bound), and LAHC's algorithm is intrinsically single-threaded *within* a trajectory by construction (one history list, one currentRoster, sequential accept-reject decisions). The cap is single-thread throughput, not core count, and it shapes both strategies the same way. **LAHC cloud projection** (extrapolated from local LAHC + cloud SRB slowdown — NOT a measured cloud LAHC run, since FW-0035 is parked): local LAHC per-trajectory cost ~16s × cloud's same-strategy slowdown factor ≈ ~80s/trajectory; K=10 LAHC cloud → ~13 min, well past the 5-min Cloud Run timeout, before any parallelism. **Algorithm-level parallelism is structurally permitted by `docs/solver_contract.md` §12A.2** — K trajectories are independent (no shared state past the seed-derivation step + final candidate aggregation), so K trajectories CAN run in parallel without changing the LAHC algorithm itself. Maintainer 2026-05-09 direction: "we should move to parallelize our code... super a lot of effort... benefit is worth... should be our next milestone... by then we should aim to start doing parallel core and multiple active CPU runs hopefully we can do 64 runs in less than 5 mins ideally". M7 (parallel solver) is the candidate next milestone but explicitly held in study before activation per the M6 closure verdict (D-0069) — the deployment shape is not pre-decided.

- **Design-rationale archive — candidate shapes considered.** Preserved here because Cloud Batch was picked over Cloud Run Jobs after explicit comparison; the reasoning shouldn't be lost when M7 closes. The shapes evaluated:
  - **(a) Multiprocessing-spawn (Python).** `multiprocessing.Pool(processes=N)` over K trajectories with each worker running one full LAHC inner loop on its own seed. Spawn over fork to avoid CoW issues with imported module state; serialize seed + normalized model + scoring config to each worker; worker returns its emitted `bestRoster` + per-trajectory diagnostics. Wall-time savings ≈ K/N at the limit (linear scaling) until per-process startup overhead amortization caps the win; on Cloud Run, container-bound → workers share one container's RAM budget (512 MB → bump to 2-4 GiB likely needed). Lowest-friction option.
  - **(b) Threading (Python).** Won't help — GIL holds across pure-Python compute; the M6 C4 4-vCPU benchmark already showed CPU isn't the cap. **Rejected.** Listed for completeness so future maintainers don't re-relitigate.
  - **(c) Cloud Run job parallelism.** N separate per-trajectory Cloud Run job invocations with result aggregation in the orchestrator. Linearly scalable but high per-invocation overhead (cold-start + auth + container init per trajectory); cost-per-cycle climbs faster than (a). Likely overkill for K=64.
  - **(d) Cloud Run per-instance concurrency.** Multiple K-trajectory requests hitting the same container simultaneously, each in its own thread/process. Requires careful sync-budget interaction with the existing 6-min `UrlFetchApp` cap from the bound shim. Doesn't help a single Quick Solve request — only helps when multiple operators compute concurrently.
  - **(e) Hybrid — multi-process within Cloud Run.** Shape (a) + Cloud Run vertical scaling: one container, N=4-8 worker processes, K-trajectory split across them. Most likely deployment shape; combines (a)'s linear scaling with Cloud Run's existing operator-facing surface unchanged.
  - **Goal:** 64 K-trajectory LAHC runs in <5 min on cloud (per maintainer framing — "hopefully... ideally"; stretch target, not a contract). **Capacity math under the conservative LAHC cloud projection** (~80s/trajectory at §12A.5 default LAHC params per the context above): N=8 workers → 64 × 80s / 8 = ~640s ≈ ~10.7 min — does NOT hit <5 min by itself. **Only intra-request fanout shapes count for one Quick Solve invocation** — shape (d) Cloud Run per-instance concurrency is explicitly excluded here per its own definition above ("doesn't help a single Quick Solve request — only helps when multiple operators compute concurrently"). The viable shapes for cutting one request's wall time are (a) multiprocessing-spawn within the request, (c) Cloud Run job parallelism with per-trajectory invocations + result aggregation, and (e) hybrid multi-process within one Cloud Run container. Closing the gap to <5 min requires either (i) **more in-request workers**: 64 × 80s / N < 300s solves to N > 17.07 ⇒ N ≥ 18 at conservative projection (and that 18-worker number assumes zero orchestration / cold-start / aggregation overhead — realistic deployment needs additional margin, so plan ≥ 20 workers; Cloud Run vertical scaling supports up to 8 vCPU per instance under shape (a) / (e), so hitting 18+ workers requires shape (c) cross-job parallelism or shape (e) × (c) hybrid; shape (c) scales workers across separate per-trajectory job invocations at the cost of per-invocation overhead), and/or (ii) **lower per-trajectory cost via the empirically-tuned LAHC defaults** (FW-0037 sweet-spot `L=10` + `idleThreshold=2000` likely runs significantly faster per trajectory than the §12A.5 L=1000 baseline that anchors the ~80s projection — actual cloud cost at the tuned defaults is unmeasured and needs cloud-LAHC piloting to nail down), and/or (iii) **deployment-shape choice**: shape (e)'s N=4-8 worker processes per container can stack with shape (c)'s cross-job parallelism (per-trajectory job invocations dispatched in parallel from the orchestrator, each running shape (e) internally) for compounded scaling within a single Quick Solve request. The 64-runs-in-<5-min target is reachable but the *combination* of workers + tuned defaults + deployment shape is what makes it reachable; M7 activation needs to pin which combination based on cloud-cost vs operator-responsiveness trade-offs. Failing to hit <5 min at the conservative projection isn't a deal-breaker — even ~10 min cloud LAHC at K=64 substantially beats the current single-thread cloud LAHC projection (~13 min at K=10) and unlocks operator-facing cloud LAHC (FW-0035).
- **Why these shapes were rejected** (preserved as design rationale; these were the load-bearing reasons Cloud Batch + dense-pack hybrid won):
  1. **(b) Threading: rejected.** GIL holds across LAHC's pure-Python inner loop (move generator + rule-engine call + scorer call all bounce through the GIL); the M6 C4 4-vCPU benchmark already showed CPU isn't the cap. Listed for completeness so future maintainers don't re-relitigate.
  2. **(d) Cloud Run per-instance concurrency: excluded.** Per the Cloud Run docs, concurrency caps simultaneous HTTP requests per container — not a way to split one request's CPU-bound work. Helps multiple operators concurrent, not the single-request 64-trajectories-in-<5-min target.
  3. **(a) Cloud Run multiprocessing-spawn alone: insufficient ceiling.** 8 vCPU per Cloud Run instance caps parallelism well below the K=2,500 target; Cloud Run vertical scaling can't reach 313+ workers per request. Useful as a sub-pattern inside (e) hybrid, but not as the parallel layer on its own.
  4. **(c) Cloud Run Jobs alone: workable but bested by Cloud Batch.** Cloud Run Jobs caps per-task at 8 vCPU (matches our dense pack) but limits total task count to ~10K and is less flexible on machine families; Cloud Batch is purpose-built for fan-out compute with deeper retry semantics + mixed-machine-type support, which matters when "if pilot project goes well we may very well goes into much much heavier calculations with several fold of complexity" (maintainer 2026-05-10 direction). Setup-investment delta is modest (~half-day of YAML config); headroom for growth justifies the choice.
  5. **Spot pricing: rejected.** 60-91% discount irrelevant when cost is ~$2.20/run on-demand. Spot's preemption-retry behavior would push wall time over the sync ceiling on ~5-10% of busy days, breaking the sync UX contract. Predictability beats price in this regime.
- **Determinism preservation re-audited at M7 C2** per `docs/solver_contract.md` §12A.4. The current `_per_candidate_seed(rng: Random)` in `python/rostermonster/solver/solver.py:52` is **NOT index-addressable** — it advances a single `Random(masterSeed)` stream sequentially via `rng.getrandbits(63)`, matching the local CLI's K-trajectory loop. M7 C2 preserves byte-identical determinism via **orchestrator-side seed pre-derivation** per D-0070 sub-decision 8: the Cloud Run Service generates all K_approved seeds upfront from `Random(masterSeed)` (same sequential draws as the local CLI), then serializes each task's assigned seed slice into Cloud Storage alongside the input snapshot. Each Batch task reads its pre-computed seed list rather than re-deriving from masterSeed; the worker code path is decoupled from the seed derivation function. This avoids both modifying `_per_candidate_seed` (which would break local-CLI byte-identity at the source) AND having each task replay the master RNG to its starting index (wasteful + bug-prone). M7 C2 verifies same `(masterSeed, K_approved)` produces byte-identical winners across local-CLI vs Cloud-Batch. No solver `contractVersion` bump — §12A.2 K-trajectory independence already permits parallelism at the algorithm level; orchestrator-side seed pre-derivation is a deployment-shape implementation detail.
